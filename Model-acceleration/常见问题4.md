## 用 CompletableFuture 比老的 Future 好在哪？怎么编排多个异步任务？

在 AI 服务里同时调用多个模型时，`CompletableFuture` 相比老的 `Future` 最大优势是：它不只是“拿结果”，而是能把异步任务编排成一条流程。

老 `Future` 的问题比较明显：只能通过 `get()` 阻塞等待，不方便注册回调，也不擅长表达“多个任务都完成后再聚合”或“任意一个先返回就用它”。如果要组合多个模型调用，代码很容易变成线程池提交一堆任务，然后到处 `get()`，异常处理、超时和降级逻辑也会比较分散。

`CompletableFuture` 更适合业务流水线：

1. 用 `thenApply`、`thenCompose`、`thenAccept` 串联后续逻辑。
2. 用 `allOf` 并行调多个模型后统一聚合。
3. 用 `anyOf` 做最快结果返回或多模型兜底。
4. 用 `exceptionally`、`handle`、`whenComplete` 集中处理异常、日志和降级。

示例思路如下：

```java
CompletableFuture<String> gpt = CompletableFuture.supplyAsync(() -> callGpt(prompt), pool);
CompletableFuture<String> claude = CompletableFuture.supplyAsync(() -> callClaude(prompt), pool);
CompletableFuture<String> local = CompletableFuture.supplyAsync(() -> callLocalModel(prompt), pool);

CompletableFuture<Result> result =
    CompletableFuture.allOf(gpt, claude, local)
        .thenApply(v -> {
            List<String> answers = List.of(gpt.join(), claude.join(), local.join());
            return mergeAndRank(answers);
        })
        .exceptionally(ex -> fallbackResult(prompt));
```

生产环境还要配合自定义线程池、超时控制、熔断限流和上下文透传，否则异步并发很容易把下游模型服务打爆。

## 模型服务流量突然暴增，怎么做自动扩缩容？看哪些指标？

模型服务的自动扩缩容通常分两层：在线推理服务层和底层资源层。在线层可以用 Kubernetes HPA、KEDA 或自研扩缩容器，底层 GPU 节点用 Cluster Autoscaler 或云厂商弹性伸缩。核心目标是先保证队列不被打爆，再控制成本，而不是单纯看 CPU。

扩容指标主要看几类：

1. 请求侧指标：QPS、并发数、入口队列长度、排队时间。
2. 延迟指标：重点看 P95、P99，而不是平均延迟。
3. 资源指标：GPU 利用率、显存占用、CPU、内存、网络带宽。
4. 稳定性指标：5xx、timeout、熔断比例、重试比例。

实际策略上，一般会做组合判断，例如“队列长度持续超过阈值 + P95 延迟升高 + GPU 利用率较高”才扩容，避免短暂尖峰导致频繁抖动。扩容要设置最小副本数、最大副本数、冷却时间，并保留少量 warm replicas，因为大模型加载权重比较慢，冷启动成本很高。对于活动流量、定时任务这类可预测峰值，可以提前定时扩容。

缩容要更保守，一般要求低 QPS、低队列、低资源利用率持续一段时间才缩。整体上，自动扩缩容应该围绕 SLO 做闭环，而不是只盯某一个瞬时资源指标。

## 预处理和后处理怎么设计才能灵活增减步骤？

预处理和后处理适合设计成“可编排的流水线”。每个处理步骤都是独立组件，统一实现类似 `process(context) -> context` 的接口。`context` 中放原始输入、清洗后的输入、模型输出、格式化结果、trace 信息等。新增步骤时只需要新增一个 processor 并注册到流水线，不需要改推理主流程。

工程上可以分三层：

1. 步骤接口：约定输入输出、异常处理、是否可跳过、是否短路。
2. pipeline 编排器：负责按顺序执行步骤，支持条件执行、失败重试、短路返回。
3. 配置层：用 YAML、JSON 或数据库配置启用哪些步骤、执行顺序和参数。

例如预处理阶段可以配置敏感词过滤、归一化、脱敏、prompt 拼装；后处理阶段可以配置 JSON 解析、字段校验、单位转换、格式化和兜底文案。不同业务复用同一套步骤库，只是 pipeline 配置不同。

还要注意可观测性。每个步骤都应该记录耗时、输入输出摘要和错误码，方便定位到底是清洗、模型推理还是格式化出了问题。关键步骤要有单元测试和回放测试，保证调整顺序或增减步骤时不会破坏整体链路。

## 多张 GPU 卡怎么把请求合理分配到不同卡上？

可以把多张 GPU 当成一个资源池，在服务层做调度，而不是让请求直接绑定某张卡。每张卡维护一份运行状态，例如当前并发数、显存占用、队列长度、最近平均推理耗时和健康状态。新请求进来时，调度器先过滤掉不可用或显存不足的卡，再从候选卡中选择负载最低的一张。

简单场景可以用“最少正在处理请求数”或“最短队列优先”。如果请求大小差异明显，就不能只看请求数，还要估算 token 数、batch size、显存需求和预计耗时，做加权评分：

```text
score = 当前队列预计耗时 + 显存压力权重 + 并发惩罚
```

工程上通常给每张 GPU 配一个 worker 进程，worker 只负责自己那张卡。前面放 dispatcher，负责接收请求、查看各 worker 心跳和指标、分发任务。如果所有 GPU 都忙，请求进入全局队列，或者按优先级排队；超过等待时间就返回限流或降级结果。

还要考虑故障隔离。某张卡 OOM、进程异常或心跳超时后，要临时摘除，避免继续分发。对于大模型推理，还可以结合动态 batching，把短时间内分到同一张卡的请求合并执行，提高 GPU 利用率。

## 对外 API 怎么限制不同客户的调用频率？

对外提供 API 时，限流一般按客户维度做，而不是只按 IP。每个客户有 `app_id`、`api_key` 或 token，网关或服务端完成鉴权后，根据客户标识读取限流配置，例如客户 A 每秒 10 次，客户 B 每秒 100 次。

常见算法包括固定窗口、滑动窗口、令牌桶和漏桶。固定窗口实现简单，例如 Redis 中 `INCR customer:{id}:second:{timestamp}` 并设置过期时间，但窗口边界可能出现瞬时双倍流量。工程上更常用令牌桶：每个客户有一个桶，按配置速率补充令牌，请求进来先尝试扣 1 个令牌，扣成功放行，否则返回 `429 Too Many Requests`。

分布式服务中，限流状态不能只放单机内存，否则多实例会放大额度。常见做法是在 API Gateway、Envoy、Kong、OpenResty 或自研网关层统一限流，底层用 Redis、一致性哈希内存节点或专门限流服务。Redis 方案要用 Lua 脚本保证原子性，避免并发超发。

超过限制时返回 HTTP 429，并带上 `Retry-After`、当前限额、剩余额度等响应头。限流配置应该动态化，支持不同套餐、临时提额、黑白名单和灰度生效。

## 大模型生成结果是流式的，怎么让前端实时看到内容？

通常让后端把大模型的 token 增量以流式响应转发给前端，而不是等完整结果生成完再返回。常见方案是 SSE 和 WebSocket。

如果只是服务端持续推送文本给浏览器，优先选 SSE。前端用 `EventSource` 建立连接，后端设置 `Content-Type: text/event-stream`，模型每生成一段内容，就按 `data: xxx\n\n` 写入响应并 `flush`。前端监听 `message` 事件，把收到的增量内容追加到页面状态中。SSE 基于 HTTP，天然支持断线重连，实现和排查都比较简单。

如果业务需要双向实时交互，例如用户中途打断生成、实时发送控制指令、多路会话复用，WebSocket 更合适。后端收到模型流式输出后逐片发送消息，前端按消息顺序拼接展示。

工程上要注意几件事：前端不要每个 token 都触发重渲染，可以节流或批量追加；浏览器断开连接后后端要及时取消模型请求，避免浪费算力；协议里要定义结束事件和错误事件，例如 `[DONE]` 或结构化 JSON；生产环境要检查网关和 Nginx 是否开启了响应缓冲，否则流式数据可能被缓存到最后一次性返回。

## 模型新版本上线怎么做到服务不中断？热切换怎么做？

模型热切换的核心是把“模型加载”和“流量切换”解耦，不要在请求线程里临时加载新模型。

工程上可以在模型服务进程内部维护版本注册表，例如 `v1`、`v2` 对应不同的模型实例、配置、特征处理逻辑和依赖资源。新版本发布时，先在后台异步加载模型，完成权重读取、GPU/CPU 内存分配、预热推理和健康检查。只有这些都通过后，才把路由指针从旧版本原子切到新版本。

热切换要注意两点。第一，旧版本不能马上释放，因为切换瞬间可能还有请求正在使用它。可以用引用计数、读写锁或 RCU 思路：新请求走新版本，旧请求继续跑旧版本，等旧版本无请求后再卸载。第二，要有回滚机制，新版本指标异常时能快速把流量切回旧版本。

更稳妥的上线方式是灰度发布或金丝雀发布。比如先让 1% 流量进入新模型，观察延迟、错误率、显存占用、业务指标和模型效果，再逐步扩大到 10%、50%、100%。大模型还要提前预热 KV cache、编译图或 TensorRT engine，避免切换后首批请求抖动。

## 怎么判断一个模型服务现在是健康的？挂了怎么快速发现？

判断模型服务是否健康，不能只看进程还在不在，而要分层看。

首先是基础存活：服务进程、端口、容器状态、GPU/CPU/内存是否正常，依赖的模型文件、配置、缓存、队列、数据库或向量库是否可访问。其次是业务健康：定期发轻量级探测请求，验证接口能返回、状态码正确、延迟在阈值内、结果格式符合预期。

模型服务光返回 200 还不够，还要看推理耗时、错误率、超时率、队列长度、GPU 显存、batch 堆积、请求量突降等指标。服务侧应暴露 `/healthz`、`/readyz`、`/metrics`，接入 Prometheus、Grafana 或云监控；日志接入 ELK、Loki 等系统，按错误码、异常栈、超时关键字聚合。

生产上还要做外部探活，也就是从用户访问路径定时调用真实 API，避免“内部看着正常，用户实际不可用”。告警规则可以包括连续健康检查失败、P95/P99 延迟升高、5xx 增多、请求量异常下降、GPU 显存打满、队列积压超过阈值。发现故障后触发自动重启、摘除实例、流量切换或降级策略。

## 监控模型服务性能，哪些指标最重要？

模型服务性能监控优先看以下几类指标。

第一是延迟，尤其是 P50、P95、P99，而不是只看平均值。在线推理服务里，长尾延迟更能反映用户体验，也能暴露批处理、队列堆积、模型冷启动或下游依赖变慢的问题。

第二是吞吐量，比如 QPS、每秒处理 token 数、batch size 利用率。它反映服务承载能力，也方便评估扩容、限流和成本效率。

第三是错误率和超时率，包括 5xx、请求失败、模型推理异常、依赖服务失败等。延迟升高但错误率正常，和错误率突然上升，排查方向完全不同。

第四是资源利用率，重点看 GPU 利用率、显存占用、CPU、内存、网络 IO。模型服务常见问题不是单纯机器不够，而是 GPU 空闲但队列很长，或者显存碎片、数据预处理卡在 CPU。

第五是队列和排队时间。如果系统有异步队列或动态 batch，这个指标可以区分是模型本身慢，还是请求在入口层、调度层等待太久。

最后还要关注业务和模型质量指标，例如空结果率、置信度分布、输入长度分布、输出 token 数、命中率等，避免线上数据分布或模型效果异常漂移。

## 用 K8s 部署模型服务有什么需要注意的？

用 K8s 部署模型服务，首先要关注资源管理。模型服务通常对 CPU、内存、GPU 都敏感，要明确设置 `requests` 和 `limits`，尤其是 GPU 资源，避免调度不确定或多个服务争抢资源。大模型还要关注显存占用、模型加载时间和节点亲和性，必要时用 node selector、taint/toleration 或 GPU node pool 做隔离。

其次是启动和健康检查。模型服务启动可能很慢，不能简单套普通 Web 服务的探针参数。`startupProbe` 要给足模型加载时间，`readinessProbe` 要等模型真正可推理后再接流量，`livenessProbe` 不宜过于激进，否则可能在高负载或长推理时误杀 Pod。

第三是弹性和流量治理。模型推理一般延迟高、资源重，HPA 不能只看 CPU，最好结合 QPS、队列长度、GPU 利用率等指标。滚动发布时要控制 `maxUnavailable`，避免新 Pod 还在加载模型、旧 Pod 已经下线，导致容量骤降。

第四是模型与配置管理。模型文件不要直接打进过大的镜像，常见做法是启动时从对象存储拉取，或用 PVC、本地缓存层加速。模型版本、配置、依赖和镜像版本要能对应，方便回滚和排查。

## 参数服务器怎么保证多台机器数据一致？

参数服务器保证多机数据一致，核心靠“参数所有权”和“同步协议”。

在典型 PS 架构里，模型参数会按 key 或 tensor 分片，每个分片由固定的 parameter server 负责维护。worker 不直接互相修改参数，而是把梯度或更新请求发给对应 PS。这样同一份参数只有一个权威写入点，避免多台机器各改各的。

一致性策略一般有三类。最强的是同步 SGD：所有 worker 先拉取同一版本参数，计算梯度后统一 barrier，PS 聚合完再更新，再进入下一轮。这种一致性最好，但慢 worker 会拖住整体训练。异步 SGD 则是 worker 拉参数、算梯度、推更新不等待其他 worker，吞吐更高，但会有 stale gradient。折中方案是 SSP，也就是 bounded staleness，允许落后固定步数，超过阈值就阻塞快 worker。

PS 还会通过版本号、时间戳、step counter 判断更新新旧；通过原子更新、锁或单线程队列保证同一参数分片上的更新顺序；通过 checkpoint 和 WAL 处理故障恢复。实际系统通常是在强一致、最终一致和有界延迟一致之间取舍，保证训练可收敛并兼顾吞吐。

## All-Reduce 具体是怎么工作的？

在数据并行训练里，每台机器有一份完整模型和不同 mini-batch。反向传播后，每台机器都会算出本地梯度，但这些梯度只代表本机数据。为了让所有模型参数保持一致，需要把各机器梯度做全局聚合，通常是求和再除以 worker 数，这个通信操作就是 All-Reduce。

All-Reduce 可以理解成两件事合在一起：先 `reduce`，把所有机器上的同名梯度张量相加；再 `broadcast`，把聚合后的结果发回每台机器。最终每台机器拿到完全相同的平均梯度，然后各自执行 optimizer step，所以参数更新结果一致。

工程上常见实现是 ring all-reduce。假设有 N 台机器，每个梯度张量被切成 N 份，机器按环形拓扑通信。第一阶段是 reduce-scatter：每台机器不断把自己的一块数据发给下一个邻居，同时从上一个邻居接收并累加，经过 N-1 轮后，每台机器负责的一块梯度已经是全局求和结果。第二阶段是 all-gather：这些求和后的分片继续在环上传播，经过 N-1 轮后，每台机器都收齐完整的全局梯度。

PyTorch DDP、Horovod、NCCL 等会把梯度按 bucket 打包，并尽量让反向计算和通信重叠，某些层的梯度一算完就开始 all-reduce，而不是等整个 backward 结束。

## 训练到一半怕机器挂了，怎么设计检查点保存和恢复？

训练中断恢复的核心是让训练状态可复现，而不是只保存模型参数。检查点里至少要保存模型权重、优化器状态、学习率调度器状态、当前 epoch/step、随机数状态，以及混合精度训练里的 scaler 状态。如果是分布式训练，还要考虑 rank、采样器状态和全局 step 对齐。

工程上一般按固定间隔保存，例如每 N 个 step 或每个 epoch 保存一次，同时保留 `latest` 和若干历史版本。`latest` 用于故障恢复，`best` 用于最终评估或部署。为了避免保存过程中机器挂掉导致文件损坏，可以先写临时文件，写完后再原子重命名。

恢复时不能只 `load_state_dict(model)`，还要恢复 optimizer、scheduler、scaler 和 step，然后从对应的数据位置继续训练。否则虽然模型权重接上了，但学习率、动量、Adam 的一阶二阶矩都会丢，训练曲线可能明显偏掉。

长任务的 checkpoint 应该存到远端存储，例如对象存储或共享文件系统，并限制保留数量，避免磁盘被打满。大模型场景如果使用 ZeRO、FSDP、Megatron 等，还要按框架提供的分片 checkpoint 方式保存和恢复。

## 混合精度训练中 FP16 和 FP32 怎么转换和同步？

混合精度训练里，FP16 和 FP32 通常不是简单地全程互转，而是各司其职。常见做法是前向和反向的大部分计算用 FP16，利用 Tensor Core 提升吞吐、降低显存；但模型主权重保留一份 FP32 master weight，用来做参数更新，保证数值精度。

一次训练迭代大致是：FP32 master weight cast 成 FP16 weight 参与 forward；loss 计算后，为避免 FP16 梯度下溢，先做 loss scaling，把 loss 放大再 backward。反向得到梯度后，优化器更新前把梯度 unscale 回真实尺度，并检查是否出现 inf/NaN。如果梯度正常，就把梯度转换或累积到 FP32 精度，用它更新 FP32 master weight；更新完成后，再同步一份新的 FP16 权重给下一轮 forward 使用。如果发现溢出，本轮 update 会跳过，并降低 loss scale。

分布式训练时，同步的一般是梯度而不是参数。DDP all-reduce 前后要保持梯度 bucket 的 dtype 策略统一。有些框架用 FP16 通信减少带宽，也可以用 FP32 all-reduce 提高稳定性。关键点是：计算可以 FP16，通信可以按策略压缩，但优化器状态、主权重和关键归约最好保持 FP32。

## 分布式训练时某个节点挂了，怎么让训练继续？

分布式训练里单个节点挂掉，核心是把故障恢复设计成训练链路的一部分，而不是事后手工补救。

首先要做可靠的周期性 checkpoint，保存模型参数、优化器状态、学习率调度器状态、随机数种子、当前 global step，以及数据采样器进度。checkpoint 一般写到共享存储或对象存储，并采用临时文件加原子重命名，避免写到一半的坏 checkpoint 被加载。

其次要支持从最近一次 checkpoint 自动拉起。节点挂了后，由调度系统重新拉起 worker，或者在支持弹性训练的框架下缩容继续训练。恢复时所有 rank 重新初始化通信组，从 checkpoint 读取一致的 step 和状态，再继续跑。为了少浪费进度，checkpoint 间隔要结合训练速度和存储成本设置，可以每 N 分钟或每 N 个 step 保存一次，只保留最近几个版本。

如果使用 PyTorch，可以结合 `torchrun` elastic、Kubernetes Job、Slurm requeue 或框架层 fault tolerance。数据层也要恢复 `DistributedSampler` 的 epoch、shuffle seed 和已消费样本位置，否则可能重复训练或漏样本。

还要配置通信超时和健康检查，发现某个 rank OOM、失联或 all-reduce 卡死时快速失败并触发重启，而不是所有节点一直等待。

## 多机多卡训练时数据加载成为瓶颈怎么优化？

多机多卡训练里数据加载成为瓶颈，先要用 profiler 或日志确认瓶颈在 I/O、解码、数据增强还是通信同步，而不是直接调参数。

常见优化有几类。第一，数据格式要适合顺序读取和分片读取，把大量小文件转成 TFRecord、WebDataset、LMDB、Parquet 等，减少元数据访问和随机 I/O。第二，把数据放到离计算更近的位置，比如本地 NVMe、分布式缓存，避免所有节点反复从远端对象存储拉数据。第三，合理设置 `num_workers`、`prefetch_factor`、`pin_memory`、persistent workers，让 CPU 预取和 GPU 计算重叠。

如果有复杂数据增强，要区分哪些能提前离线处理。在线增强太重时，可以用 DALI、GPU transform，或者降低增强复杂度。分布式场景还要保证每个 rank 读取不同 shard，避免重复读取和数据倾斜，同时 shuffle 不能破坏吞吐太多。

最后要监控 GPU 利用率、batch 等待时间、DataLoader 时间占比和存储带宽。如果 GPU 经常空等，说明还要继续优化输入流水线；如果 GPU 已经打满，就不应该过度优化数据层。

## 多个人提交训练任务抢 GPU，怎么公平调度？

多个人抢 GPU 时，不适合只做简单 FIFO，而应该按“配额 + 优先级 + 可抢占”来调度。

首先按团队、项目或用户设置 GPU 配额，比如每人最多同时占用 N 张卡，避免一个人提交大量任务把集群打满。其次任务进入队列时带上优先级、预计资源、最长运行时间和是否可抢占。线上紧急实验、阻塞发布的任务优先级更高，普通探索任务排在后面。

公平性上可以用 weighted fair sharing 或 DRF。核心思想是看每个用户已经消耗了多少 GPU 时间、显存和卡数，谁用得少谁优先拿资源，而不是只看提交顺序。对长任务可以做 checkpoint 后抢占，把资源让给等待过久的高优先级任务。

工程上还要配套几个机制：任务必须声明资源需求；空闲 GPU 自动回收；超时任务自动停止或降级；支持排队状态可观测；调度日志可审计。训练平台可以基于 Kubernetes + Volcano、Kueue，或 Slurm 实现队列、配额和抢占。

目标是保证资源利用率高，同时让每个人在长期窗口内拿到接近自己配额的 GPU 时间，避免短任务饿死，也避免大用户长期垄断。

## 动态批处理和静态批处理怎么选？

推理服务里，动态批处理和静态批处理的选择核心看流量形态、延迟约束和实现复杂度。

静态批处理适合离线推理、流量稳定、请求 shape 比较一致的场景，例如固定 batch size 做图片分类、特征抽取、离线打分。它的优点是实现简单、吞吐稳定、显存规划容易，TensorRT 这类优化也更好做；缺点是在线请求不一定刚好凑满 batch，低峰期会浪费等待时间，高峰期又可能排队明显。

动态批处理更适合在线服务，尤其是请求到达不均匀但希望提高 GPU 利用率的场景。它会在很短的时间窗口内收集请求，例如 2 到 10ms，凑成 batch 后一起推理。优点是兼顾吞吐和延迟，低流量时可以快速放行，高流量时自动合批；缺点是调度逻辑更复杂，需要处理超时、不同输入长度、优先级、取消请求、显存峰值等问题。

工程上一般是离线任务优先静态批处理，在线 API 优先动态批处理，并设置最大 batch size 和最大等待时间。大模型推理还要结合 continuous batching、KV cache 管理和请求长度分桶，否则简单合批可能被长请求拖慢。

## 几十个模型版本怎么高效存储和快速检索？

几十个模型版本不要直接用文件夹手工管理，应该当成模型制品管理问题。

存储上，模型文件本体放对象存储或制品库，例如 S3、MinIO、Nexus、MLflow Model Registry；数据库只存元数据，包括模型名、版本号、训练数据版本、代码 commit、超参数、指标、框架版本、文件 hash、存储地址和状态。大模型文件可以按 hash 去重，能复用的 tokenizer、配置、特征处理文件单独存，避免每个版本完整复制一份。

版本管理上，采用语义化版本或递增版本号，并区分 `dev`、`staging`、`prod`、`archived` 状态。线上服务不要写死路径，而是通过模型注册表按标签加载，比如“当前生产版本”“灰度版本”“回滚版本”。

检索上，关键是元数据索引。常见查询包括按任务、指标、数据集、时间、负责人、线上状态查找模型，所以这些字段要建索引。模型评估指标、训练参数可以结构化存储，方便筛选“最近一个月 AUC 最高且延迟小于 20ms 的版本”。

加载优化上，热版本放本地缓存或镜像层，冷版本留在对象存储；服务启动时预拉取生产模型，支持懒加载和缓存淘汰。

## A/B 测试怎么按比例把流量分给不同模型版本？

做两个模型的 A/B 测试，核心是稳定、可控、可回滚的流量分流机制。

工程上一般不对每次请求简单 `rand()`，因为同一个用户可能一会儿进 A、一会儿进 B，影响体验和指标判断。更常见的做法是按用户 ID、设备 ID、账号 ID 或请求中的稳定标识做哈希，然后映射到固定区间，例如 `0-9999`。如果希望 10% 流量进入新模型 B，就让哈希值落在 `0-999` 的用户走 B，其余走 A；如果要 50/50，就各占一半。这样同一个用户在实验期间稳定命中同一个版本。

分流比例最好通过配置中心或实验平台控制，而不是写死在代码里。上线时可以先从 1%、5%、10% 逐步放量，观察延迟、错误率、成本和核心业务指标，没问题再扩大比例。如果发现异常，可以快速把 B 的比例调成 0，实现回滚。

还要注意分流维度和统计口径一致，比如按用户分流就按用户维度分析，避免请求级随机导致样本污染。实验日志里要记录实验名、版本、分流桶、模型版本和关键输入输出，方便后续指标分析和问题排查。

## 设计完整模型平台，从训练到部署到监控怎么考虑？

完整模型平台可以拆成数据、训练、模型管理、部署、监控和治理几层。

数据层接入离线数仓、实时日志和业务数据库，统一进入数据湖或特征平台。这里要保证数据版本、口径一致性、权限控制和可追溯性，训练集、验证集、特征工程逻辑都应该能复现。

训练层提供标准化训练环境，例如基于 Kubernetes 的 GPU 调度，支持 PyTorch、TensorFlow 等框架。训练任务通过 Airflow、Kubeflow、Argo Workflows 等流水线编排，完成数据准备、训练、评估、模型产出。超参搜索、分布式训练、断点续训和资源隔离也要考虑。

模型管理层负责模型注册、版本管理、指标记录和审批流。每个模型版本都要记录代码版本、数据版本、训练参数、评估指标和制品地址。只有通过离线评估和必要审核后，才能进入发布阶段。

部署层要同时支持在线推理和离线批处理。在线服务可以用 Kubernetes 加 Triton、TorchServe 或自研推理服务，支持灰度发布、A/B 实验、自动扩缩容、回滚和多模型路由。低延迟场景还要考虑模型压缩、量化、缓存和特征预计算。

监控层不仅监控 CPU、GPU、延迟、QPS、错误率，也要监控预测分布、特征漂移、数据缺失、线上效果和业务转化。如果发现数据漂移或效果下降，要触发告警，并推动重新训练或回滚。平台治理还包括权限、审计、成本管理、合规和可观测性。

## 多城市机房里模型文件怎么同步？用户就近访问怎么快？

多城市机房里，模型文件同步可以按“中心仓库 + 分发缓存 + 版本化发布”设计。训练产物先进入统一模型仓库或对象存储，例如 S3、OSS、MinIO，文件按 `model_name/version/checksum` 管理，元数据记录灰度状态、依赖配置和回滚版本。

各城市机房通过异步复制、定时拉取或消息触发下载，把模型同步到本地对象存储或本地 SSD 缓存。发布时不要直接覆盖文件，而是先下载新版本、校验 checksum，预热完成后再切换软链接或注册中心版本，避免线上读到半成品。

一致性上，模型通常不要求强一致，更适合最终一致。关键是保证同一实例加载的是完整且可追溯的版本。如果有多模型依赖，需要用 manifest 固定一组版本，避免 A 城市加载新模型、旧 tokenizer 这类问题。

用户就近访问主要靠全局流量调度。入口可以用 GeoDNS、Anycast、全局负载均衡或云厂商 GTM，把用户解析到最近、健康、容量足够的机房。机房内再经过本地负载均衡到推理服务。静态模型文件可以走 CDN 或边缘缓存，但推理请求本身通常需要路由到有 GPU 资源的区域。

## 模型压缩、量化流程怎么自动化？

模型压缩、量化的自动化，核心是把它做成可回归、可灰度、可验收的流水线，而不是手工调参。

通常先固定基线模型、评测集和指标，例如精度、延迟、吞吐、模型大小、显存占用。然后由压缩策略模块自动生成候选方案，包括剪枝比例、蒸馏配置、INT8/INT4 量化方式、校准数据规模等。

工程上可以拆成几个阶段：

1. 自动导出模型，例如 ONNX、TorchScript 或 TensorRT 可接受格式。
2. 做结构化剪枝或蒸馏训练，并记录每个实验的超参和指标。
3. 执行量化，常见是 PTQ 先快速探索，如果精度掉得明显，再走 QAT。
4. 在目标硬件上自动 benchmark，而不是只看离线精度。

最后需要准入规则，例如精度下降不超过 1%，P99 延迟下降 30%，模型体积减少 50% 才允许进入候选。所有结果进入实验追踪系统，并自动产出对比报告。上线时要支持回滚和灰度，因为压缩模型可能在长尾样本上退化。

## 联邦学习只共享模型更新不共享数据，怎么保证安全？

联邦学习“不共享原始数据”只能降低数据暴露面，并不等于天然安全，因为模型更新本身也可能泄露信息，例如梯度反推、成员推断、投毒攻击等。工程上要从通信、聚合、隐私和鲁棒性几层一起做。

首先，通信链路要加密，使用 TLS、双向认证、签名和更新完整性校验，保证更新在传输中不被窃听或篡改。其次，服务端最好采用安全聚合协议，让聚合方只能看到多方更新的总和，不能看到单个参与方的梯度或参数变化。

如果对隐私要求更高，可以引入差分隐私，对本地更新做裁剪并加入噪声，限制单个样本对模型更新的影响，降低反推训练数据的风险。但差分隐私会影响模型精度，需要在隐私预算和效果之间权衡。

另外还要防恶意参与方。常见做法包括参与方身份准入、设备可信执行环境、更新范围校验、异常梯度检测、鲁棒聚合算法，例如 median、trimmed mean、Krum 等，避免少数恶意更新污染全局模型。

联邦学习的安全不是靠“不传数据”单点保证，而是靠安全通信、安全聚合、差分隐私、参与方认证和鲁棒聚合共同构成防护体系。

## 模型服务花了多少钱怎么算清楚？每个请求消耗多少资源？

算模型服务成本，不能只看 GPU 租金，要拆到可归因到请求的口径。通常先把总成本分层：计算成本，例如 GPU/CPU 实例、推理框架常驻开销；存储和网络成本，例如模型文件、日志、出入带宽；平台成本，例如网关、监控、队列、缓存；再加人工运维或折旧。

线上更常用的是先算单位时间成本，比如某个模型服务每小时花多少钱，再结合吞吐和利用率摊到请求上。每个请求消耗多少资源，核心看输入 token、输出 token、请求时延、GPU 显存占用、GPU 计算时间、批处理大小、是否命中缓存。

对于大模型，成本可以按 token 近似：单请求成本约等于输入 token 成本加输出 token 成本，再加固定调用开销。自建服务更常见按 GPU 时间摊销：请求成本约等于实例每秒成本乘以该请求占用的有效推理时间，再考虑 batch 分摊。例如一个 A100 服务每小时 20 元，每秒约 0.0056 元，如果某批请求推理 2 秒，batch 里有 8 个请求，那么粗略每个请求计算成本是 `0.0056 * 2 / 8`。

工程上必须做请求级埋点：记录模型名、版本、租户、输入输出 token、首 token 延迟、总延迟、batch id、缓存命中、错误重试次数、资源池信息。然后在离线账单里把资源成本、QPS、GPU 利用率和请求日志 join 起来，形成按模型、业务、用户维度的成本报表。

## 怎么自动评估模型效果？除了准确率还要看什么？

自动评估模型效果，不是只跑一次测试集准确率，而是建立固定评估流水线：数据集版本固定、预处理逻辑固定、模型版本固定，然后在验证集、测试集或线上回放数据上批量推理，自动产出指标、报告和对比结论。这样每次模型迭代后，都能和上一版基线做可复现比较。

除了准确率，还要看任务类型。分类任务常看 precision、recall、F1、AUC、PR-AUC、混淆矩阵。风控、医疗、内容安全这类场景，漏判和误判成本不同，只看 accuracy 容易误导。样本极不均衡时，一个模型全预测多数类也可能有很高准确率，更应该关注少数类召回率、宏平均 F1 等指标。

回归任务通常看 MAE、MSE、RMSE、MAPE、R²，同时分析误差分布，而不是只看平均误差。排序、推荐、搜索场景更关注 NDCG、MAP、MRR、Recall@K、CTR/CVR 等指标。

还要看工程侧指标：推理延迟、吞吐量、显存/内存占用、模型大小、稳定性、异常率、成本。最后做分层评估和鲁棒性评估，例如按用户类型、地区、设备、类别、长尾样本分别看指标，同时关注置信度校准、边界样本表现、数据漂移和线上 A/B 实验结果。

## 怎么防止别人用恶意输入攻击模型？

防止恶意输入攻击模型，不能只依赖模型本身足够聪明，工程上要做分层防护。

首先是输入侧治理。对用户输入做长度限制、格式校验、字符过滤和结构化解析，避免超长文本、异常编码、脚本片段、越权指令等直接进入模型。高风险场景可以加分类器或规则引擎，先识别提示注入、越狱、恶意代码、隐私探测等输入。

其次是权限隔离。模型不应该直接拥有数据库、文件系统、支付、发信等高权限能力。即使接入工具调用，也要通过后端做白名单、参数校验、权限检查和必要的人工确认。模型只负责生成意图，真正执行动作必须由受控服务完成。

第三是提示词和上下文隔离。系统指令、开发者指令、用户输入、检索内容要明确分层，不要把外部文档内容当成可信指令执行。RAG 场景下尤其要防止文档里的“忽略之前指令”影响模型行为。

第四是输出侧防护。对模型输出做安全检测，拦截敏感信息、危险操作建议、恶意代码、越权数据等内容。重要业务还要有审计日志、速率限制、异常监控和灰度策略。最后持续做红队测试和对抗样本评估，把常见攻击样本加入测试集。

## 一个平台给多个客户用，怎么保证客户之间完全隔离？

多租户平台里，客户隔离要从数据、权限、计算资源和运维流程几层一起做，不能只靠业务代码里加 `tenant_id`。

首先是数据隔离。常见方案有共享库共享表、共享库分表、独立库。隔离要求越高，越倾向独立库或至少独立 schema；如果使用共享表，所有查询必须强制带租户条件，并在 ORM、DAO 或数据库层做统一约束，避免开发漏写。高安全场景还可以使用行级安全策略、独立加密密钥和独立备份。

其次是身份和权限隔离。用户登录后要绑定租户上下文，后端所有接口都基于当前租户鉴权，不能相信前端传来的租户 ID。管理员权限也要区分平台管理员和客户管理员，防止越权访问。

第三是资源隔离。计算、队列、缓存、对象存储、日志索引都要带租户维度。重要客户可以使用独立实例、独立队列或独立命名空间，避免一个客户的大流量影响其他客户。

最后是工程保障。要有自动化测试覆盖越权访问场景，日志和审计能追踪租户访问行为，发布、迁移、备份恢复也必须按租户维度验证。客户隔离是租户上下文贯穿认证、授权、数据访问、资源调度和运维审计的完整体系。

## 客户想自定义计算操作，怎么让他们安全扩展？

客户自定义计算应该设计成受控扩展点，而不是让客户直接改核心代码或执行任意脚本。

首先定义清晰的插件接口，例如输入输出 schema、支持的数据类型、错误码、超时约束和版本号。客户只能实现这个接口，平台负责调度、校验和生命周期管理。这样核心系统和扩展逻辑之间有稳定边界。

安全上，把自定义计算放到沙箱环境里运行，比如独立进程、容器或 WASM runtime，限制 CPU、内存、磁盘、网络和执行时间。默认不允许访问内部服务、文件系统和环境变量，确实需要的能力通过白名单授权。

其次要做输入输出校验。执行前校验参数、类型和范围，执行后校验结果格式，避免脏数据进入主流程。对异常、超时、资源耗尽要有统一降级策略，比如返回明确错误、跳过该步骤或使用默认实现，不能拖垮主链路。

还需要可观测性和治理能力。每个扩展都要记录调用日志、耗时、资源消耗、错误率，并支持灰度发布、版本回滚和禁用开关。高风险客户代码可以增加审核、签名和依赖白名单。

## 自动扩缩容怎么避免频繁伸缩导致服务抖动？

自动扩缩容要避免频繁伸缩，核心是给扩缩容决策加稳定性约束，不要只盯瞬时指标。

首先，指标选择要合理。不要只看某一秒的 CPU 或 QPS 峰值，通常用滑动窗口、平均值、P95/P99 延迟、队列长度等综合判断。比如连续 3 到 5 分钟 CPU 都超过 70% 才触发扩容，而不是一次抖动就扩。

其次，要设置冷却时间和滞后区间。扩容后给新实例预热、注册、接流量留出时间，在这段冷却期内不重复扩容。缩容更要保守，比如扩容阈值是 CPU > 70%，缩容阈值可以设成 CPU < 35% 且持续一段时间，避免在 60% 到 70% 附近来回横跳。

第三，缩容要慢于扩容。扩容可以相对激进，保证容量兜底；缩容则分批、限速执行，例如每次最多缩 10% 或 1 台，并结合连接数、请求完成情况做优雅下线，避免正在处理请求的实例被直接摘掉。

第四，要考虑业务周期和容量下限。有明显高峰的业务可以结合定时扩容提前预热，而不是完全依赖实时指标。同时设置最小实例数，保证基础容量和容灾能力。

最后还要配合限流、熔断、队列削峰和负载均衡健康检查。经验上是：扩容决策要及时，缩容决策要谨慎，指标要看趋势，执行要有冷却和优雅退出。
