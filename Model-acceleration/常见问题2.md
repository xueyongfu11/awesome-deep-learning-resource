## 对 AI Infra 这个方向的理解

我理解 AI Infra 是连接算法和底层硬件/系统的工程层，目标是让模型训练、推理、部署更高效、更稳定、更易用。它不只是搭环境或写脚本，而是覆盖数据管道、训练框架、分布式调度、算子优化、模型压缩、推理服务、监控运维等完整链路。

从训练侧看，AI Infra 需要解决 GPU 利用率、显存不足、通信瓶颈、容错恢复、任务调度等问题；从推理侧看，要解决低延迟、高吞吐、弹性扩缩容、模型热更新和成本优化。工程上比较核心的是性能意识和系统化定位能力，比如能判断瓶颈在数据、计算、通信还是调度层。好的 AI Infra 能让算法同学更快迭代，让业务以更低成本稳定使用模型能力。

## 混合精度训练的原理是什么？FP16、BF16 和 FP32 如何转换与同步？

混合精度训练的核心是用 FP16 或 BF16 加速大部分矩阵计算和减少显存，同时保留部分关键计算的数值稳定性。GPU Tensor Core 对 FP16/BF16 计算有明显加速，参数、激活和梯度占用也会下降。一般做法是前向和反向的主要算子用低精度，权重更新保留 FP32 master weights，避免长期累积误差。

一次 FP16 训练迭代通常是：将 FP32 master weights 转为 FP16 参与 forward；先放大 loss 再 backward，避免小梯度下溢；优化器更新前将梯度 unscale，并检查 inf/NaN。如果梯度正常，就用 FP32 梯度或累积结果更新 FP32 master weights，再同步新的 FP16 权重供下一轮计算使用；如果溢出，则跳过本轮更新并降低 loss scale。BF16 的指数位与 FP32 接近，数值范围更好，通常不需要 loss scaling，但有效精度位更少。

实现时要注意 LayerNorm、Softmax、Loss、梯度归约等敏感算子可能需要 FP32。分布式训练通常同步梯度而不是参数，DDP all-reduce 前后要统一梯度 bucket 的 dtype 策略：FP16/BF16 通信能减少带宽，FP32 归约稳定性更好。优化器状态、主权重和关键归约通常保留 FP32，还要保证不同卡的 overflow 状态同步以及 checkpoint 的 dtype 兼容性。常见错误是盲目全 FP16 导致 loss NaN。



## Tokenizer 编码逻辑

Tokenizer 的作用是将文本转换为 token id，常见算法包括 BPE、SentencePiece、Unigram、WordPiece 和 Byte-level BPE。

在高并发服务中，Tokenizer 多在 CPU 侧运行，可能成为 GPU 前的隐性瓶颈。

实际应用中主要存在以下瓶颈：

- 长 prompt、高 QPS 下 CPU 编码耗时高。
- 多轮对话重复编码历史上下文。
- 模板拼接、特殊 token、stop words、流式 detokenize 阻塞主流程。
- Tokenizer 串行导致 GPU 等输入。

常见实现和框架包括 Hugging Face Tokenizers（Rust）、SentencePiece、tiktoken、vLLM 异步 tokenizer、TGI router 和 SGLang runtime。

可以从以下几个方面进行优化：

- 异步 tokenizer 线程池：请求接入、编码、推理调度流水化。
- Prompt token cache：缓存 system prompt、chat template、RAG 模板的编码结果。
- 批量 tokenize：离线批处理或高吞吐场景合并编码。
- 增量 detokenize：避免每轮从完整序列重新 decode。
- Stop words token-level 匹配：减少字符串级扫描。

不同技术的收益和代价如下：


| 技术            | 收益             | 代价                     |
| --------------- | ---------------- | ------------------------ |
| 异步 tokenizer  | 降低 GPU 等待    | 线程池与调度复杂度增加   |
| Tokenizer cache | 减少重复编码     | 需要缓存淘汰和多租户隔离 |
| 批量编码        | CPU 吞吐更高     | 可能增加排队延迟         |
| 增量 detokenize | 降低流式输出开销 | 需处理 Unicode 边界      |

落地时需要监控 `tokenize_latency`、`detokenize_latency`、CPU 使用率和 GPU idle time。RAG/Agent 场景尤其需要缓存固定 prompt 与历史前缀。

## 前后处理如何设计成可编排流水线并优化算子性能？

前处理包含 padding/packing、attention mask、position ids 和 batch metadata。后处理包含 temperature、top-k/top-p、repetition penalty、softmax、sampling、stop condition 和 detokenize。

工程设计上适合采用可编排流水线。每个步骤实现统一的 `process(context) -> context` 接口，`context` 保存原始输入、清洗结果、模型输出、格式化结果和 trace 信息。整体可分为三层：

1. 步骤接口：约定输入输出、异常处理、是否可跳过和是否允许短路。
2. Pipeline 编排器：按顺序执行，支持条件执行、失败重试和短路返回。
3. 配置层：用 YAML、JSON 或数据库配置步骤、顺序和参数。

预处理可以配置敏感词过滤、归一化、脱敏、prompt 拼装、padding 和 mask；后处理可以配置采样、JSON 解析、字段校验、格式化和兜底文案。每个步骤应记录耗时、输入输出摘要和错误码，并通过单元测试与回放测试保证调整顺序时不破坏链路。

这一阶段主要存在以下瓶颈：

- Decode 每 token 都采样，logits processor 开销被输出长度放大。
- GPU logits 拷贝到 CPU 采样会导致 CPU-GPU 同步。
- 动态 batch 状态变化频繁，metadata 更新成本高。
- Padding 浪费计算。

典型实现包括 vLLM GPU sampling、TensorRT-LLM optimized sampling kernels、TGI continuous batching 和 SGLang structured decoding。

常见优化方式包括：

- GPU sampling：logits 在 GPU 上完成 softmax/top-k/top-p/采样。
- Fused sampling kernel：融合 temperature、penalty、top-k/top-p、随机采样。
- Guided decoding mask 下沉到 GPU 或高效状态机。
- Packing / Continuous Batching 减少 padding 与 batch 空洞。

落地时需要关注每个 token 的 CPU-GPU memcpy 次数、采样 kernel 数和 logits processor 耗时。结构化输出或 JSON schema 会显著增加约束解码成本。

## 常规算子加速优化

核心算子包括 GEMM、RMSNorm/LayerNorm、RoPE、Softmax、Residual add、Activation、Quantize/Dequantize、KV Cache 读写和 AllReduce/AllGather。

主要瓶颈包括：

- MLP/Linear 的 GEMM 计算量大。
- Norm、RoPE、elementwise 算子 FLOPs 低但访存频繁。
- 小 batch decode 的 kernel launch 开销明显。
- 量化推理中的 dequant 可能抵消收益。

典型实现包括 cuBLAS/cuBLASLt、CUTLASS、Triton、TensorRT-LLM、FasterTransformer、vLLM custom ops 和 FlashInfer。

优化方向主要包括：

- 算子融合：RMSNorm+Residual、QKV+RoPE+KV 写入、Dequant+GEMM、Linear+Bias+Activation。
- 高性能 GEMM：Tensor Core、FP16/BF16/FP8、INT8/INT4、合适 tile 和 layout。
- Norm 优化：warp-level reduction、向量化加载、fused residual。
- RoPE 优化：sin/cos cache、RoPE 与 Q/K projection 或 cache 写入融合。
- 量化 kernel：避免先完整 dequant 再 GEMM，优先 fused dequant matmul。

Prefill 更接近大矩阵计算，Tensor Core 利用率高。Decode 每步处理的 token 较少，更依赖 batching、算子融合、KV cache layout 和低延迟通信。



## 标准 Attention 瓶颈

标准自注意力：

```text
Attention(Q,K,V)=softmax(QK^T/sqrt(d)+mask)V
```

对序列长度 `N`，计算复杂度约 `O(N^2*d)`，显存中间量 `O(N^2)`。长上下文 prefill 时，attention score 矩阵和 softmax 中间结果带来巨大 HBM 读写。

## FlashAttention

FlashAttention 按 block 加载 Q/K/V，在 SRAM/shared memory 中分块计算，并使用 online softmax 累积结果，不把完整 attention matrix 写入 HBM。

```text
传统：QK^T -> HBM -> softmax -> HBM -> PV -> HBM
FlashAttention：分块 Q/K/V -> 片上 online softmax -> 只写最终输出
```

这种方式减少了 HBM IO 和中间显存占用，不改变 Attention 的数学结果，也不降低理论上的 `O(N^2)` FLOPs。

典型实现包括 FlashAttention 1/2/3、PyTorch SDPA、xFormers、TensorRT-LLM attention kernel、FlashInfer 和 vLLM backend。

使用时需要注意以下权衡：

- 长序列 prefill 收益最大，短序列收益有限。
- 对 head_dim、dtype、mask、GPU 架构有约束。
- FlashAttention-3 更偏 Hopper/H100 异步能力与 FP8 优化。

## PagedAttention

PagedAttention 将 KV Cache 切成固定大小的 block，并用 block table 将逻辑 token 映射到物理 KV block，类似虚拟内存分页。

它主要解决变长请求的显存碎片、KV 预分配浪费、Continuous Batching 下的 cache 动态增长，以及 prefix sharing 和 beam search 复用问题。

典型实现包括 vLLM PagedAttention、TensorRT-LLM paged KV cache、FlashInfer paged attention 和 SGLang。

需要注意的是，block table 间接寻址会增加 kernel 复杂度。block size 太小会产生较多元数据，太大则会增加内部碎片。

## MQA / GQA

MHA 中每个 Q head 都有独立的 K/V，MQA 中所有 Q heads 共享一组 K/V，GQA 中多个 Q heads 共享一组 K/V。

Decode 阶段的 KV 读取量与 `num_kv_heads` 成正比，因此减少 KV heads 可以显著降低显存占用和 HBM 带宽需求。

这类结构必须在模型训练时得到支持，直接从 MHA 修改结构通常无法做到无损转换。



## Attention 与 RMSNorm 融合

将 RMSNorm、Residual、QKV projection、RoPE、KV cache 写入和 Attention 前处理等相邻操作融合，可以减少中间张量写入显存。

这种方式可以降低 HBM 读写、kernel launch 和 decode 小 batch latency，同时提高 L2/cache locality。

典型实现包括 TensorRT-LLM fused kernels、Transformer Engine、Triton RMSNorm/RoPE kernels、FlashAttention 和 vLLM custom ops。

融合越深，寄存器压力和代码复杂度越高，并且通常需要针对 hidden size、head_dim、dtype 和 GPU 架构进行特化。

## 计算与显存读取融合

计算与显存读取融合的核心是一次读取数据后完成多步计算，避免 `read -> op -> write -> read -> op` 的 HBM 往返。

典型融合方式包括 RMSNorm+Residual、Dequant+GEMM、QKV+RoPE+KV 写入、SwiGLU activation 和 Attention score+mask+softmax。

落地时应优先融合 memory-bound 算子，并使用 Nsight Systems/Compute 观察 kernel 数、HBM 带宽和 occupancy。



## 如何手推 Transformer 一次前向传播的 FLOPs？

先统一口径：下面把一次乘法和一次加法合计为 `2 FLOPs`，忽略 bias、LayerNorm、Softmax、激活函数和位置编码等低阶项。设：

- `L`：Transformer 层数，`B`：batch size，`S`：序列长度。
- `d`：hidden size，`d_ff=r×d`：FFN 中间维度，经典配置中 `r=4`。
- `V`：词表大小。

以下先讨论标准 Dense Decoder-only Transformer：使用 MHA 和包含两个线性层的普通 FFN。单层各部分的 FLOPs 为：

推导只需要一条矩阵乘法规则：形状为 `[m,k]` 和 `[k,n]` 的两个矩阵相乘，会生成 `m×n` 个输出元素；每个输出元素需要长度为 `k` 的点积，即约 `k` 次乘法和 `k` 次加法。因此共有 `m×n×k` 次乘加，也就是约 `2mnk FLOPs`。下面把 batch 和 token 两个维度展平，将 hidden state 记为 `X:[BS,d]`。

- **Q、K、V 投影**：三者都是 `X:[BS,d]` 乘以权重 `[d,d]`。一次投影为 `2×BS×d×d=2BSd²`，三次合计 `6BSd²`。
- **计算 `QK^T`**：设 head 数为 `h`、单个 head 维度为 `d_h=d/h`。每个 head 执行 `[S,d_h]×[d_h,S]`，成本为 `2S²d_h`；乘上 `B` 个样本和 `h` 个 head，得到 `2BhS²d_h=2BS²d`。
- **计算 `Attention×V`**：每个 head 执行 `[S,S]×[S,d_h]`，成本同样为 `2S²d_h`；汇总 batch 和所有 head 后也是 `2BS²d`。因此两个 Attention 矩阵乘合计 `4BS²d`。
- **Attention 输出投影**：完成 `Attention×V` 后，将所有 head 的结果拼接为 `[BS,h×d_h]=[BS,d]`，再乘输出权重 `[d,d]`，所以是 `2BSd²`。该结果随后进入残差连接。
- **FFN 两个线性层**：第一层 `[BS,d]×[d,d_ff]` 为 `2BSd×d_ff`，第二层 `[BS,d_ff]×[d_ff,d]` 也是 `2BSd_ff×d`，合计 `4BSd×d_ff`。

整理后得到：

```text
Q、K、V 投影：3 × 2BSd² = 6BSd²
QK^T 和 Attention×V：2BS²d + 2BS²d = 4BS²d
Attention 输出投影：2BSd²
FFN 两个线性层：2BSd×d_ff + 2BSd_ff×d = 4BSd×d_ff
```

因此单层为：

```text
F_layer ≈ 8BSd² + 4BS²d + 4BSd×d_ff
        = (8+4r)BSd² + 4BS²d
```

当 `r=4` 时：

```text
F_layer ≈ 24BSd² + 4BS²d
```

如果所有位置都计算词表 logits，完整前向约为：

```text
F_forward ≈ L×[(8+4r)BSd² + 4BS²d] + 2BSdV
```

输入 embedding 是查表，通常不计矩阵乘法 FLOPs。推理框架如果只为最后一个位置计算 logits，则最后一项由 `2BSdV` 降为 `2BdV`。

常见的 `F_forward≈2NBS` 近似来自“每个参数对每个 token 参与一次乘加”，适合快速估算参数矩阵乘法；它没有显式体现 Attention 的 `4LBS²d`，而且是否准确计入 embedding/LM Head 取决于权重共享方式和实际计算了多少位置的 logits。长上下文时应使用展开公式。

对于 causal attention，理论上的有效位置是下三角，若 kernel 能跳过被 mask 的区域，Attention 两项可由约 `4BS²d` 降到约 `2BS²d`。普通稠密矩阵乘实现仍可能执行完整方阵计算；FlashAttention 的核心收益首先是减少 HBM 读写和中间张量，并不意味着 FLOPs 必然按同样比例下降。

使用 KV Cache 逐 token decode 时，本次 query 长度为 1，历史上下文长度为 `T`。每生成一个 token 的近似 FLOPs 为：

```text
F_decode/token ≈ L×[(8+4r)Bd² + 4BTd] + 2BdV
```

这说明 prefill 的 Attention 随序列长度近似二次增长，而带 KV Cache 的单 token decode 随历史长度 `T` 线性增长。GQA/MQA 会减少 K/V 投影成本；SwiGLU 有三个权重矩阵，均应按实际投影维度重新代入，不能直接套用普通 FFN 的系数。

以 GPT-2 Small 为例，取 `B=1`、`L=12`、`d=768`、`S=1024`、`V=50257`、`r=4`。若所有位置都计算 logits，完整序列一次前向约为 `291.6 GFLOPs`；若推理 prefill 只计算最后一个位置的 logits，则约为 `212.7 GFLOPs`。因此讨论 FLOPs 时必须同时说明计数口径、是否利用 causal 三角、是否计算全位置 logits，以及是在做 prefill 还是 decode。

### Llama-2-70B 在 A100 上生成一个 token 需要多久？

假设模型使用 FP16，`batch size=1`，A100 的 FP16 Tensor Core 峰值算力约为 `312 TFLOPS`、HBM 带宽约为 `2 TB/s`，并且已经完成 prefill。这里估算的是使用 KV Cache 逐 token decode 的相邻 token 延迟，不是包含整段 prompt 处理时间的首 token 延迟。

Llama-2-70B 约有 700 亿参数。每个参数对当前 token 大致参与一次乘加，所以参数矩阵乘法的计算量约为：

```text
F_decode/token ≈ 2N
               ≈ 2 × 70B
               ≈ 140 GFLOPs
```

在完全达到峰值算力的理想情况下，计算时间下限为：

```text
T_compute ≈ 140 GFLOPs / 312 TFLOPS
          ≈ 0.45 ms
```

FP16 每个参数占 2 Bytes，模型权重约为：

```text
M_weight ≈ 70B × 2 Bytes ≈ 140 GB
```

小 batch decode 时权重复用率低，每生成一个 token 基本需要从 HBM 读取一遍权重。若假设单卡能够容纳模型，理想访存时间为：

```text
T_memory ≈ 140 GB / 2 TB/s ≈ 70 ms
```

`70 ms` 远大于 `0.45 ms`，因此该场景明显是显存带宽瓶颈，而不是计算瓶颈。等价地看，参数矩阵乘法的算术强度约为 `140 GFLOPs / 140 GB = 1 FLOP/Byte`，远低于 A100 约 `312 TFLOPS / 2 TB/s = 156 FLOPs/Byte` 的算力与带宽比。

但 FP16 权重约为 140 GB，单张 A100 80GB 实际放不下完整模型。至少需要两张 A100 做 Tensor Parallel。理想情况下，每张卡并行读取约 70 GB 权重：

```text
T_memory,2GPU ≈ 70 GB / 2 TB/s ≈ 35 ms/token
```

考虑实际 HBM 带宽利用率、Tensor Parallel 通信、KV Cache 读取、kernel launch、采样和推理框架调度开销，可以粗略估计为：

```text
约 40～60 ms/token，即约 17～25 tokens/s
```

这是量级估算，不是性能承诺；上下文长度、A100 型号与互联方式、推理框架、并行策略和量化方式都会改变结果。随着上下文增长，decode 还需要读取更多 KV Cache，并执行与上下文长度线性相关的 Attention 计算。

Prefill 与 decode 的区别是：每个参数对每个输入 token 大致参与一次乘加。长度为 `S` 的 prompt 在 prefill 中对应约 `2NS` 的参数矩阵计算，并有约 `4LBS²d` 的 Attention 计算；多个 token 可以复用一次加载的权重，更容易发挥矩阵乘和 Tensor Core 的效率，因此通常更偏计算密集。逐 token decode 每步只处理一个新 token，约为 `2N` 加上随历史长度线性增长的 Attention 计算，权重复用少，因此小 batch 时通常更偏显存带宽瓶颈。



## 如何手推 Transformer 单次前向推理的显存占用？

以下讨论标准 Dense Decoder-only Transformer 的推理态，不包含反向传播。设：

- `L`：Transformer 层数。
- `B`：batch size。
- `Q`：本次前向的 query token 数；prefill 时通常为输入长度，逐 token decode 时为 1。
- `T`：本次前向结束后的上下文长度，即每条序列已经缓存的 token 数。
- `d`：hidden size，`h`：Q head 数，`h_kv`：KV head 数，`d_h=d/h`：head dimension。
- `d_ff`：FFN 中间维度，`V`：词表大小。
- `b_p`、`b_kv`、`b_a`：参数、KV Cache、激活每个元素的字节数。FP32 为 4，FP16/BF16 为 2，INT8 约为 1，INT4 理论值约为 0.5。

理论峰值显存可以拆成：

```text
峰值显存 ≈ 参数显存 + KV Cache + 当前层激活峰值 + 运行时额外开销
```

### 1. 参数显存

对于 MHA/GQA，Q 和输出投影各有一个 `d×d` 矩阵，K、V 投影各有一个 `d×(h_kv×d_h)` 矩阵，因此每层 Attention 参数量为：

```text
N_attn = 2d² + 2d×h_kv×d_h
       = 2d²(1 + h_kv/h)
```

SwiGLU FFN 包含 gate、up、down 三个矩阵：

```text
N_ffn = 3d×d_ff
```

忽略 bias 和 `O(Ld)` 的 Norm 参数，模型总参数量约为：

```text
N_param ≈ L × [2d²(1 + h_kv/h) + 3d×d_ff] + c_vocab×Vd
M_param = N_param×b_p
```

输入 embedding 与 LM Head 权重共享时 `c_vocab=1`，不共享时为 2。经典 MHA、普通两层 FFN 且 `d_ff=4d` 时，常用近似为 `N_param≈12Ld²+Vd`。量化部署还要加入 scale、zero point、分组元数据等开销，实际值会略高于纯 bit 数的理论值。

### 2. KV Cache

每层、每个 token 都要保存 K 和 V，各自包含 `h_kv×d_h` 个元素，因此：

```text
M_KV = 2 × L × B × T × h_kv × d_h × b_kv
     = 2LBTd × (h_kv/h) × b_kv
```

MHA 中 `h_kv=h`，所以 `M_KV=2LBTd×b_kv`。GQA/MQA 则按 `h_kv/h` 的比例降低 KV 容量。每增加一个 token，整个 batch 的增量为：

```text
ΔM_KV/token = 2LBd × (h_kv/h) × b_kv
```

### 3. 激活峰值

推理时不需要为反向传播保存所有层的激活，而且不同层的临时 buffer 通常可以复用，所以激活峰值一般不直接乘 `L`。

主要中间张量包括：

- hidden state、Q/K/V、残差和归一化结果：量级为 `O(BQd)`。
- SwiGLU 中间张量：量级为 `O(BQd_ff)`。
- 普通 Attention 显式生成的 score/softmax：shape 为 `[B,h,Q,T]`，量级为 `O(BhQT)`。

用 `c_h`、`c_f`、`c_s` 表示算子融合、原地复用和实现细节带来的常数，普通 Attention 可估为：

```text
M_act,peak ≈ b_a × max(c_h×BQd, c_f×BQd_ff, c_s×BhQT)
```

FlashAttention 不把完整的 `[B,h,Q,T]` 矩阵写入 HBM，因此其激活峰值近似为：

```text
M_act,peak,flash ≈ b_a × max(c_h×BQd, c_f×BQd_ff)
```

如果返回所有位置的完整 logits，还要额外考虑 `[B,Q,V]`，即约 `BQV×b_logits`；只保留最后一个 token 的 logits 时则为 `BV×b_logits`。

### 4. Prefill 与 Decode

Prefill 时令 `Q=T=S`。普通 Attention 的临时激活包含 `O(BhS²)`，FlashAttention 则避免物化这个二次方大小的矩阵：

```text
M_prefill ≈ M_param
          + 2LBSd×(h_kv/h)×b_kv
          + M_act,peak
```

逐 token decode 时令 `Q=1`，`T` 为当前上下文长度：

```text
M_decode ≈ M_param
         + 2LBTd×(h_kv/h)×b_kv
         + b_a×max(c_h×Bd, c_f×Bd_ff, c_s×BhT)
```

因此，长序列 prefill 在普通 Attention 下容易被 `S²` 激活主导；decode 的当前 token 激活较小，通常由随上下文和并发数线性增长的 KV Cache 主导。

最终用于容量规划的公式是：

```text
M_peak ≈ N_param×b_p
       + 2LBTd×(h_kv/h)×b_kv
       + M_act,peak
       + M_logits
       + M_runtime
```

其中 `M_runtime` 包括 CUDA context、算子 workspace、通信 buffer、内存对齐、框架预留和 allocator 碎片。理论值不能直接当作可部署上限，工程上通常还需根据实测预留约 10%～20% 的安全空间。

## KV Cache 优化

基于上面的 KV Cache 公式，优化目标主要包括：

- 减容量：GQA/MQA、MLA、KV 量化、低秩压缩。
- 减读取：PagedAttention、FlashDecoding、Sliding Window。
- 强管理：分页、prefix cache、continuous batching、copy-on-write。

KV 量化或压缩可能影响模型质量，prefix cache 的收益依赖业务前缀重复率，Paged KV 则会增加间接寻址开销。

## 静态批处理、动态批处理与 Packing 怎么选？

Packing 将不同长度的序列紧凑拼接，并使用 `cu_seqlens`、offset、position ids 和 mask 防止序列之间互相 attend，从而减少 padding。

典型实现包括 FlashAttention varlen API、vLLM continuous batching、TensorRT-LLM in-flight batching 和 SGLang scheduler。

静态批处理适合离线推理、流量稳定、请求 shape 接近的场景，例如图片分类、特征抽取和离线打分。它实现简单、吞吐稳定、显存规划容易，也利于 TensorRT 等引擎优化；但在线低峰期不容易凑满 batch，高峰期又可能出现明显排队。

动态批处理更适合请求到达不均匀的在线服务。调度器在很短的窗口内收集请求，例如 2 到 10ms，达到最大 batch size 或等待上限后执行。它能兼顾延迟和吞吐，但需要处理请求超时、长度差异、优先级、取消和显存峰值。大模型在线推理通常进一步使用 continuous batching，在请求完成后立即补入新序列，避免整个 batch 等待最长请求。

落地时，离线任务可结合静态 batch、length bucketing 和 packing；在线任务可结合动态或 continuous batching、请求长度分桶和 KV Cache 管理。需要同时限制最大 batch size、最大等待时间和 token budget，并监控 padding ratio、有效 token 数、队列等待时间及长请求造成的队头阻塞。

## Paged Attention 分片

单卡内分片是将 KV 按 block 存储，以降低碎片和预留浪费。

多卡或多节点分片可以按 layer、head、sequence block 或并行策略切分 KV。

关键参数包括 block size、num_gpu_blocks、KV dtype、prefix cache、swap space 和 max_num_batched_tokens。



## 推理整体架构

典型 serving pipeline：

```text
Client/API -> Request Queue -> Tokenizer -> Scheduler
  -> Batch Builder -> Model Executor -> Sampler -> Detokenizer/Streaming -> Client
```

核心模块：

- API server / router：接入、鉴权、限流、OpenAI-compatible 协议。
- Tokenizer / preprocessor：文本编码、chat template、prompt cache。
- Scheduler：prefill/decode 队列、token budget、优先级、准入控制。
- Model executor：GPU kernels、并行策略、CUDA Graph、采样。
- KV Cache manager：block 分配、释放、prefix cache、swap/recompute。
- Streaming output：增量 detokenize、断连取消、stop condition。

## GPU 显存调度

显存组成：

```text
Model Weights + KV Cache + Activation/Temporary Buffers + CUDA Workspace
+ Communication Buffers + Runtime Metadata + Fragmentation/Reserved
```

KV Cache 是在线推理中变化最频繁、最容易引发 OOM 的部分。显存调度需要进行 request admission control，判断当前 KV block pool 是否能够容纳新请求。

常见调度策略包括：

- 预分配显存池，避免频繁 cuda malloc。
- Paged KV block allocator，降低外部碎片。
- 设置 `gpu_memory_utilization` 安全水位。
- 长请求限流、最大上下文限制、超时取消释放 KV。
- CPU swap 或 recompute 作为极端情况下的 preemption 策略。

典型框架包括 vLLM block manager、TensorRT-LLM KV Cache Manager、TGI token limit 和 SGLang Radix Cache。



## PD 分离架构（Prefill-Decode Disaggregation）

PD 分离架构将 Prefill 和 Decode 部署在不同的 GPU worker 或实例上：

```text
Client -> Router
       -> Prefill Worker: 处理长 prompt，构建 KV Cache
       -> KV Transfer / Cache Store
       -> Decode Worker: 持续生成 token
       -> Streaming Response
```

采用分离架构主要有以下原因：

- Prefill 是计算密集型，长 prompt 会占用大量 GEMM/Attention 资源。
- Decode 是低延迟、访存密集型，每 token 小步执行，对 ITL 敏感。
- 两者混跑容易互相干扰：长 prefill 抬高 decode P99，decode 小步又会切碎 prefill 吞吐。

这种架构主要解决以下瓶颈：

- 降低长 prompt 对流式 decode 的干扰。
- 可对 prefill 与 decode 使用不同硬件/并行度配置。
- 提高集群资源利用：prefill 池按 prompt token 吞吐扩缩，decode 池按并发输出扩缩。
- 支持长上下文/RAG 中 KV 预热与复用。

典型实现和技术方向包括 DistServe、Mooncake、SGLang PD disaggregation、vLLM 相关分离式 serving 方案，以及 LMCache 等 KV cache 传输或复用系统。

主要收益和代价如下：


| 维度     | 收益                      | 代价             |
| -------- | ------------------------- | ---------------- |
| 延迟隔离 | Decode P99 更稳定         | 架构复杂         |
| 资源配置 | Prefill/Decode 可独立扩缩 | 需要准确容量规划 |
| KV 复用  | 长 prompt 可预热          | KV 传输成本高    |
| 多机支持 | 支持大规模服务            | 网络成为新瓶颈   |

## 缓存传输流水线

PD 分离后，Prefill worker 生成的 KV Cache 必须传给 Decode worker。KV 传输可能很大，成为瓶颈。

缓存传输可以按照以下方式进行流水线化：

```text
Prefill chunk 1 完成 -> 传输 chunk 1 KV -> Decode 准备
Prefill chunk 2 计算中 -> 传输 chunk 2 KV
Prefill chunk 3 计算中 -> Decode 可开始部分调度
```

主要优化方向包括：

- Chunked KV transfer：按 layer/block/chunk 分批传。
- 计算通信重叠：prefill 下一个 chunk 时传上一个 chunk KV。
- RDMA / GPUDirect：减少 CPU 拷贝。
- KV 压缩/量化传输：FP8/INT8 KV，减少带宽。
- 拓扑感知：prefill 和 decode worker 尽量 colocate 或同机架。
- Cache locality：同一会话后续请求路由到持有 prefix KV 的 decode worker。

需要注意的是，如果 KV 传输速度低于 decode 消费速度，decode worker 就会等待。如果传输占满网络，还会影响跨节点 TP/EP 通信。



## 集群调度优化

集群层调度不同于单机 scheduler，需要考虑 worker 池、模型副本、GPU 拓扑、KV locality、租户优先级、冷热缓存。

服务层应把多张 GPU 看作统一资源池，通常每张卡对应一个 worker 进程，由前置 dispatcher 根据心跳和运行指标分发请求。新请求先过滤 OOM、进程异常、心跳超时或显存不足的 worker，再选择负载较低的候选。简单场景可以采用最少并发数或最短队列；请求长度差异较大时，应估算 token 数、显存需求和预计执行时间，例如：

```text
score = 当前队列预计耗时 + 显存压力权重 + 并发惩罚
```

调度时需要考虑以下输入：

- 请求 prompt/output 长度估计。
- 当前 prefill/decode worker 负载。
- KV Cache/Prefix Cache 命中位置。
- GPU 显存水位、KV block 剩余量。
- 网络拓扑、跨机带宽、节点健康。
- 租户 quota、优先级、SLA。

常见调度策略包括：

- 负载均衡：按 queue length、running tokens、KV blocks、GPU utilization 分配。
- Cache-aware routing：有 prefix KV 的 worker 优先，减少重复 prefill。
- Length-aware routing：长 prompt 送到 prefill 能力强的 worker，短请求走低延迟路径。
- SLA-aware scheduling：实时请求优先，离线任务使用较低优先级。
- Admission control：超长请求、显存不足或租户超额时进行排队、拒绝或降级。
- 弹性扩缩容：根据 prompt token/s 和 output token/s 分别扩缩 prefill 池和 decode 池。

如果所有 GPU 都繁忙，请求可进入全局优先级队列；等待超过 SLO 后应限流或降级。故障 worker 要及时摘除，并结合动态 batching 将短时间内进入同一 worker 的请求合并，以提高 GPU 利用率。

关键指标包括 queue time、TTFT/ITL P99、worker utilization、KV locality hit rate、cache eviction rate、recompute rate 和 network transfer time。

## Prefix Cache

当多个请求共享相同前缀时，Prefix Cache 可以复用该前缀的 KV Cache，避免重复 prefill。

常见共享前缀：

- 固定 system prompt。
- RAG 模板和工具说明。
- 多轮对话历史。
- Agent 工作流中的公共上下文。
- 多候选采样 / beam search 的公共 prefix。

常见实现形式包括：

- Hash-based prefix cache：对 token prefix 分块 hash。
- Radix tree cache：SGLang RadixAttention 类思想，按前缀树管理共享 KV。
- Block-level ref count：Paged KV block 可被多个序列引用。
- Copy-on-write：共享 prefix 后续分叉时按需复制。

Prefix Cache 可以降低 TTFT、减少 prefill FLOPs、节省显存，并提升高重复业务的吞吐。

使用时需要注意以下问题：

- 命中率依赖业务；开放聊天随机 prompt 命中较低。
- 缓存占显存，需要淘汰策略。
- 多租户场景需隔离，防止跨用户数据泄露。
- 分布式 cache 需要路由命中，否则跨机器拉 KV 可能不划算。

## RDMA 通信相比传统 TCP/IP 有哪些优势？

RDMA（Remote Direct Memory Access，远程直接内存访问）的核心优势是让网卡直接访问已注册的应用内存，减少传统 TCP/IP 通信路径中的内核协议栈处理、上下文切换和数据拷贝。

相比传统 TCP/IP，RDMA 主要有以下优势：

- **低延迟**：通过内核旁路和较短的数据路径，减少系统调用、上下文切换及协议栈处理带来的开销。
- **零拷贝**：数据可以在应用内存与网卡之间直接传输，避免在用户态和内核态缓冲区之间反复复制。
- **低 CPU 占用**：传输、可靠性等工作可以由支持 RDMA 的网卡卸载，使 CPU 更专注于模型计算和业务逻辑。
- **高吞吐**：降低 CPU 和内存拷贝瓶颈后，更容易接近高速网络的链路带宽。
- **支持单边通信**：`RDMA READ/WRITE` 可以直接读写已授权的远端内存，无需远端应用执行对应的 `recv`，适合 KV Cache 传输、参数同步和分布式存储等场景。
- **尾延迟更稳定**：数据路径和软件处理环节更少，通常能降低高负载下的延迟抖动。

RDMA 并非在所有场景都优于 TCP/IP。它需要支持 InfiniBand、RoCE 等协议的网卡和网络环境，应用还要处理内存注册、队列和连接管理。RoCE 对拥塞控制、PFC/ECN、MTU、网卡与 NUMA 亲和性等配置也较敏感。因此，RDMA 更适合分布式训练、PD 分离中的 KV Cache 传输、高性能存储等对延迟、吞吐和 CPU 效率要求较高的场景；普通 Web 服务通常继续使用更成熟、运维成本更低的 TCP/IP。

## 跨机器通信优化

跨机器通信主要包括以下类型：

- 模型并行通信：TP AllReduce、EP AllToAll、PP Send/Recv。
- KV transfer：PD 分离中 prefill -> decode 的 KV 传输。
- Prefix Cache 迁移：worker 间共享/拉取缓存。
- 控制面通信：调度、健康检查、指标上报。

优化时可以遵循以下原则：

1. 高频小通信尽量留在节点内 NVLink/NVSwitch。
2. 跨节点通信尽量大块、流水线化、可重叠。
3. 用拓扑感知 rank placement 避免跨 NUMA/GPU-NIC 非亲和。
4. 对 MoE AllToAll 做专家亲和和热点专家副本。
5. 对 PD KV transfer 使用 RDMA/GPUDirect，必要时 KV 量化。
6. 通过 cache-aware routing 避免不必要的远程 KV 拉取。

落地时需要检查以下项目：

- 跑 NCCL tests：all_reduce、all_to_all、sendrecv 带宽/延迟。
- 确认 IB/RoCE、GDR、NUMA 绑定、MTU、PFC/ECN 配置。
- 监控跨节点 bytes/s、传输排队、重传、P99 网络延迟。
- 将模型并行通信和 KV transfer 的网络流量隔离或限流。

## 常用 Profiling 与验收指标

### 端到端指标

- TTFT P50/P90/P99
- ITL/TPOT P50/P90/P99
- Output tokens/s、Total tokens/s
- QPS、并发请求数
- 请求排队时间、取消率、超时率
- 5xx、OOM 率、重试率

### GPU 指标

- SM utilization
- HBM bandwidth
- Tensor Core utilization
- Kernel launch 数量
- GEMM shape 与 occupancy
- Attention/MLP/Norm/Sampling kernel 耗时

### KV Cache 指标

- KV cache memory usage
- KV block utilization
- prefix cache hit rate
- cache eviction / recompute / swap 次数
- 每请求 token 数分布

### 通信指标

- AllReduce / AllToAll / SendRecv 耗时
- 跨节点传输 bytes/s
- KV transfer latency
- 网络 P99/P999
- NCCL timeout / retry / slow rank

### 主机、业务与模型质量指标

- CPU、内存、网络 IO、tokenize/detokenize latency
- Batch size 利用率、输入长度和输出 token 数分布
- 空结果率、缓存命中率、置信度或预测分布
- 数据漂移、异常输入比例和核心业务指标

验收时不能只看平均延迟或 GPU 利用率。P95/P99 能暴露排队、冷启动和长请求导致的尾延迟；GPU 空闲但队列很长，通常说明瓶颈在 CPU 前处理、调度或数据传输。工程指标还要和模型质量、业务结果共同评估，避免性能提高但线上效果退化。
