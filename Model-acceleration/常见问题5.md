## 对 AI Infra 这个方向的理解

我理解 AI Infra 是连接算法和底层硬件/系统的工程层，目标是让模型训练、推理、部署更高效、更稳定、更易用。它不只是搭环境或写脚本，而是覆盖数据管道、训练框架、分布式调度、算子优化、模型压缩、推理服务、监控运维等完整链路。

从训练侧看，AI Infra 需要解决 GPU 利用率、显存不足、通信瓶颈、容错恢复、任务调度等问题；从推理侧看，要解决低延迟、高吞吐、弹性扩缩容、模型热更新和成本优化。工程上比较核心的是性能意识和系统化定位能力，比如能判断瓶颈在数据、计算、通信还是调度层。好的 AI Infra 能让算法同学更快迭代，让业务以更低成本稳定使用模型能力。

## 混合精度训练的原理和实现细节

混合精度训练的核心是用 FP16 或 BF16 加速大部分矩阵计算和减少显存，同时保留部分关键计算的数值稳定性。GPU Tensor Core 对 FP16/BF16 计算有明显加速，参数、激活和梯度占用也会下降。一般做法是前向和反向的主要算子用低精度，权重更新保留 FP32 master weights，避免长期累积误差。

FP16 动态范围较小，容易梯度下溢，所以常配合 loss scaling：先放大 loss 再反传，更新前把梯度缩回来，并检测 overflow。BF16 指数位和 FP32 接近，数值范围更好，通常不需要 loss scaling，但精度位更少。实现时要注意 LayerNorm、Softmax、Loss、梯度归约等敏感算子可能需要 FP32；还要关注通信时梯度 dtype、优化器状态 dtype、checkpoint 保存兼容性。常见坑是盲目全 FP16 导致 loss NaN，或者不同卡 overflow 状态不同步。




## Tokenizer 编码逻辑

Tokenizer 的作用是将文本转换为 token id，常见算法包括 BPE、SentencePiece、Unigram、WordPiece 和 Byte-level BPE。

在高并发服务中，Tokenizer 多在 CPU 侧运行，可能成为 GPU 前的隐性瓶颈。

实际应用中主要存在以下瓶颈：

- 长 prompt、高 QPS 下 CPU 编码耗时高。
- 多轮对话重复编码历史上下文。
- 模板拼接、特殊 token、stop words、流式 detokenize 阻塞主流程。
- Tokenizer 串行导致 GPU 等输入。

常见实现和框架包括 Hugging Face Tokenizers（Rust）、SentencePiece、tiktoken、vLLM 异步 tokenizer、TGI router 和 SGLang runtime。

可以从以下几个方面进行优化：

- 异步 tokenizer 线程池：请求接入、编码、推理调度流水化。
- Prompt token cache：缓存 system prompt、chat template、RAG 模板的编码结果。
- 批量 tokenize：离线批处理或高吞吐场景合并编码。
- 增量 detokenize：避免每轮从完整序列重新 decode。
- Stop words token-level 匹配：减少字符串级扫描。

不同技术的收益和代价如下：


| 技术            | 收益             | 代价                     |
| --------------- | ---------------- | ------------------------ |
| 异步 tokenizer  | 降低 GPU 等待    | 线程池与调度复杂度增加   |
| Tokenizer cache | 减少重复编码     | 需要缓存淘汰和多租户隔离 |
| 批量编码        | CPU 吞吐更高     | 可能增加排队延迟         |
| 增量 detokenize | 降低流式输出开销 | 需处理 Unicode 边界      |

落地时需要监控 `tokenize_latency`、`detokenize_latency`、CPU 使用率和 GPU idle time。RAG/Agent 场景尤其需要缓存固定 prompt 与历史前缀。

## 前后处理算子

前处理包含 padding/packing、attention mask、position ids 和 batch metadata。后处理包含 temperature、top-k/top-p、repetition penalty、softmax、sampling、stop condition 和 detokenize。

这一阶段主要存在以下瓶颈：

- Decode 每 token 都采样，logits processor 开销被输出长度放大。
- GPU logits 拷贝到 CPU 采样会导致 CPU-GPU 同步。
- 动态 batch 状态变化频繁，metadata 更新成本高。
- Padding 浪费计算。

典型实现包括 vLLM GPU sampling、TensorRT-LLM optimized sampling kernels、TGI continuous batching 和 SGLang structured decoding。

常见优化方式包括：

- GPU sampling：logits 在 GPU 上完成 softmax/top-k/top-p/采样。
- Fused sampling kernel：融合 temperature、penalty、top-k/top-p、随机采样。
- Guided decoding mask 下沉到 GPU 或高效状态机。
- Packing / Continuous Batching 减少 padding 与 batch 空洞。

落地时需要关注每个 token 的 CPU-GPU memcpy 次数、采样 kernel 数和 logits processor 耗时。结构化输出或 JSON schema 会显著增加约束解码成本。

## 常规算子加速优化

核心算子包括 GEMM、RMSNorm/LayerNorm、RoPE、Softmax、Residual add、Activation、Quantize/Dequantize、KV Cache 读写和 AllReduce/AllGather。

主要瓶颈包括：

- MLP/Linear 的 GEMM 计算量大。
- Norm、RoPE、elementwise 算子 FLOPs 低但访存频繁。
- 小 batch decode 的 kernel launch 开销明显。
- 量化推理中的 dequant 可能抵消收益。

典型实现包括 cuBLAS/cuBLASLt、CUTLASS、Triton、TensorRT-LLM、FasterTransformer、vLLM custom ops 和 FlashInfer。

优化方向主要包括：

- 算子融合：RMSNorm+Residual、QKV+RoPE+KV 写入、Dequant+GEMM、Linear+Bias+Activation。
- 高性能 GEMM：Tensor Core、FP16/BF16/FP8、INT8/INT4、合适 tile 和 layout。
- Norm 优化：warp-level reduction、向量化加载、fused residual。
- RoPE 优化：sin/cos cache、RoPE 与 Q/K projection 或 cache 写入融合。
- 量化 kernel：避免先完整 dequant 再 GEMM，优先 fused dequant matmul。

Prefill 更接近大矩阵计算，Tensor Core 利用率高。Decode 每步处理的 token 较少，更依赖 batching、算子融合、KV cache layout 和低延迟通信。



## 标准 Attention 瓶颈

标准自注意力：

```text
Attention(Q,K,V)=softmax(QK^T/sqrt(d)+mask)V
```

对序列长度 `N`，计算复杂度约 `O(N^2*d)`，显存中间量 `O(N^2)`。长上下文 prefill 时，attention score 矩阵和 softmax 中间结果带来巨大 HBM 读写。

## FlashAttention

FlashAttention 按 block 加载 Q/K/V，在 SRAM/shared memory 中分块计算，并使用 online softmax 累积结果，不把完整 attention matrix 写入 HBM。

```text
传统：QK^T -> HBM -> softmax -> HBM -> PV -> HBM
FlashAttention：分块 Q/K/V -> 片上 online softmax -> 只写最终输出
```

这种方式减少了 HBM IO 和中间显存占用，不改变 Attention 的数学结果，也不降低理论上的 `O(N^2)` FLOPs。

典型实现包括 FlashAttention 1/2/3、PyTorch SDPA、xFormers、TensorRT-LLM attention kernel、FlashInfer 和 vLLM backend。

使用时需要注意以下权衡：

- 长序列 prefill 收益最大，短序列收益有限。
- 对 head_dim、dtype、mask、GPU 架构有约束。
- FlashAttention-3 更偏 Hopper/H100 异步能力与 FP8 优化。

## PagedAttention

PagedAttention 将 KV Cache 切成固定大小的 block，并用 block table 将逻辑 token 映射到物理 KV block，类似虚拟内存分页。

它主要解决变长请求的显存碎片、KV 预分配浪费、Continuous Batching 下的 cache 动态增长，以及 prefix sharing 和 beam search 复用问题。

典型实现包括 vLLM PagedAttention、TensorRT-LLM paged KV cache、FlashInfer paged attention 和 SGLang。

需要注意的是，block table 间接寻址会增加 kernel 复杂度。block size 太小会产生较多元数据，太大则会增加内部碎片。

## MQA / GQA

MHA 中每个 Q head 都有独立的 K/V，MQA 中所有 Q heads 共享一组 K/V，GQA 中多个 Q heads 共享一组 K/V。

Decode 阶段的 KV 读取量与 `num_kv_heads` 成正比，因此减少 KV heads 可以显著降低显存占用和 HBM 带宽需求。

这类结构必须在模型训练时得到支持，直接从 MHA 修改结构通常无法做到无损转换。



## Attention 与 RMSNorm 融合

将 RMSNorm、Residual、QKV projection、RoPE、KV cache 写入和 Attention 前处理等相邻操作融合，可以减少中间张量写入显存。

这种方式可以降低 HBM 读写、kernel launch 和 decode 小 batch latency，同时提高 L2/cache locality。

典型实现包括 TensorRT-LLM fused kernels、Transformer Engine、Triton RMSNorm/RoPE kernels、FlashAttention 和 vLLM custom ops。

融合越深，寄存器压力和代码复杂度越高，并且通常需要针对 hidden size、head_dim、dtype 和 GPU 架构进行特化。

## 计算与显存读取融合

计算与显存读取融合的核心是一次读取数据后完成多步计算，避免 `read -> op -> write -> read -> op` 的 HBM 往返。

典型融合方式包括 RMSNorm+Residual、Dequant+GEMM、QKV+RoPE+KV 写入、SwiGLU activation 和 Attention score+mask+softmax。

落地时应优先融合 memory-bound 算子，并使用 Nsight Systems/Compute 观察 kernel 数、HBM 带宽和 occupancy。



## KV Cache 优化

KV Cache 大小近似：

```text
num_layers × batch_tokens × num_kv_heads × head_dim × 2(K,V) × dtype_size
```

KV Cache 的优化目标主要包括：

- 减容量：GQA/MQA、MLA、KV 量化、低秩压缩。
- 减读取：PagedAttention、FlashDecoding、Sliding Window。
- 强管理：分页、prefix cache、continuous batching、copy-on-write。

KV 量化或压缩可能影响模型质量，prefix cache 的收益依赖业务前缀重复率，Paged KV 则会增加间接寻址开销。

## Packing 优化

Packing 将不同长度的序列紧凑拼接，并使用 `cu_seqlens`、offset、position ids 和 mask 防止序列之间互相 attend，从而减少 padding。

典型实现包括 FlashAttention varlen API、vLLM continuous batching、TensorRT-LLM in-flight batching 和 SGLang scheduler。

落地时，离线任务可以使用 length bucketing 和 packing，在线任务可以使用 continuous batching，同时需要监控 padding ratio、有效 token 数和队列等待时间。

## Paged Attention 分片

单卡内分片是将 KV 按 block 存储，以降低碎片和预留浪费。

多卡或多节点分片可以按 layer、head、sequence block 或并行策略切分 KV。

关键参数包括 block size、num_gpu_blocks、KV dtype、prefix cache、swap space 和 max_num_batched_tokens。



## 推理整体架构

典型 serving pipeline：

```text
Client/API -> Request Queue -> Tokenizer -> Scheduler
  -> Batch Builder -> Model Executor -> Sampler -> Detokenizer/Streaming -> Client
```

核心模块：

- API server / router：接入、鉴权、限流、OpenAI-compatible 协议。
- Tokenizer / preprocessor：文本编码、chat template、prompt cache。
- Scheduler：prefill/decode 队列、token budget、优先级、准入控制。
- Model executor：GPU kernels、并行策略、CUDA Graph、采样。
- KV Cache manager：block 分配、释放、prefix cache、swap/recompute。
- Streaming output：增量 detokenize、断连取消、stop condition。

## GPU 显存调度

显存组成：

```text
Model Weights + KV Cache + Activation/Temporary Buffers + CUDA Workspace
+ Communication Buffers + Runtime Metadata + Fragmentation/Reserved
```

KV Cache 是在线推理中变化最频繁、最容易引发 OOM 的部分。显存调度需要进行 request admission control，判断当前 KV block pool 是否能够容纳新请求。

常见调度策略包括：

- 预分配显存池，避免频繁 cuda malloc。
- Paged KV block allocator，降低外部碎片。
- 设置 `gpu_memory_utilization` 安全水位。
- 长请求限流、最大上下文限制、超时取消释放 KV。
- CPU swap 或 recompute 作为极端情况下的 preemption 策略。

典型框架包括 vLLM block manager、TensorRT-LLM KV Cache Manager、TGI token limit 和 SGLang Radix Cache。



## PD 分离架构（Prefill-Decode Disaggregation）

PD 分离架构将 Prefill 和 Decode 部署在不同的 GPU worker 或实例上：

```text
Client -> Router
       -> Prefill Worker: 处理长 prompt，构建 KV Cache
       -> KV Transfer / Cache Store
       -> Decode Worker: 持续生成 token
       -> Streaming Response
```

采用分离架构主要有以下原因：

- Prefill 是计算密集型，长 prompt 会占用大量 GEMM/Attention 资源。
- Decode 是低延迟、访存密集型，每 token 小步执行，对 ITL 敏感。
- 两者混跑容易互相干扰：长 prefill 抬高 decode P99，decode 小步又会切碎 prefill 吞吐。

这种架构主要解决以下瓶颈：

- 降低长 prompt 对流式 decode 的干扰。
- 可对 prefill 与 decode 使用不同硬件/并行度配置。
- 提高集群资源利用：prefill 池按 prompt token 吞吐扩缩，decode 池按并发输出扩缩。
- 支持长上下文/RAG 中 KV 预热与复用。

典型实现和技术方向包括 DistServe、Mooncake、SGLang PD disaggregation、vLLM 相关分离式 serving 方案，以及 LMCache 等 KV cache 传输或复用系统。

主要收益和代价如下：


| 维度     | 收益                      | 代价             |
| -------- | ------------------------- | ---------------- |
| 延迟隔离 | Decode P99 更稳定         | 架构复杂         |
| 资源配置 | Prefill/Decode 可独立扩缩 | 需要准确容量规划 |
| KV 复用  | 长 prompt 可预热          | KV 传输成本高    |
| 多机支持 | 支持大规模服务            | 网络成为新瓶颈   |

## 缓存传输流水线

PD 分离后，Prefill worker 生成的 KV Cache 必须传给 Decode worker。KV 传输可能很大，成为瓶颈。

缓存传输可以按照以下方式进行流水线化：

```text
Prefill chunk 1 完成 -> 传输 chunk 1 KV -> Decode 准备
Prefill chunk 2 计算中 -> 传输 chunk 2 KV
Prefill chunk 3 计算中 -> Decode 可开始部分调度
```

主要优化方向包括：

- Chunked KV transfer：按 layer/block/chunk 分批传。
- 计算通信重叠：prefill 下一个 chunk 时传上一个 chunk KV。
- RDMA / GPUDirect：减少 CPU 拷贝。
- KV 压缩/量化传输：FP8/INT8 KV，减少带宽。
- 拓扑感知：prefill 和 decode worker 尽量 colocate 或同机架。
- Cache locality：同一会话后续请求路由到持有 prefix KV 的 decode worker。

需要注意的是，如果 KV 传输速度低于 decode 消费速度，decode worker 就会等待。如果传输占满网络，还会影响跨节点 TP/EP 通信。



## 集群调度优化

集群层调度不同于单机 scheduler，需要考虑 worker 池、模型副本、GPU 拓扑、KV locality、租户优先级、冷热缓存。

调度时需要考虑以下输入：

- 请求 prompt/output 长度估计。
- 当前 prefill/decode worker 负载。
- KV Cache/Prefix Cache 命中位置。
- GPU 显存水位、KV block 剩余量。
- 网络拓扑、跨机带宽、节点健康。
- 租户 quota、优先级、SLA。

常见调度策略包括：

- 负载均衡：按 queue length、running tokens、KV blocks、GPU utilization 分配。
- Cache-aware routing：有 prefix KV 的 worker 优先，减少重复 prefill。
- Length-aware routing：长 prompt 送到 prefill 能力强的 worker，短请求走低延迟路径。
- SLA-aware scheduling：实时请求优先，离线任务使用较低优先级。
- Admission control：超长请求、显存不足或租户超额时进行排队、拒绝或降级。
- 弹性扩缩容：根据 prompt token/s 和 output token/s 分别扩缩 prefill 池和 decode 池。

关键指标包括 queue time、TTFT/ITL P99、worker utilization、KV locality hit rate、cache eviction rate、recompute rate 和 network transfer time。

## Prefix Cache

当多个请求共享相同前缀时，Prefix Cache 可以复用该前缀的 KV Cache，避免重复 prefill。

常见共享前缀：

- 固定 system prompt。
- RAG 模板和工具说明。
- 多轮对话历史。
- Agent 工作流中的公共上下文。
- 多候选采样 / beam search 的公共 prefix。

常见实现形式包括：

- Hash-based prefix cache：对 token prefix 分块 hash。
- Radix tree cache：SGLang RadixAttention 类思想，按前缀树管理共享 KV。
- Block-level ref count：Paged KV block 可被多个序列引用。
- Copy-on-write：共享 prefix 后续分叉时按需复制。

Prefix Cache 可以降低 TTFT、减少 prefill FLOPs、节省显存，并提升高重复业务的吞吐。

使用时需要注意以下问题：

- 命中率依赖业务；开放聊天随机 prompt 命中较低。
- 缓存占显存，需要淘汰策略。
- 多租户场景需隔离，防止跨用户数据泄露。
- 分布式 cache 需要路由命中，否则跨机器拉 KV 可能不划算。

## 跨机器通信优化

跨机器通信主要包括以下类型：

- 模型并行通信：TP AllReduce、EP AllToAll、PP Send/Recv。
- KV transfer：PD 分离中 prefill -> decode 的 KV 传输。
- Prefix Cache 迁移：worker 间共享/拉取缓存。
- 控制面通信：调度、健康检查、指标上报。

优化时可以遵循以下原则：

1. 高频小通信尽量留在节点内 NVLink/NVSwitch。
2. 跨节点通信尽量大块、流水线化、可重叠。
3. 用拓扑感知 rank placement 避免跨 NUMA/GPU-NIC 非亲和。
4. 对 MoE AllToAll 做专家亲和和热点专家副本。
5. 对 PD KV transfer 使用 RDMA/GPUDirect，必要时 KV 量化。
6. 通过 cache-aware routing 避免不必要的远程 KV 拉取。

落地时需要检查以下项目：

- 跑 NCCL tests：all_reduce、all_to_all、sendrecv 带宽/延迟。
- 确认 IB/RoCE、GDR、NUMA 绑定、MTU、PFC/ECN 配置。
- 监控跨节点 bytes/s、传输排队、重传、P99 网络延迟。
- 将模型并行通信和 KV transfer 的网络流量隔离或限流。

## 常用 Profiling 与验收指标

### 端到端指标

- TTFT P50/P90/P99
- ITL/TPOT P50/P90/P99
- Output tokens/s、Total tokens/s
- QPS、并发请求数
- 请求排队时间、取消率、超时率
- OOM 率、重试率

### GPU 指标

- SM utilization
- HBM bandwidth
- Tensor Core utilization
- Kernel launch 数量
- GEMM shape 与 occupancy
- Attention/MLP/Norm/Sampling kernel 耗时

### KV Cache 指标

- KV cache memory usage
- KV block utilization
- prefix cache hit rate
- cache eviction / recompute / swap 次数
- 每请求 token 数分布

### 通信指标

- AllReduce / AllToAll / SendRecv 耗时
- 跨节点传输 bytes/s
- KV transfer latency
- 网络 P99/P999
- NCCL timeout / retry / slow rank
