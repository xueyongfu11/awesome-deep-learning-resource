# TencentDB Agent Memory 对话流程说明

本文从流程视角说明 TencentDB Agent Memory 插件如何接入 Agent 系统，以及一轮用户对话中记忆是如何被读取、注入、清理、保存和异步整理的。

本文不做源码逐行走读，但会保留理解系统必需的关键机制：记忆注入到哪个 prompt、为什么这样注入、各层 memory 如何被构建和召回、tools 与 hooks 在链路中的边界。

## 1. 一句话理解

TencentDB Agent Memory 的核心流程是：

> 请求前读取并注入相关记忆，请求中允许 Agent 主动查记忆，请求后捕获真实对话，再在后台把原始对话整理成长期记忆。

也就是说，用户等待 Agent 回复时，主要发生的是“召回已有记忆”；而“把本轮对话沉淀成新记忆”通常发生在回复完成之后，由后台任务继续处理。

## 2. 插件接入 Agent 系统后做了什么

插件启动后主要完成三类接入：

1. 准备记忆系统

   插件会初始化本地数据目录、存储、向量检索能力和记忆处理组件。后续所有原始对话、结构化记忆、场景信息和用户画像都会围绕这套能力读写。
2. 注册 Agent 可调用工具

   插件向 Agent 暴露两个查询工具：

   - `tdai_memory_search`：查询结构化长期记忆。
   - `tdai_conversation_search`：查询原始历史对话。

   这两个工具不是每轮都必然调用，而是 Agent 在觉得自动注入的记忆不够时，主动发起检索。
3. 注册生命周期 hooks

   插件监听 Agent 运行过程中的关键事件：

   - `before_prompt_build`：构建 prompt 前，自动召回并注入记忆。
   - `before_message_write`：写入历史前，清理临时注入的记忆内容。
   - `agent_end`：Agent 回复完成后，捕获本轮真实对话。
   - `gateway_stop`：系统停止时，释放资源并收尾。

## 3. Agent 可调用工具和 hooks 的差异

Agent 可调用工具和 hooks 都是插件接入 Agent 系统的扩展点，但控制权不同。


| 类型       | 谁触发                 | 发生时机                         | 主要用途                                 | 本插件示例                                                                 |
| ------------ | ------------------------ | ---------------------------------- | ------------------------------------------ | ---------------------------------------------------------------------------- |
| Agent 工具 | Agent/模型按需触发     | Agent 推理过程中                 | 主动查询记忆或原始对话，补充回答所需证据 | `tdai_memory_search`、`tdai_conversation_search`                           |
| Hooks      | Agent 宿主框架自动触发 | 请求前、写历史前、请求后、关闭时 | 自动召回、清理、捕获、后台构建、释放资源 | `before_prompt_build`、`before_message_write`、`agent_end`、`gateway_stop` |

可以简单理解为：

- 工具是“给 Agent 主动使用的查询接口”。
- hooks 是“插件跟随 Agent 生命周期自动运行的流程节点”。

自动召回和自动捕获主要靠 hooks 完成；当自动注入的记忆不够时，Agent 才通过工具继续主动检索。

## 4. 一轮用户对话的完整流程

下面按时间顺序说明一轮对话中发生了什么。

### 4.1 用户发起请求

用户输入一条消息后，Agent 系统准备构建本轮 prompt。

此时插件还不会直接生成回答，而是先介入 prompt 构建前的阶段，判断是否需要给本轮对话补充历史记忆。

### 4.2 请求前：自动召回并注入记忆

在 prompt 构建前，`before_prompt_build` 会被触发。

这一阶段的目标是：在 Agent 真正开始推理前，把可能有帮助的长期记忆放进上下文。

插件通常会做几件事：

1. 识别当前用户输入和 session。
2. 根据用户输入检索相关记忆。
3. 选择适合本轮使用的内容。
4. 按内容稳定性，把记忆注入到不同位置。

这里需要区分两类注入位置。


| 注入位置           | 注入内容                                             | 内容特点                           | 为什么放这里                                                                             |
| -------------------- | ------------------------------------------------------ | ------------------------------------ | ------------------------------------------------------------------------------------------ |
| user prompt 前缀   | L1 相关记忆，即本轮根据用户问题召回出的结构化记忆    | 每轮都可能不同，强依赖当前用户问题 | 它是“本轮问题的相关参考”，放在用户输入旁边更贴近当前任务，也避免频繁改动 system prompt |
| system prompt 末尾 | L3 用户画像、L2 scene navigation、memory tools guide | 相对稳定，变化不频繁               | 它们是长期背景和工具使用规则，适合作为稳定系统上下文，也更利于 prompt cache              |

也就是说，自动召回不是简单地把所有记忆塞到同一个地方，而是分成动态上下文和稳定上下文。

#### 放入 user prompt 的内容

放到 user prompt 前面的主要是 L1 相关记忆。

这些内容会被包在类似 `<relevant-memories>...</relevant-memories>` 的区块里，含义是：

- 这些是根据当前用户输入临时召回出来的参考信息。
- 它们不代表当前任务的最新进度。
- Agent 可以参考，但不能把它们当作用户本轮新输入。

例如用户问“继续昨天那个数据库记忆插件的文档”，系统可能把之前抽取过的相关 L1 记忆放到本轮 user prompt 前面，让 Agent 知道“数据库记忆插件”“文档”“流程解释”这些历史背景。

这部分每轮都可能变化，所以它属于动态注入。

#### 放入 system prompt 的内容

放到 system prompt 末尾的主要是稳定上下文：

- L3 persona：用户长期画像，例如偏好的沟通方式、技术栈、输出风格。
- L2 scene navigation：场景导航，告诉 Agent 当前有哪些长期场景、项目或主题可供定位。
- memory tools guide：告诉 Agent 什么时候应该主动调用记忆工具，以及调用限制。

这些内容不是某一轮问题独有的，而是 Agent 在多轮对话中都可以复用的背景。因此它们更适合放在 system prompt 末尾。

这种拆分还有一个工程上的原因：system prompt 如果频繁变化，会影响模型服务商的 prompt cache。L1 相关记忆每轮变化大，所以放在 user prompt；L2/L3 和工具指南变化较少，所以放在 system prompt。

#### 这一阶段的结果

`before_prompt_build` 结束后，Agent 看到的上下文大致是：

```text
system prompt
  + L3 用户画像
  + L2 场景导航
  + 记忆工具使用指南

user prompt
  + <relevant-memories>
      本轮召回的 L1 相关记忆
    </relevant-memories>
  + 用户真实输入
```

这一步的重点是“帮 Agent 想起来”，但同时控制记忆进入 prompt 的位置，避免长期稳定背景和本轮动态参考混在一起。

### 4.3 Agent 正常推理和回答

Agent 拿到增强后的 prompt 后开始正常推理。

如果自动注入的内容已经足够，Agent 可以直接回答用户。

如果 Agent 发现还缺少信息，可以在推理过程中主动调用记忆工具：

- 想查结构化长期事实时，调用 `tdai_memory_search`。
- 想查历史原话或完整上下文时，调用 `tdai_conversation_search`。

所以记忆进入 Agent 上下文有两条路径：

1. 请求前自动注入。
2. 推理中由 Agent 主动检索。

### 4.4 写历史前：清理临时注入内容

Agent 系统准备把消息写入历史记录前，`before_message_write` 会被触发。

这一步非常重要，它的职责是清理“只给本轮推理看的临时记忆上下文”。

请求前自动注入的记忆是为了帮助 Agent 回答，不是用户真实输入。如果不清理，这些注入内容就会被写进历史对话，带来几个问题：

- 历史记录不再等于用户真实说过的话。
- 后续检索原始对话时，会检索到插件自己注入的内容。
- 后台抽取新记忆时，可能把旧记忆再次抽取成新记忆。
- 记忆会出现重复、回音和污染。

因此，这一步会在写入历史前移除类似 `<relevant-memories>...</relevant-memories>` 的临时内容，让历史里只保留真实用户消息。

注意，这里清理的是被塞进 user prompt 的动态 L1 召回内容。放在 system prompt 的 L2/L3 稳定上下文和工具指南，本来就不是用户消息的一部分，不应该作为用户历史写入。

### 4.5 请求后：捕获本轮真实对话

Agent 成功完成回复后，`agent_end` 会被触发。

这时插件会捕获本轮对话，包括用户真实输入和 Agent 的回复，并把它作为原始对话记录保存下来。

这里保存的是“事实发生过的对话”，也就是后面 L0 层的基础数据。

捕获时会特别注意使用清理后的用户输入，而不是带有 `<relevant-memories>` 的增强 prompt。否则 L0 会被污染，后续所有从 L0 派生出来的 L1/L2/L3 都会受到影响。

这一步通常不会阻塞用户继续使用系统。它更像是把本轮对话交给后台记忆管线，告诉系统：“这里有一轮新对话，可以后续整理成长期记忆。”

### 4.6 后台异步整理长期记忆

本轮对话保存后，后台任务会逐步判断是否需要从原始对话中提炼更高层的记忆。

后台整理不是每轮都完整跑完 L1/L2/L3。它会根据配置、checkpoint、累计的新对话数量、场景变化等条件决定是否触发某一层任务。

这个过程通常包括：

1. 从原始对话中抽取结构化事实。
2. 合并、去重或更新已有记忆。
3. 识别对话所属场景。
4. 更新用户画像或长期偏好。

更具体地看：


| 后台阶段     | 输入                          | 输出                           | 作用                                   |
| -------------- | ------------------------------- | -------------------------------- | ---------------------------------------- |
| L0 保存      | 本轮真实用户消息和 Agent 回复 | 原始对话记录、可检索索引       | 保留事实来源，支持后续证据追溯         |
| L1 抽取      | 新增 L0 对话                  | 结构化 memory 条目             | 把对话中可复用的信息提炼成长期记忆     |
| L1 去重/合并 | 新 memory + 旧 memory         | 更新后的结构化记忆             | 避免重复、冲突、过期信息长期堆积       |
| L2 场景整理  | 一批 L1 记忆                  | scene blocks、scene navigation | 把零散记忆组织到项目、主题、长期任务中 |
| L3 画像更新  | L2 场景信息和已有 persona     | 更新后的 persona               | 总结跨场景稳定偏好和用户画像           |

这部分不是 Agent 当场回答所必须的链路，所以一般放在后台处理。这样既能减少用户等待时间，也能让记忆构建更稳定。

## 5. L0-L3 四层记忆分别负责什么

TencentDB Agent Memory 可以理解为四层记忆体系。

### 5.1 L0：原始对话层

L0 保存用户和 Agent 真实发生过的对话。

它的特点是：

- 最接近事实来源。
- 保留原始上下文。
- 适合做证据追溯。

L0 的来源是 `agent_end` 捕获到的真实对话。这里强调“真实”，是因为请求前注入的 `<relevant-memories>` 只是临时上下文，不能进入 L0。

L0 通常还会建立检索索引，方便后续按关键词或语义搜索原始对话。

L0 默认不会大量自动注入到每轮 prompt，因为原始对话可能很长、很杂，直接塞进去会浪费上下文，也容易引入无关历史。Agent 需要查历史原话、完整上下文或证据时，可以通过 `tdai_conversation_search` 主动查询。

### 5.2 L1：结构化记忆层

L1 是从原始对话中提炼出来的结构化长期记忆。

它关注的是可以复用的事实，例如：

- 用户偏好。
- 用户长期目标。
- 用户明确说过的约束。
- 项目、任务、环境相关事实。
- 之前已经确认过的结论。

L1 是最常用于自动召回的一层。请求前自动注入的“相关记忆”通常主要来自 L1。

L1 的构建通常分两步。

第一步是抽取。后台会读取新增 L0 对话，判断其中哪些信息值得长期保存，并把它们整理成结构化条目。L1 记忆通常会区分类型，例如：

- `persona`：用户身份、偏好、长期稳定特征。
- `episodic`：历史事件、阶段性活动、发生过的事情。
- `instruction`：用户明确给出的长期规则、要求或命令。

第二步是去重和合并。新抽取出的记忆可能和旧记忆重复、冲突或只是更具体的更新版本。系统会判断应该新增、合并、更新还是忽略，避免长期记忆不断膨胀。

在 recall 阶段，L1 会根据本轮用户输入被动态检索，然后作为 `<relevant-memories>` 注入到 user prompt 前面。

### 5.3 L2：场景层

L2 用来组织更大的上下文场景。

它不是单条事实，而是帮助系统理解“用户现在处在什么长期任务、项目或主题中”。

例如：

- 某个代码仓库的长期开发场景。
- 某个产品方案的持续讨论。
- 某个用户工作流的连续上下文。

L2 的价值是帮助 Agent 不只记住零散事实，还能知道这些事实属于哪个场景，以及当前对话可能应该关联到哪些历史主题。

L2 的构建对象通常不是单轮对话，而是一批已经形成的 L1 记忆。系统会把这些记忆归纳进不同 scene block，例如某个项目、某类长期任务、某个持续讨论的主题。

L2 还有一个关键产物是 scene navigation。它不是把所有场景全文都塞进 prompt，而是给 Agent 一个导航索引，让 Agent 知道：

- 当前有哪些场景。
- 每个场景大概讲什么。
- 如果需要更完整的场景上下文，应该去读哪个 scene block。

因此，L2 在请求前通常以“导航”的形式进入 system prompt，而不是把所有场景细节直接注入。

### 5.4 L3：用户画像层

L3 是更稳定、更长期的用户画像。

它关注的是跨场景也成立的信息，例如：

- 用户的沟通偏好。
- 用户的技术栈偏好。
- 用户对输出格式的偏好。
- 用户长期稳定的工作习惯。

L3 通常会作为稳定上下文注入，帮助 Agent 以更符合用户习惯的方式工作。

L3 不是简单复制 L1 中的 persona 类型记忆，而是基于更长周期、更高层的信息做综合总结。它会参考场景、历史记忆和已有 persona，提炼出跨任务也稳定成立的用户画像。

请求前，L3 persona 会作为稳定上下文进入 system prompt。它的作用不是回答某个具体问题，而是影响 Agent 的整体行为方式，例如回答详略、默认假设、技术表达风格等。

## 6. 后台整理 L1-L3 memory 的时机

后台整理不是固定 cron 一次性跑完 L1-L3，而是由对话结束事件触发，再通过队列和 timer 逐层推进。

### 6.1 总体链路

每轮对话结束后，系统先写入 L0 原始对话，然后通知后台调度器。

```text
agent_end / sync_turn
  -> L0 capture
  -> notifyConversation(sessionKey)
  -> L1 结构化记忆抽取
  -> L2 scene block 整理
  -> L3 persona 尝试更新
```

### 6.2 L1 触发时机

L1 主要按对话轮数或空闲时间触发：

- 新 session 默认开启 warm-up：第 1 轮就触发，之后阈值变为 `2 -> 4 -> 5`。
- warm-up 结束后，默认每 `5` 轮对话触发一次。
- 如果没到轮数阈值，但 session 空闲超过 `600` 秒，也会触发。
- session 结束或进程关闭时，会 flush 未处理内容。

L1 runner 实际从 L0 的 VectorStore DB 或 JSONL 文件增量读取数据，而不是依赖内存里的消息数组。

### 6.3 L2 触发时机

L2 跟在 L1 成功之后，但不会立刻执行：

- L1 成功后，默认等待 `90` 秒再触发 L2。
- 同一个 session 的两次 L2 默认至少间隔 `900` 秒，也就是 15 分钟。
- L2 完成后，会设置一个 `3600` 秒后的兜底检查。
- 如果 session 超过 `24` 小时不活跃，周期检查会停止；下一次 L1 成功后再重新唤醒。

L2 自身会做增量判断：没有新的 L1 records 时，会跳过 scene extraction。

### 6.4 L3 触发时机

每次 L2 完成后都会尝试触发 L3，但真正生成 persona 还要满足触发条件：

- 第一次生成 scene 后。
- 累计新增 memory 达到 `persona.triggerEveryN`，默认 `50` 条。
- L2 显式请求 persona 更新。
- persona 文件缺失或损坏，需要恢复。

L3 是全局串行执行的。如果 L3 正在运行，新的触发只会标记为 pending，当前任务结束后最多再补跑一次。

## 7. Prompt 在记忆系统里的作用

Prompt 主要用在两类地方。

第一类是“给 Agent 回答用的上下文注入”。

也就是请求前把召回到的记忆整理成 Agent 能理解的上下文，让它知道哪些信息和本轮问题相关。这里又分成：

- L1 相关记忆注入 prompt：告诉 Agent “这些是本轮可能相关的历史记忆”。
- L3 persona 注入 prompt：告诉 Agent “这是用户长期画像”。
- L2 scene navigation 注入 prompt：告诉 Agent “这些是可用场景索引，需要时可深入读取”。
- memory tools guide 注入 prompt：告诉 Agent “自动注入不够时，应该如何主动搜索”。

第二类是“后台构建记忆”。

后台任务会使用 prompt 指导模型完成这些工作：

- 从对话中抽取结构化记忆。
- 判断新旧记忆是否重复、冲突或需要合并。
- 从一批记忆中归纳场景，并维护 scene block。
- 更新长期用户画像，并控制 persona 不无限膨胀。

后台 prompt 的职责不是直接回答用户，而是把对话整理成更适合长期使用的记忆资产。不同层的 prompt 关注点不同：


| Prompt 类型           | 主要输入                 | 主要输出                   | 作用                             |
| ----------------------- | -------------------------- | ---------------------------- | ---------------------------------- |
| L1 extraction         | 原始对话                 | 结构化 memory              | 从自然对话中抽取可长期复用的信息 |
| L1 conflict/dedup     | 新旧 memory              | 新增、合并、更新或忽略决策 | 控制记忆质量，减少重复和冲突     |
| L2 scene extraction   | 一批 L1 memory           | scene block                | 把零散记忆组织成场景             |
| L3 persona generation | scene 信息、已有 persona | persona 文档               | 提炼跨场景稳定用户画像           |

所以 prompt 不只服务于回答，也服务于记忆本身的整理和演化。

## 8. Context Offload 的位置

Context Offload 不属于 L0-L3 长期用户记忆主链路。

它更像是另一套上下文压缩机制，用来处理长任务上下文、工具日志和中间状态，避免 Agent 上下文过长。

可以这样区分：

- L0-L3 Memory：关注“用户长期记忆”和“历史对话事实”。
- Context Offload：关注“长任务运行过程中的上下文压缩和恢复”。

两者都和上下文管理有关，但目标不同。

## 10. 总结

完整流程可以压缩成下面这条链路：

```text
用户输入
  -> 请求前自动召回 L1/L2/L3
  -> 注入相关记忆到本轮 prompt
  -> Agent 推理并可主动调用记忆工具
  -> 写历史前清理临时注入内容
  -> Agent 回复完成
  -> 保存真实对话到 L0
  -> 后台从 L0 抽取、合并、更新 L1/L2/L3
  -> 后续对话继续召回使用
```

核心要点是：

- hooks 负责自动化流程：召回、清理、捕获、收尾。
- tools 负责 Agent 主动查询：查结构化记忆或原始对话。
- L0 是事实来源，L1 是常用长期记忆，L2 是场景组织，L3 是用户画像。
- 请求前主要做 recall，请求后主要做 capture 和后台记忆构建。
- `before_message_write` 的作用是防止临时注入的记忆污染真实历史。
