# 多向量表示是什么意思？

多向量表示就是：一篇文档或一个知识单元，不只存一个 embedding，而是从多个角度存多个 embedding。

因为一篇长文档可能同时包含很多主题。如果只生成一个向量，会变成“平均语义”，检索时容易不准。

例如一篇文档叫：

```text
《网关接入与故障排查手册》
```

里面可能有这些内容：

```text
1. 网关安装
2. API Key 配置
3. OAuth 登录
4. 401 token_invalidated 报错
5. 日志查看
6. 重启服务
```

如果整篇文档只存一个向量：

```text
doc_vector = embedding(整篇文档)
```

那这个向量会很泛，可能表示成：

```text
网关、配置、认证、故障排查
```

用户问：

```text
token_invalidated 401 怎么处理？
```

这个问题很具体，但整篇文档向量太“平均”，可能排不到前面。

多向量表示会这样存：

```text
vector_1 = embedding("标题：网关接入与故障排查手册")
vector_2 = embedding("摘要：本文介绍网关安装、认证配置、401 报错处理...")
vector_3 = embedding("章节：网关安装 ...")
vector_4 = embedding("章节：API Key 配置 ...")
vector_5 = embedding("章节：OAuth 登录 ...")
vector_6 = embedding("章节：401 token_invalidated 报错 ...")
vector_7 = embedding("章节：日志查看 ...")
```

每个向量都指向同一篇文档，或者同一个文档片段。

检索时，用户 query 会和这些向量分别匹配：

```text
query = "token_invalidated 401 怎么处理？"
```

最可能命中的是：

```text
vector_6 = "章节：401 token_invalidated 报错 ..."
```

然后系统返回这个章节，而不是只返回整篇文档。

可以理解成：

```text
单向量：
一篇文档只有一个入口。

多向量：
一篇文档有多个入口，标题、摘要、章节、段落、FAQ 都能被命中。
```

常见的多向量表示方式：

1. Chunk 向量

```text
文档切成多个 chunk，每个 chunk 一个 embedding。
```

2. 标题向量

```text
用户问得很短时，标题向量可能更容易命中，例如“OAuth 登录失败”。
```

3. 摘要向量

```text
长章节内容很多时，可以先生成章节摘要，再对摘要向量化，适合做粗召回。
```

4. 问题向量 / FAQ 向量

```text
如果文档是 FAQ，可以给每个问题单独建向量，例如“token_invalidated 报错怎么办？”。
```

5. 实体向量 / 关键词字段

```text
error_code: 401
keyword: token_invalidated
module: auth
product: gateway
```

严格说实体和关键词字段不一定都是 embedding，也可以是 metadata 或倒排索引，但思路一样：给知识提供多个命中入口。

一个简单的数据结构可以是：

```json
[
  {
    "vector_id": "v1",
    "doc_id": "doc_001",
    "type": "title",
    "text": "网关接入与故障排查手册"
  },
  {
    "vector_id": "v2",
    "doc_id": "doc_001",
    "type": "summary",
    "text": "本文介绍网关认证、OAuth 登录、401 token_invalidated 报错处理"
  },
  {
    "vector_id": "v3",
    "doc_id": "doc_001",
    "type": "chunk",
    "text": "401 token_invalidated 表示认证 token 已失效..."
  }
]
```

检索后再合并：

```text
v3 命中 -> 返回 doc_001 的对应 chunk
v2 命中 -> 再去 doc_001 里找相关 chunk
v1 命中 -> 返回文档标题或章节候选
```

面试里可以这样说：

```text
多向量表示的意思是，一个文档不只用一个 embedding 表示，而是按标题、摘要、章节、chunk、FAQ 问法等多个粒度生成多个向量。这样用户无论是问概念、问具体错误码，还是问某个章节内容，都有机会命中更准确的向量，避免长文本被一个平均向量稀释语义。
```

一句话总结：

```text
多向量表示 = 给同一份知识建立多个语义入口，提升长文本和长尾问题的召回率。
```

# 分块策略：文档分块时，如何确定 Chunk Size 和 Overlap？遇到过哪些坑？

确定 `Chunk Size` 和 `Overlap`，一般不能只拍脑袋给一个固定值，而是要结合文档类型、模型能力和检索评测来调。

## 1. Chunk Size 怎么定？

我会先看内容结构和任务类型。

如果是 FAQ、接口文档、知识库：

```text
chunk 可以偏小：200～500 中文字 / 或 300～800 tokens
```

因为问题通常对应一个明确答案，太大会引入噪声。

如果是制度文档、技术方案、论文、代码说明：

```text
chunk 可以稍大：500～1000 中文字 / 或 800～1500 tokens
```

因为答案可能依赖上下文，需要保留完整段落或章节。

如果是代码：

```text
优先按函数、类、文件结构切，而不是按固定 token
```

例如一个函数、一个类、一个接口定义作为基本 chunk。代码被硬切断后，embedding 和 LLM 都容易误判。

我通常会从一个基线开始，比如：

```text
中文知识库：400～800 tokens
Overlap：10%～20%
```

然后用真实 query 集评估：

```text
Top-K Recall
MRR / nDCG
答案准确率
引用命中率
上下文 token 成本
延迟
```

如果召回结果经常答非所问，说明 chunk 可能太大或主题混杂。

如果召回片段经常缺前因后果，说明 chunk 可能太小或 overlap 不够。

## 2. Overlap 怎么定？

Overlap 的作用是防止关键语义被切在边界上。

我一般这样定：

```text
Overlap = Chunk Size 的 10%～20%
```

比如：

```text
chunk 800 tokens
overlap 80～160 tokens
```

但不是越大越好。Overlap 太大会导致：

```text
重复 chunk 太多
索引变大
召回结果高度重复
上下文浪费
reranker 排序受干扰
```

所以我会根据失败 case 调整：

- 边界处信息经常丢：增加 overlap。
- Top-K 里重复片段太多：减少 overlap，或做去重。
- 文档结构很清晰：减少 overlap，优先按标题、段落切。
- 句子依赖前文强：增加 overlap，或加入标题上下文。

## 3. 更推荐的实际策略

不是简单固定长度切，而是：

```text
先按结构切，再按 token 长度兜底
```

例如 Markdown 文档：

```text
一级标题
二级标题
段落
列表
代码块
表格
```

切的时候尽量保留完整语义单元。

如果一个章节过长，再递归切：

```text
章节 -> 段落 -> 句子 -> token
```

并给每个 chunk 添加上下文路径：

```text
文档标题 / 一级标题 / 二级标题 / 当前段落
```

这样通常比单纯 overlap 更有效。

## 4. 遇到过的坑

1. 固定长度硬切，切断语义

中文没有明显空格，直接按字符或 token 切，容易把定义、条件、结论拆开。后果是召回片段看起来相关，但缺关键约束。

2. Chunk 太大，语义被平均

一个 chunk 里有多个主题，embedding 变成混合语义，导致召回不精准。

3. Chunk 太小，答案不完整

单段被召回了，但缺前面的定义、后面的例外条件，最终 LLM 容易编补上下文。

4. Overlap 太大，重复召回

Top-K 结果全是同一段附近内容。看似召回很多，实际信息量很低。

5. 忽略标题和层级

很多段落本身没有主语。例如“该功能默认关闭”，如果不带标题，完全不知道说的是哪个功能。

6. 表格处理不好

表格按行切会丢列名，按整表切又可能太大。通常要把表头带到每一行，或按语义行组处理。

7. 代码块被切碎

函数签名和实现分离，import、类型定义、调用方被拆散。代码类文档要按 AST 或 Markdown code block 处理。

8. 只看离线 recall，不看生成质量

召回命中不代表答案好。有时 Top-K 包含正确片段，但上下文太杂，LLM 仍会答错。

9. 没做去重和相邻块合并

检索结果可能召回多个相邻 chunk。进入 LLM 前应该合并连续片段，减少重复和割裂。

10. 不同文档类型用同一套参数

FAQ、法律条款、代码、产品说明、论文适合的粒度都不同。统一参数通常效果一般。

面试可以这样回答：

```text
我一般先按文档结构做语义切分，再用 token 长度做兜底。Chunk Size 会根据文档类型和任务来定，比如 FAQ 可以小一些，技术文档和制度文档稍大，代码则优先按函数和类切。Overlap 通常从 10%～20% 开始，用真实 query 集评估 Top-K Recall、MRR、答案准确率和 token 成本，再根据失败 case 调整。实际踩过的坑包括 chunk 太大导致语义漂移，太小导致上下文不足，overlap 太大导致重复召回，表格和代码被硬切，忽略标题层级导致指代丢失。我的原则是让每个 chunk 尽量表达一个完整主题，同时控制噪声和成本。
```

# 检索优化：除了向量检索，你还用了哪些检索手段？如何做加权融合？

除了向量检索，我通常会组合关键词检索、BM25、精确匹配、元数据过滤、多路 query 改写召回和 rerank 精排。

## 1. 常用的检索手段

1. 向量检索

向量检索适合语义相似召回。比如用户问法和文档原文不完全一致，但含义接近时，向量检索更容易命中。

```text
用户问题：登录态失效怎么办？
文档内容：token 过期后需要重新刷新 access_token
```

这类场景关键词不一定完全匹配，但向量检索通常能召回。

2. 关键词检索 / BM25

BM25 适合精确词、专有名词、错误码、接口名、字段名、代码符号等场景。

```text
401
token_invalidated
order_id
CreateUser API
```

向量检索可能会把语义相近但关键 token 不一致的内容排到前面，而 BM25 对这些精确匹配更稳。

3. 精确匹配 / 短语匹配

对一些强约束查询，我会额外做 exact match 或 phrase match，例如：

```text
"token_invalidated"
"GET /api/v1/users"
"错误码 10023"
```

如果命中，可以给结果一个 boost，但一般不会只依赖精确匹配，因为用户可能换一种说法。

4. 元数据过滤

元数据过滤通常在召回前或召回中完成，用来减少无关候选。

```text
product = gateway
doc_type = faq
language = zh
permission = team_a
updated_at >= 2025-01-01
```

它不一定直接参与相似度打分，但对降低噪声非常重要，尤其是企业知识库、权限知识库、多产品文档场景。

5. Query 改写 / 多路召回

复杂问题可以生成多个检索 query 分别召回：

```text
原始 query
关键词提取 query
同义改写 query
HyDE query
英文术语 query
```

比如用户问：

```text
为什么登录一会儿就掉了？
```

可以改写出：

```text
登录态失效
token 过期
refresh token
session timeout
```

然后对这些 query 分别走向量检索和 BM25，再融合结果。

6. Rerank 精排

多路召回之后，我一般会对 top N 做 rerank。召回阶段负责尽量找全，rerank 阶段负责把真正回答问题的内容排到前面。

常见流程是：

```text
dense top 50
BM25 top 50
exact match top 20
RRF 融合 top 50
reranker 精排 top 10
```

## 2. 加权分数融合

如果各路检索分数比较可靠，可以先做归一化，再加权相加。

```text
score(d) =
  w_dense * norm_dense_score(d)
+ w_bm25  * norm_bm25_score(d)
+ w_exact * exact_match_score(d)
+ w_meta  * metadata_boost(d)
```

例如：

```text
score(d) =
  0.55 * vector_score
+ 0.35 * bm25_score
+ 0.10 * exact_match_boost
```

这里的关键是不能直接把 BM25 分数和向量相似度相加，因为它们的分数尺度不同。

常见归一化方式包括：

```text
min-max normalization
z-score normalization
sigmoid normalization
按当前 query 的排名转成 rank score
```

例如 min-max：

```text
norm_score = (score - min_score) / (max_score - min_score)
```

如果某个文档没有出现在某一路结果里，这一路分数可以记为 0。

## 3. Weighted RRF 融合

实际项目里我更常用 RRF，因为它只依赖排名，不依赖原始分数尺度。

普通 RRF：

```text
RRF(d) = Σ 1 / (k + rank_i(d))
```

加权 RRF：

```text
score(d) = Σ weight_i / (k + rank_i(d))
```

例如 dense 和 BM25 融合：

```text
score(d) =
  0.6 / (60 + dense_rank)
+ 0.4 / (60 + bm25_rank)
```

如果文档没有出现在某一路召回结果中，该路分数就是 0。

简单 Python 实现：

```python
def weighted_rrf(result_lists, weights, k=60):
    scores = {}

    for name, results in result_lists.items():
        weight = weights[name]

        for rank, doc_id in enumerate(results, start=1):
            scores[doc_id] = scores.get(doc_id, 0) + weight / (k + rank)

    return sorted(scores.items(), key=lambda x: x[1], reverse=True)


result_lists = {
    "dense": ["A", "B", "C"],
    "bm25": ["B", "D", "A"],
    "exact": ["A", "E"],
}

weights = {
    "dense": 0.55,
    "bm25": 0.35,
    "exact": 0.10,
}

final_results = weighted_rrf(result_lists, weights)
```

这种方式的好处是：

```text
不要求不同检索器分数可比
对异常高分不敏感
实现简单
适合 dense + sparse hybrid search
```

## 4. 权重怎么定？

权重一般不是拍脑袋固定，而是根据业务 query 集合调参。

我通常会先给一个初始值：

```text
dense: 0.5 ~ 0.7
BM25: 0.3 ~ 0.5
exact match: 0.05 ~ 0.15
metadata boost: 视业务规则决定
```

如果问题偏自然语言、语义问答，向量权重可以高一些。

```text
dense = 0.65
BM25 = 0.30
exact = 0.05
```

如果问题里大量包含错误码、接口名、字段名、编号，BM25 和 exact match 权重应该提高。

```text
dense = 0.45
BM25 = 0.45
exact = 0.10
```

如果是强权限、强产品线隔离的知识库，metadata 不建议只做 boost，而应该优先做 hard filter。

## 5. 生产里的完整流程

我一般会这样设计：

```text
1. query 预处理
   - 识别语言、产品线、实体、错误码、接口名

2. metadata filter
   - 先过滤权限、产品、文档类型、时间范围

3. 多路召回
   - dense vector top K
   - BM25 top K
   - exact match top K
   - query rewrite top K

4. weighted RRF 融合
   - 合并多路结果
   - 去重
   - 保留 top N

5. reranker 精排
   - cross-encoder 或 LLM reranker
   - 输出 top K

6. context packing
   - 合并相邻 chunk
   - 去重
   - 控制 token 成本
   - 保留引用来源
```

面试里可以这样回答：

```text
除了向量检索，我还会结合 BM25、关键词精确匹配、短语匹配、元数据过滤和 query 改写多路召回。向量检索解决语义相似问题，BM25 解决专有名词、错误码、字段名这类精确匹配问题，metadata 用来做权限和业务范围过滤。融合方式上，如果各路分数可靠，可以先做归一化再加权相加；但实际更常用 weighted RRF，因为它只依赖排名，不要求 BM25 分数和向量相似度在同一尺度。一般流程是 dense top K、BM25 top K、exact top K 多路召回，再用 weighted RRF 融合去重，最后用 reranker 精排 top N。权重会基于真实 query 集评估 Top-K Recall、MRR、nDCG 和最终答案准确率来调。
```

一句话总结：

```text
检索优化不是只靠向量检索，而是 dense 负责语义召回，BM25 和 exact match 负责精确召回，metadata 控制业务范围，最后用 weighted RRF 或归一化加权融合，再通过 reranker 精排。
```

# 什么是 HyDE 改写？在 RAG 检索中为什么要用它？

HyDE，全称是 Hypothetical Document Embeddings，意思是“假想文档嵌入”。

它的做法不是直接拿用户原始问题去检索，而是先让大模型根据问题生成一段“假想答案”或“假想文档”，再用这段更完整、更接近知识库表达方式的文本去做向量检索。

例如用户问：

```text
自由职业要交哪些税？
```

普通向量检索会直接用这个短问题去匹配知识库。

Enhanced 的做法会强制执行 HyDE 改写，把问题改写成类似：

```text
自由职业者通常需要根据收入类型缴纳个人所得税，部分情况下还可能涉及增值税、附加税等，具体税种取决于业务形式、开票方式和当地政策。
```

然后用这段“假想答案”去匹配知识库。

这样做的好处是：

```text
1. 原始问题通常很短，语义信息不足
2. 假想答案包含更多关键词和上下文
3. 假想答案更接近知识库文档的表达方式
4. 能提升向量检索的召回率
```

它特别适合这类场景：

```text
用户问题很短
用户表达口语化
知识库文档偏正式
问题和文档之间存在表达差异
```

不过 HyDE 也有风险。因为假想答案是模型生成的，可能引入错误方向。如果改写内容偏了，检索也会被带偏。

所以生产中通常不会只依赖 HyDE，而是结合多路召回：

```text
原始 query 检索
HyDE query 检索
BM25 检索
RRF 融合
reranker 精排
```

面试里可以这样回答：

```text
HyDE 是一种查询改写方法，它会先让 LLM 根据用户问题生成一段假想答案，再用这段假想答案去做向量检索。这样可以把短问题扩展成语义更完整、表达更接近知识库文档的文本，从而提升召回效果。例如用户问“自由职业要交哪些税”，系统会先生成“自由职业者通常需要缴纳个人所得税，部分情况下还涉及增值税等”这样的假想答案，再用它去检索。但 HyDE 可能引入错误假设，所以一般会和原始 query、BM25、RRF 融合以及 reranker 一起使用。
```

一句话总结：

```text
HyDE 的核心是：先生成一段可能的答案，再用这段答案去检索，让短问题变成长语义查询，从而提高召回率。
```
