### 2、讲一下你对 AI Infra 这个方向的理解

我理解 AI Infra 是连接算法和底层硬件/系统的工程层，目标是让模型训练、推理、部署更高效、更稳定、更易用。它不只是搭环境或写脚本，而是覆盖数据管道、训练框架、分布式调度、算子优化、模型压缩、推理服务、监控运维等完整链路。

从训练侧看，AI Infra 需要解决 GPU 利用率、显存不足、通信瓶颈、容错恢复、任务调度等问题；从推理侧看，要解决低延迟、高吞吐、弹性扩缩容、模型热更新和成本优化。工程上比较核心的是性能意识和系统化定位能力，比如能判断瓶颈在数据、计算、通信还是调度层。好的 AI Infra 能让算法同学更快迭代，让业务以更低成本稳定使用模型能力。

### 4、在训练大规模模型时，遇到显存不足的问题，有哪些常见的优化手段？

显存主要由模型参数、梯度、优化器状态、激活值和临时 buffer 组成。优化时我会先用 profiler 或 memory summary 判断主要占用来源，再选择手段。常见方法包括减小 batch size、使用 gradient accumulation 保持等效 batch；开启混合精度，用 FP16/BF16 降低参数和激活占用；使用 activation checkpointing，用重新计算换显存。

如果是大模型训练，还可以用 ZeRO/FSDP，把参数、梯度、优化器状态分片到多卡；模型并行或流水线并行也能拆分单卡压力。优化器方面可以用 8-bit optimizer 或 Adafactor。数据上要避免一次性缓存过多 tensor，及时 detach 和释放无用引用。常见坑是只看 batch size，不看 sequence length；或者 Python 变量持有计算图导致显存泄漏。工程上要结合吞吐评估，因为省显存的方法往往会增加计算或通信开销。

### 5、讲一下混合精度训练的原理和实现时需要注意的细节

混合精度训练的核心是用 FP16 或 BF16 加速大部分矩阵计算和减少显存，同时保留部分关键计算的数值稳定性。GPU Tensor Core 对 FP16/BF16 计算有明显加速，参数、激活和梯度占用也会下降。一般做法是前向和反向的主要算子用低精度，权重更新保留 FP32 master weights，避免长期累积误差。

FP16 动态范围较小，容易梯度下溢，所以常配合 loss scaling：先放大 loss 再反传，更新前把梯度缩回来，并检测 overflow。BF16 指数位和 FP32 接近，数值范围更好，通常不需要 loss scaling，但精度位更少。实现时要注意 LayerNorm、Softmax、Loss、梯度归约等敏感算子可能需要 FP32；还要关注通信时梯度 dtype、优化器状态 dtype、checkpoint 保存兼容性。常见坑是盲目全 FP16 导致 loss NaN，或者不同卡 overflow 状态不同步。

### 6、场景题：假设要部署一个百亿参数的大模型推理服务，你会从哪些方面来设计这个服务，以保证高并发和低延迟？

我会从模型、引擎、服务和调度四层设计。模型层先做权重量化，比如 FP16/BF16、INT8、INT4，降低显存和带宽压力；必要时做张量并行或流水线并行，把百亿参数切到多卡。推理引擎可以选 TensorRT-LLM、vLLM、FasterTransformer 等，重点利用 KV Cache、PagedAttention、算子融合、CUDA Graph 和高效 attention kernel。

服务层要支持动态 batching，把短时间窗口内请求合批，提高吞吐，但要设置最大等待时间避免拖高 P99。LLM 推理还要区分 prefill 和 decode，prefill 计算密集，decode 更受访存和 KV cache 影响，可以做连续 batching。调度层要按请求长度、租户、优先级做排队和限流，并支持熔断、超时、降级。监控上关注 TTFT、TPOT、P50/P99、GPU 利用率、KV cache 命中和显存碎片。常见坑是只追求大 batch，结果尾延迟不可控。

### 7、了解 CUDA 编程吗？简单说一下 kernel 函数、线程网格和线程块的概念

CUDA kernel 是运行在 GPU 上的函数，通常用 `__global__` 修饰，由 CPU 端发起调用。调用时会指定 grid 和 block 维度，例如 `kernel<<<gridDim, blockDim>>>()`。grid 是线程块的集合，block 是线程的集合，线程通过 `blockIdx`、`threadIdx`、`blockDim` 计算自己负责的数据下标。

GPU 执行时，一个 block 会被调度到一个 SM 上，block 内线程可以通过 shared memory 通信，也可以用 `__syncthreads()` 同步；不同 block 之间通常不能直接同步。线程实际以 warp 为单位执行，一般 32 个线程一个 warp。工程上写 kernel 要关注访存合并、shared memory bank conflict、warp divergence、occupancy 和寄存器占用。比如连续线程访问连续地址通常更高效；分支严重会导致同一个 warp 串行执行不同路径。AI Infra 中自定义算子、融合算子和性能分析都需要这些基础。

### 8、在分布式训练中，你了解哪些并行策略（如数据并行、模型并行、流水线并行）？它们分别适用于什么场景？

数据并行是最常见的方式，每张卡保存完整模型，处理不同 mini-batch，反向后通过 All-Reduce 同步梯度。它实现简单、扩展性好，适合模型能放进单卡、batch 可以增大的场景，瓶颈通常是梯度通信。

模型并行是把模型切到多张卡。张量并行会把单层矩阵乘按维度切分，适合单层参数很大、单卡放不下的 Transformer；但层内通信频繁，对 NVLink/高速互联要求高。流水线并行按层切分，不同 micro-batch 在不同 stage 上流水执行，适合层数很多的模型，但要处理 pipeline bubble 和负载均衡。还有 ZeRO/FSDP，本质是参数、梯度、优化器状态分片，适合大模型训练中节省显存。实际工程里常组合使用，比如数据并行 + ZeRO + 张量并行 + 流水线并行，并根据模型大小、网络带宽、batch size 做权衡。

### 9、手写算法：实现一个函数，计算两个非常大（以字符串形式给出）的二进制数的和，结果也以二进制字符串返回。

思路和十进制大数加法类似，从两个字符串末尾开始逐位相加。每一位的和为 `bitA + bitB + carry`，当前结果位是 `sum % 2`，进位是 `sum / 2`。因为输入可能非常大，不能转成整数类型，否则会溢出。最后把结果反转，并去掉不必要的前导零；如果结果为空则返回 `"0"`。

时间复杂度是 `O(max(n, m))`，其中 `n` 和 `m` 是两个字符串长度；空间复杂度也是 `O(max(n, m))`，用于保存结果。工程上要注意输入 `"0"`、长度不一致、最后仍有进位、以及前导零场景。面试时可以说明假设输入只包含 `'0'` 和 `'1'`，如果生产代码还需要做合法性校验。

```cpp
#include <algorithm>
#include <string>
using namespace std;

string addBinary(string a, string b) {
    int i = static_cast<int>(a.size()) - 1;
    int j = static_cast<int>(b.size()) - 1;
    int carry = 0;
    string res;

    while (i >= 0 || j >= 0 || carry) {
        int x = (i >= 0) ? a[i--] - '0' : 0;
        int y = (j >= 0) ? b[j--] - '0' : 0;
        int sum = x + y + carry;
        res.push_back(char('0' + (sum & 1)));
        carry = sum >> 1;
    }

    reverse(res.begin(), res.end());

    size_t pos = res.find_first_not_of('0');
    if (pos == string::npos) return "0";
    return res.substr(pos);
}
```

### 10、如何监控一个 AI 训练任务，通常需要关注哪些核心指标？

我会把训练监控分成模型指标、系统指标和任务健康指标。模型侧看 loss、accuracy、AUC、perplexity、梯度 norm、学习率、参数 norm、是否出现 NaN/Inf，以及训练集和验证集差距。系统侧看 GPU utilization、显存占用、Tensor Core 使用率、CPU、内存、磁盘 IO、网络带宽、数据加载耗时、step time 和 samples/sec。

分布式训练还要特别关注各 rank 的 step time 是否一致、All-Reduce 通信耗时、straggler、节点掉线和 checkpoint 成功率。任务健康方面需要监控日志错误、重试次数、数据读取失败、队列堆积、训练进度是否停滞。工程上通常会用 Prometheus/Grafana、TensorBoard、W&B 或自研平台做统一观测。常见坑是只看平均 GPU 利用率，不看波动和长尾；或者 loss 正常但数据 pipeline 已经成为瓶颈，导致资源成本很高。

### 11、在模型训练中，除了 Loss，你还通过什么来判断模型是否在向好的方向收敛？

除了 loss，我会看任务相关指标，比如分类任务的 accuracy、precision、recall、F1、AUC，生成任务的 perplexity、BLEU、ROUGE 或人工评估指标。更重要的是同时看训练集和验证集表现：如果训练 loss 降但验证指标变差，可能过拟合；如果两者都不动，可能学习率、数据或模型容量有问题。

还会看梯度和参数状态，比如 gradient norm 是否稳定，是否出现梯度爆炸或消失，参数分布是否异常，激活值是否大量饱和。对于大模型训练，还会看 learning rate schedule 下 loss 曲线是否符合预期、不同数据 shard 的 loss 是否异常、是否出现 NaN/Inf。工程实践中我也会抽样看预测结果，尤其是 bad case，因为指标平均值可能掩盖结构性问题。判断收敛不能只看单点，而要看趋势、验证表现和业务目标是否一致。

### 12、了解强化学习吗？它和传统的监督学习在训练范式上有什么根本不同？

强化学习的核心是智能体和环境交互，智能体根据状态选择动作，环境返回奖励和新状态，目标是最大化长期累计奖励。它和监督学习最大的区别是没有直接的标准答案标签，训练信号来自延迟奖励，而且当前策略会影响后续采样到的数据分布。

监督学习通常是在固定数据集上学习 `x -> y` 的映射，样本近似独立同分布，优化目标比较直接。强化学习则有探索与利用的权衡、奖励稀疏、训练不稳定、样本效率低等问题。常见方法包括 value-based 的 Q-learning、policy gradient、Actor-Critic、PPO 等。在大模型场景里，RLHF/RLAIF 会用奖励模型评价输出，再用 PPO 或类似方法优化策略。工程上强化学习对分布式采样、经验缓存、策略更新一致性和评估体系要求较高，训练稳定性通常比普通监督学习更难控制。

### 13、你认为 AI Infra 工程师和算法工程师的核心区别在哪里？

算法工程师更关注模型效果本身，比如网络结构、训练目标、特征、数据策略和指标提升；AI Infra 工程师更关注让模型以更高效率、更低成本、更稳定的方式完成训练和服务化。两者目标不同，但边界不是割裂的。

AI Infra 需要理解模型和训练过程，否则很难判断性能瓶颈和数值问题；同时也要有系统能力，包括 GPU/CUDA、分布式通信、存储 IO、调度、监控、服务治理等。比如算法同学提出更大的模型或更长的上下文，Infra 要回答怎么切分、怎么省显存、怎么提升吞吐、怎么保证失败恢复。我的理解是，算法工程师偏“把模型做准”，AI Infra 偏“把模型跑快、跑稳、跑便宜、跑得可规模化”。优秀的 Infra 最终会放大算法团队的迭代效率。

### 4、在分布式训练中，如果出现了节点间通信带宽成为瓶颈的情况，除了换硬件，有哪些软件层面的优化思路？

### 5、了解参数服务器（Parameter Server）和 All-Reduce 这两种同步架构吗？比较一下它们的优缺点。

Parameter Server 适合超大规模稀疏参数，比如推荐系统 Embedding。Worker 拉取和推送参数，PS 负责分片存储和更新，优点是支持稀疏更新、参数可大于单机内存、易做异步训练；缺点是 PS 容易成为热点，参数一致性复杂，网络和负载均衡压力大。All-Reduce 更适合稠密模型，比如 CNN、Transformer 的权重同步，每张卡本地算梯度，然后通过环形或树形 All-Reduce 聚合，优点是同步语义清晰、无中心瓶颈、NCCL 优化成熟；缺点是所有参数通常要每步同步，超大稀疏参数不经济。实际系统里常混合使用：稠密层 All-Reduce，稀疏 Embedding 走 PS 或分片存储。

### 6、讲一下你对计算图（Computational Graph）的理解。在训练和推理阶段，计算图分别扮演什么角色？

计算图本质是把模型计算表示成节点和边，节点是算子，边是 Tensor 依赖。训练阶段，计算图不仅描述前向计算，还要支持自动求导、反向图生成、梯度累积、参数更新和显存生命周期管理。框架可以基于图做算子融合、内存复用、通信插入和 checkpoint。推理阶段更强调静态优化和低延迟执行，例如常量折叠、算子融合、layout 转换、量化、TensorRT engine 构建等。动态图灵活，适合研发和复杂控制流；静态图优化空间大，适合部署。AI Infra 里理解计算图很重要，因为性能问题经常来自图切分、shape 动态性、算子不支持或 fallback。

### 8、如何实现一个高效的 Embedding Lookup 层？当 Embedding 表非常大时，有哪些优化和存储策略？

高效 Embedding Lookup 的核心是减少随机访存和通信。单机 GPU 上可以先对 ID 去重，再 lookup，再按原位置 scatter，避免重复访问；对多 hot 特征做 fused lookup + pooling，减少 kernel launch 和中间 Tensor。表很大时，可以按 feature 或 hash 分片，多机存储，热点 ID 放 GPU cache，冷门 ID 放 CPU 内存、SSD KV 或参数服务器。还可以用混合精度存储，比如 FP16/BF16 embedding、INT8 量化，优化带宽和显存。工程上要关注哈希冲突、cache 一致性、热点倾斜、扩容 rehash、训练和推理表版本同步。推荐系统里常见做法是分层存储加异步预取，同时监控命中率和长尾延迟。

### 9、你了解哪些模型压缩和加速技术（如剪枝、量化、知识蒸馏）？挑一个你熟悉的，详细说一下其原理和工程实现上的难点。

常见技术包括剪枝、量化、蒸馏、低秩分解、算子融合和 KV cache 优化。我比较熟悉量化。量化是把 FP32/FP16 权重或激活映射到 INT8、INT4 等低比特表示，减少显存和带宽，提高推理吞吐。工程上分 PTQ 和 QAT：PTQ 用校准集统计 scale/zero point，改造成本低；QAT 在训练中模拟量化误差，精度通常更稳。难点主要有三类：一是激活分布长尾导致 scale 不好选；二是部分算子不支持低比特，需要 fallback，破坏性能；三是量化粒度选择，比如 per-tensor 简单但精度差，per-channel 精度好但实现复杂。落地时要结合 TensorRT、CUDA kernel 和业务精度指标做端到端验证。

### 10、在模型推理部署时，为了提升吞吐，可能会将多个请求进行批处理（Batching）。动态 Batching 和静态 Batching 有什么区别？如何实现一个高效的动态 Batching？

静态 batching 是固定 batch size 推理，简单稳定，适合离线任务，但在线服务请求到达不均匀，容易增加等待或浪费 padding。动态 batching 是在运行时把短时间窗口内的请求合并，根据队列长度、超时时间、shape 和优先级生成 batch。高效实现需要一个调度器：请求进入队列后按模型、输入 shape、最大延迟 SLA 分桶，达到 max batch size 或等待超过 timeout 就触发推理。LLM 还要区分 prefill 和 decode，支持 continuous batching。常见坑是 batch 变大后单请求延迟超 SLA、不同 shape padding 太多、长请求阻塞短请求、GPU 显存峰值不可控。监控上要看吞吐、p99 延迟、队列等待和 GPU 利用率。

### 11、谈谈你对最近大模型（LLM）技术浪潮的看法，这对 AI Infra 提出了哪些新的挑战和机会？

LLM 把 AI Infra 的重点从“单模型训练任务”推向了“超大规模训练、推理和数据闭环系统”。挑战首先是训练成本高，要求高效并行策略，包括 data parallel、tensor parallel、pipeline parallel、sequence parallel，以及 ZeRO/FSDP 这类参数切分。其次是推理复杂，KV cache 占用大，长上下文和多租户调度会带来显存管理、动态 batching、抢占和隔离问题。数据侧也更重要，预训练、SFT、RLHF、评测都需要稳定的数据治理。机会在于基础设施价值变高：高性能通信、低比特推理、GPU 调度、向量检索、模型服务平台和观测系统都会成为核心竞争力。我的理解是，LLM 时代拼的不只是模型，也拼系统效率。

### 12、中间件问题：在 AI 训练流水线中，如何利用消息队列（如 Kafka）来解耦数据预处理、数据加载和训练任务？

Kafka 可以作为训练数据链路的缓冲和解耦层。上游日志、样本生成和特征处理模块把数据写到不同 topic，下游训练任务按 consumer group 消费，这样预处理和训练可以独立扩缩容。架构上可以按业务、模型版本、样本类型分 topic 或 partition，保证同一用户或特征 key 的局部有序。训练侧用 micro-batch 拉取消息，落本地缓存或直接转 Tensor。工程权衡是吞吐、延迟和一致性：Kafka 天然适合高吞吐，但 exactly-once 要靠事务、offset 管理和幂等写入配合。常见坑包括消费 lag 堆积、partition 倾斜、小消息过多、schema 演进不兼容、重复消费导致样本偏置，所以需要监控 lag、数据质量和训练指标漂移。

### 13、中间件问题：如果需要存储和快速检索海量的向量数据（比如 Embedding），你会考虑什么技术方案？了解 Elasticsearch 的向量检索能力吗？

海量向量检索通常会考虑专门的向量数据库或检索库，比如 Faiss、Milvus、Vespa、HNSWlib，也可以用 Elasticsearch / OpenSearch 的 dense_vector 和 kNN 能力。方案选择取决于规模、更新频率、过滤条件和延迟要求。纯向量召回可以用 IVF、HNSW、PQ 等 ANN 索引，牺牲少量召回率换延迟；如果需要结构化过滤、关键词和向量混合检索，Elasticsearch 更方便。ES 支持向量字段和近似 kNN，适合和倒排索引结合，但超大规模高 QPS 下要关注内存占用、segment merge、索引构建成本和召回调参。工程上常见架构是离线构建索引、在线增量写入、定期合并版本，服务侧做多路召回、rerank 和结果缓存。

### 14、手写算法：实现一个简单的带时间衰减的 LRU 缓存（LeetCode 146 LRU Cache 的变种）。get 和 put 操作每次都会使对应 key 的“热度”增加 1，并且所有 key 的热度每进行一次操作就自然衰减 10%（乘以 0.9）。当缓存满时，淘汰热度最低的项。

思路是避免每次操作都遍历所有 key 衰减。设全局衰减因子 `factor = 0.9^opCount`，每个节点存储归一化热度 `score = realHeat / factor`，那么当前真实热度是 `score * factor`。每次访问某个 key 时，它的真实热度需要加 1，所以归一化后是 `score += 1 / factor`。比较热度时，所有 key 乘同一个 `factor`，因此直接比较 `score` 即可。淘汰热度最低项可以用小根堆，但访问会更新 score，堆里会有旧值，弹出时用 lazy deletion 校验。复杂度：`get/put` 平均 `O(log n)`，空间 `O(n)`。如果还要求严格 LRU tie-break，可加时间戳，热度相同淘汰更久未访问的。

```cpp
#include <bits/stdc++.h>
using namespace std;

class DecayLRUCache {
    struct Node {
        int value;
        double score;      // normalized heat = realHeat / factor
        long long ts;      // last access time, for tie-break
    };

    struct HeapItem {
        double score;
        long long ts;
        int key;

        bool operator>(const HeapItem& other) const {
            if (fabs(score - other.score) > 1e-12) return score > other.score;
            return ts > other.ts;
        }
    };

    int cap;
    long long tick = 0;
    double factor = 1.0;
    unordered_map<int, Node> mp;
    priority_queue<HeapItem, vector<HeapItem>, greater<HeapItem>> heap;

    void decayOnce() {
        ++tick;
        factor *= 0.9;

        // 防止 factor 过小导致 1 / factor 溢出，整体重标定。
        if (factor < 1e-100) {
            for (auto& [k, node] : mp) {
                node.score *= factor;
            }
            factor = 1.0;

            priority_queue<HeapItem, vector<HeapItem>, greater<HeapItem>> fresh;
            for (auto& [k, node] : mp) {
                fresh.push({node.score, node.ts, k});
            }
            heap.swap(fresh);
        }
    }

    void touch(int key) {
        auto& node = mp[key];
        node.score += 1.0 / factor;
        node.ts = tick;
        heap.push({node.score, node.ts, key});
    }

    void evictOne() {
        while (!heap.empty()) {
            auto item = heap.top();
            heap.pop();

            auto it = mp.find(item.key);
            if (it == mp.end()) continue;

            const Node& cur = it->second;
            if (fabs(cur.score - item.score) < 1e-12 && cur.ts == item.ts) {
                mp.erase(it);
                return;
            }
        }
    }

public:
    explicit DecayLRUCache(int capacity) : cap(capacity) {}

    int get(int key) {
        if (cap <= 0) return -1;
        decayOnce();

        auto it = mp.find(key);
        if (it == mp.end()) return -1;

        int val = it->second.value;
        touch(key);
        return val;
    }

    void put(int key, int value) {
        if (cap <= 0) return;
        decayOnce();

        auto it = mp.find(key);
        if (it != mp.end()) {
            it->second.value = value;
            touch(key);
            return;
        }

        if ((int)mp.size() >= cap) {
            evictOne();
        }

        mp[key] = Node{value, 0.0, tick};
        touch(key);
    }
};
```

### 12、手写算法：给定一个字符串 s 和一个字符规律 p，实现一个支持 '.' 和 '*' 的正则表达式匹配。'.' 匹配任意单个字符，'*' 匹配零个或多个前面的那一个元素。实现 bool isMatch(string s, string p) 函数。（LeetCode 10）

这题我会用动态规划。定义 `dp[i][j]` 表示 `s` 的前 `i` 个字符和 `p` 的前 `j` 个字符是否匹配。普通字符或 `.` 时，如果当前字符能匹配，则 `dp[i][j] = dp[i-1][j-1]`。遇到 `*` 时，它修饰前一个字符，有两种情况：第一，匹配零次，直接看 `dp[i][j-2]`；第二，如果 `p[j-2]` 能匹配 `s[i-1]`，则 `*` 可以继续消费一个字符，看 `dp[i-1][j]`。初始化时 `dp[0][0] = true`，并处理 `a*`、`a*b*` 这种能匹配空串的模式。时间复杂度 `O(mn)`，空间复杂度 `O(mn)`，也可以优化到 `O(n)`。

```cpp
#include <string>
#include <vector>
using namespace std;

class Solution {
public:
    bool isMatch(string s, string p) {
        int m = s.size(), n = p.size();
        vector<vector<bool>> dp(m + 1, vector<bool>(n + 1, false));
        dp[0][0] = true;

        for (int j = 2; j <= n; ++j) {
            if (p[j - 1] == '*') {
                dp[0][j] = dp[0][j - 2];
            }
        }

        auto match = [&](int i, int j) {
            return p[j - 1] == '.' || s[i - 1] == p[j - 1];
        };

        for (int i = 1; i <= m; ++i) {
            for (int j = 1; j <= n; ++j) {
                if (p[j - 1] != '*') {
                    if (match(i, j)) {
                        dp[i][j] = dp[i - 1][j - 1];
                    }
                } else {
                    dp[i][j] = dp[i][j - 2];
                    if (match(i, j - 1)) {
                        dp[i][j] = dp[i][j] || dp[i - 1][j];
                    }
                }
            }
        }

        return dp[m][n];
    }
};
```
