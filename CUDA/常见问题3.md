## 训练优化和推理优化岗位通常会考察哪些 CUDA/GPU 能力？

这类岗位通常不只考察单个 CUDA 算子的手写能力，而是更关注候选人能否从系统视角定位和优化性能瓶颈。训练优化岗位常涉及训练框架开发、RL 框架、业务算法适配、并行训练策略、通信优化、显存优化和稳定性问题；推理优化岗位常涉及 serving 框架、推理并行、batching、KV cache、CUDA Graph、量化和端到端延迟优化。

标准答案可以概括为：核心能力是能解释 GPU 性能从哪里来、瓶颈在哪里、如何 profile、如何判断优化方向。需要掌握 SM、warp、memory hierarchy、Tensor Core、Roofline Model、NCCL、Nsight 工具、CUDA Graph、并行策略和典型算子原理。对于更底层的算子开发岗位，还需要深入 GEMM、Attention、量化算子、CUTLASS、通信计算 overlap 以及不同 GPU 架构下的实现差异。

## 如何从 GPU 利用率曲线判断性能瓶颈？

GPU 利用率曲线只能作为初步信号，不能单独作为结论。GPU 利用率高不一定代表计算效率高，可能只是有大量低效 kernel 在运行；GPU 利用率低也不一定代表 GPU 算子慢，可能是 CPU 调度、数据读取、通信同步或 kernel launch overhead 导致 GPU 等待。

标准排查思路是：

1. 如果 GPU 利用率长期较低，先看 CPU dataloader、host-to-device 拷贝、kernel launch 间隙、CUDA 同步点和通信等待。
2. 如果 GPU 利用率高但吞吐低，需要看 SM occupancy、Tensor Core 使用率、memory throughput、L2 hit rate、warp stall 原因和算子 arithmetic intensity。
3. 如果利用率呈周期性锯齿，通常说明 pipeline 中存在阶段性等待，例如数据读取、all-reduce、参数同步、KV cache 调度或 batch 组装。
4. 如果多卡利用率不均衡，需要检查负载均衡、通信拓扑、并行切分方式、straggler、不同 rank 的 batch shape 差异。

最终应该结合 Nsight Systems 看端到端 timeline，再用 Nsight Compute 看关键 kernel 的微观指标。

## GPU 温度、功耗和频率对性能分析有什么意义？

GPU 温度和功耗可以帮助判断性能是否受硬件状态限制。如果温度过高或功耗达到上限，GPU 可能发生降频，导致相同 kernel 的执行时间变长。此时继续优化算子代码可能收益有限，应先确认散热、功耗限制、应用时钟、GPU boost 状态和集群调度环境。

标准答案是：性能 profile 不能只看 kernel 时间，还要看运行时硬件状态。若 GPU power 长期打满但 Tensor Core 或 SM 利用率不高，可能是 memory bound 或低效访存导致功耗高但有效计算少；若温度高导致频率下降，需要先解决 thermal throttling；若频率波动明显，benchmark 结果需要多次重复并控制环境。

## Roofline Model 是什么？如何用它判断优化方向？

Roofline Model 用来判断一个算子主要受计算能力限制还是内存带宽限制。它的横轴通常是 arithmetic intensity，即每读取或写入 1 byte 数据能完成多少 FLOPs；纵轴是实际计算吞吐。模型中有两条主要上限：内存带宽上限和峰值计算上限。

标准判断方式是：

- 如果 arithmetic intensity 低，算子落在带宽屋顶附近，通常是 memory bound，优化重点是减少 global memory 访问、提升访存合并、使用 shared memory/cache、减少中间结果读写、做 kernel fusion。
- 如果 arithmetic intensity 高，算子接近计算屋顶，通常是 compute bound，优化重点是使用 Tensor Core、提高指令效率、改善 tiling、增加并行度、减少 warp stall。
- 如果算子远低于两条屋顶，说明实现效率差，可能存在访存不合并、bank conflict、occupancy 过低、同步过多、分支发散、寄存器溢出或 launch overhead。

## SM 在 CUDA 程序执行中扮演什么角色？

SM（Streaming Multiprocessor）是 NVIDIA GPU 上执行 CUDA kernel 的核心硬件单元。一个 GPU 由多个 SM 组成，每个 SM 内部包含 warp scheduler、register file、shared memory/L1 cache、CUDA core、Tensor Core 等资源。

标准答案是：CUDA block 会被调度到某个 SM 上执行，一个 block 在执行期间不会跨 SM 迁移。SM 以 warp 为基本调度单位执行指令。一个 SM 能同时驻留多少 block/warp，取决于线程数、寄存器用量、shared memory 用量和硬件上限。优化时需要关注 SM occupancy，但 occupancy 不是越高越好；关键是是否有足够 warp 隐藏访存和指令延迟，同时避免寄存器溢出、shared memory 过大和无效并行。

## CUDA Graph 有什么作用？适合哪些场景？

CUDA Graph 用于把一段重复执行的 GPU 工作流捕获成图，然后通过一次 graph launch 重放，减少 CPU 端 kernel launch 和 API 调度开销。它适合 kernel 数量多、每次执行结构稳定、shape 相对固定、端到端延迟敏感的场景，例如推理服务、固定 batch 的训练 step、decoder 推理中的稳定执行路径。

标准答案是：CUDA Graph 的核心收益是降低 CPU launch overhead，并让 GPU 执行序列更稳定。它的限制是图结构通常需要稳定，动态 shape、动态 batch、动态内存分配、条件分支和不同请求长度会增加捕获难度。实际使用时常通过预分配内存、固定最大 shape、padding、bucket 化、多个 graph cache 或将动态部分留在 graph 外部来适配。

## NCCL 主要解决什么问题？常见通信瓶颈如何分析？

NCCL 是 NVIDIA 面向多 GPU 和多节点 GPU 通信的高性能通信库，常用于 all-reduce、all-gather、reduce-scatter、broadcast、send/recv 等操作。训练中梯度同步、张量并行、流水并行、专家并行和推理分布式服务都依赖 NCCL。

标准分析思路是：

- 单机内优先看 NVLink、PCIe、NVSwitch 拓扑，确认通信是否走到了预期链路。
- 多机通信要看 IB/RoCE、网卡绑定、跨节点带宽、NCCL 拓扑发现和 rank placement。
- 如果通信时间长，需要区分是通信量太大、拓扑不合理、rank 映射差、通信和计算没有 overlap，还是某些 rank 计算慢导致其他 rank 等待。
- 对 all-reduce，可以考虑梯度 bucket、reduce-scatter + all-gather、ZeRO/FSDP、通信压缩、张量并行粒度调整和 overlap。

## 训练和推理中常见的并行方式有哪些？如何结合拓扑选择？

常见并行方式包括数据并行（DP）、张量并行（TP）、流水并行（PP）、序列并行（SP）、上下文并行（CP）、专家并行（EP）以及它们的混合形式。不同并行方式的通信模式不同，应该结合模型结构、batch size、序列长度、显存压力和 GPU 拓扑选择。

标准答案是：高带宽低延迟的 GPU 之间适合放通信频繁的并行，例如 TP、SP；跨节点通信成本高，通常更适合 DP 或 PP 等通信频率较低或可 overlap 的策略。SP 常用于降低激活显存，并把 LayerNorm、Dropout 等按序列维度切分；TP 适合大矩阵乘和 attention/MLP 权重切分；PP 适合模型层数很深但会引入 pipeline bubble；EP 适合 MoE，但要重点处理 token dispatch、负载均衡和 all-to-all 通信。

## Tensor Core 是什么？为什么深度学习中很重要？

Tensor Core 是 NVIDIA GPU 上专门用于矩阵乘累加的硬件单元，能高效执行 FP16、BF16、TF32、INT8、FP8 等格式的 MMA 操作。深度学习中的 GEMM、卷积、Attention 和 MLP 大量由矩阵乘组成，因此 Tensor Core 利用率直接影响训练和推理吞吐。

标准答案是：要用好 Tensor Core，需要满足数据类型、矩阵维度对齐、内存布局和算子实现要求。例如 FP16/BF16 GEMM 通常需要维度按特定粒度对齐；Ampere 支持 TF32，可在 FP32 输入下使用 Tensor Core 加速；Hopper 引入更强的 Tensor Core、FP8 和 TMA/WGMMA 等机制。性能分析时应关注 Tensor Core 指令是否实际被使用，而不是只看代码里使用了 half/bfloat16。

## 基础算子面试通常会问哪些原理？

常见基础算子包括 reduce、scan、sort/topk、GEMM、softmax、LayerNorm、RMSNorm、embedding、attention 和 fused elementwise。面试通常不一定要求完整手写工业级实现，但会要求解释并行划分、访存模式、shared memory 使用、warp-level primitive、同步、数值稳定性和性能瓶颈。

标准答案是：基础算子优化首先要判断数据访问和计算模式。reduce/scan 重点是 warp/block 级归约、shared memory、bank conflict 和跨 block 合并；softmax 重点是 max-subtract-exp-sum 的数值稳定性以及行级并行；LayerNorm/RMSNorm 重点是均值/方差归约和向量化读写；GEMM 重点是 tiling、shared memory staging、register blocking 和 Tensor Core；attention 重点是避免显式 materialize 大的 attention matrix，并通过 online softmax 或 block-wise 算法降低显存读写。

## Nsight Systems 和 Nsight Compute 分别用于什么？

Nsight Systems 用于看系统级 timeline，适合分析端到端性能，包括 CPU 调度、CUDA API、kernel launch 间隙、memcpy、NCCL 通信、不同 stream 的 overlap、dataloader 和多线程行为。Nsight Compute 用于分析单个或少数 kernel 的微观性能，包括 occupancy、warp stall、访存带宽、cache 命中率、shared memory bank conflict、Tensor Core 指令、寄存器使用和 roofline。

标准答案是：先用 Nsight Systems 定位时间花在哪里，再用 Nsight Compute 深挖关键 kernel 为什么慢。不要一开始就只看单个 kernel，因为端到端瓶颈可能是 CPU、通信、同步或数据读取；也不要只看 timeline，因为 kernel 内部低效需要通过硬件指标解释。

## 如何系统定位训练或推理的性能瓶颈？

标准流程是从端到端到局部逐层收窄：

1. 先明确优化目标：吞吐、延迟、显存、成本、稳定性还是扩展效率。
2. 用日志和监控看 GPU 利用率、显存、温度、功耗、step time、tokens/s、request latency。
3. 用 Nsight Systems 看 CPU、GPU、memcpy、NCCL、stream 和同步关系。
4. 找出耗时最多或阻塞最严重的阶段。
5. 对关键 kernel 用 Nsight Compute 分析 compute、memory、occupancy、stall 和 Tensor Core 使用情况。
6. 针对瓶颈选择优化方向：算子优化、kernel fusion、通信 overlap、数据 pipeline、并行策略、CUDA Graph、batching 或量化。

标准答案强调：优化必须基于证据，而不是凭经验直接改代码。

## 什么是投机采样中的 Tree Attention？实现难点是什么？

投机采样会一次生成多个候选 token，并用目标模型验证。Tree Attention 用于处理候选 token 形成的树状依赖关系：不同候选路径之间共享前缀，但后续分支不同，因此 attention mask 不再是普通 causal mask，而是树结构 mask。

标准答案是：Tree Attention 的实现难点包括 batch size 大于 1、每个 batch 的树大小和路径长度不同、候选节点变长、KV cache 位置映射复杂、attention mask 稀疏且不规则、不同 batch 之间负载不均衡。高性能实现通常需要把树结构展平成紧凑布局，维护 parent/position/path 信息，按 block 或 bucket 组织相似 shape，减少 padding 浪费，并让 kernel 能高效读取 KV cache。若要支持 CUDA Graph，还需要固定或 bucket 化 shape，并预分配中间 buffer。

## FlashAttention、flashinfer、xFormers 这类库通常解决什么问题？

这些库主要提供高性能 attention 或相关推理算子实现。FlashAttention 的核心思想是通过 block-wise 计算和 online softmax，避免显式存储完整 attention matrix，从而降低 HBM 读写并提升速度。xFormers 提供多种 memory-efficient attention 和 transformer 组件。flashinfer 更偏推理场景，关注 decode、prefill、KV cache、paged attention 等服务化需求。

标准答案是：使用这些库时要关注它们支持的 dtype、head size、mask 类型、变长 batch、paged KV cache、GQA/MQA、CUDA Graph、不同 GPU 架构和具体版本限制。社区算子通常在常见 shape 上很强，但对特殊 mask、tree attention、复杂变长、业务自定义 KV layout 或强 CUDA Graph 约束可能支持不足，需要二次开发或自定义 kernel。

## 动态 shape 或变长 batch 为什么会影响 CUDA Graph？

CUDA Graph 更适合固定执行图和固定资源布局。动态 shape 或变长 batch 会导致 kernel 参数、grid 配置、内存地址、中间 buffer 大小、执行路径甚至 kernel 数量发生变化，从而难以复用同一张 graph。

标准答案是：工程上常用 bucket 化、padding 到固定上限、预分配最大 buffer、graph cache、把动态部分放到 graph 外、或者为不同 batch/seq length 维护多张 graph。对于 LLM 推理，prefill 和 decode 的动态性不同，decode 更容易 graph 化；prefill 由于输入长度变化大，通常需要更多 bucket 或专门处理。

## W4A16 量化算子的实现要点是什么？

W4A16 表示权重使用 4 bit，激活使用 16 bit。它的目标是降低权重带宽和显存占用，同时保持激活计算精度。推理中常见做法是在 GEMM 前或 GEMM 内对 int4 权重进行 dequantize，再与 FP16/BF16 激活做矩阵乘。

标准答案是：实现重点包括 int4 权重打包布局、scale/zero-point 存储方式、group-wise quantization、反量化开销、Tensor Core 使用方式、访存对齐、shared memory staging、寄存器压力和融合策略。高性能实现通常把 dequantize 与 GEMM 融合，避免把反量化后的 FP16 权重写回 global memory。瓶颈经常在权重解包、scale 读取、layout 转换和小 batch 下的并行度不足。

## Split-K GEMM 是什么？适合什么场景？

Split-K 是把 GEMM 的 K 维拆成多个分片并行计算，每个分片得到部分累加结果，最后再做归约。它用于提升并行度，尤其是在 M、N 较小但 K 很大时，普通 GEMM 的 tile 数量不够，无法充分占满 GPU。

标准答案是：Split-K 的收益是增加并行 block 数，提高 SM 利用率；代价是需要额外的 partial result 存储和归约开销。适合 K 大、M/N 小、普通 tiling 并行度不足的场景。不适合本身 tile 数量已经足够或归约开销超过收益的场景。

## Stream-K GEMM 是什么？和 Split-K 有什么区别？

Stream-K 是一种更细粒度地把 GEMM 工作量映射到 SM 的策略，目标是在不规则 shape 或 tile 数量不足时提高负载均衡和 SM 利用率。它会把 K 维累加工作以 stream 的方式分配，让多个 SM 更均衡地处理同一个或多个 output tile 的部分 K 工作。

标准答案是：Split-K 更像显式地把 K 切成多个独立分片再归约；Stream-K 更强调跨 SM 的工作量调度和负载均衡，减少尾部 tile 或小矩阵导致的 SM 空闲。实际选择取决于矩阵 shape、归约代价、workspace、硬件架构和 kernel 实现。

## CUTLASS 常见考点有哪些？

CUTLASS 是 NVIDIA 提供的 CUDA GEMM 和深度学习算子模板库，抽象了 threadblock、warp、instruction 级别的矩阵乘实现。常见考点包括 tile shape、warp shape、instruction shape、memory layout、epilogue fusion、pipeline stage、Tensor Core MMA、shared memory swizzle、alignment、split-k、persistent kernel 和不同架构的特性。

标准答案是：理解 CUTLASS 的关键是理解 GEMM 分层。threadblock 负责一个较大的 C tile，warp 负责其中的子 tile，Tensor Core instruction 执行更小的 MMA tile。数据通常从 global memory 搬到 shared memory，再进入 register，最后经过 epilogue 写回。优化时要在 occupancy、数据复用、shared memory、寄存器压力、pipeline stage 和 Tensor Core 利用率之间权衡。

## 算子融合为什么能提升性能？

算子融合的主要收益是减少 global memory 的中间结果读写，降低 kernel launch 次数，并提升 cache locality。对于 memory-bound 的 elementwise、norm、activation、bias、dropout 等算子，fusion 往往收益明显。

标准答案是：融合不是越多越好。过度融合可能增加寄存器压力、降低 occupancy、引入复杂分支、破坏复用或让 kernel 难以适配多种 shape。常见有效融合包括 bias + activation、add + norm、GEMM epilogue fusion、attention 中 softmax/dropout/value 的融合、量化中的 dequant + GEMM 融合。

## 通信计算 overlap 如何实现？有什么限制？

通信计算 overlap 是指让 GPU 在执行计算 kernel 的同时进行 NCCL 通信，从而隐藏部分通信时间。常见方法包括使用不同 CUDA stream、梯度 bucket、异步 all-reduce、reduce-scatter 与 backward 分层重叠、pipeline parallel 中前后向重叠、以及算子级别的通信融合。

标准答案是：overlap 的前提是计算和通信之间没有强依赖，且硬件和通信库支持并发执行。实际限制包括 stream 依赖设置错误、通信占用 SM 或带宽过多、bucket 太大或太小、rank 间负载不均、NCCL 算法选择、拓扑带宽不足。评估 overlap 时应看 Nsight Systems timeline，而不是只看通信 API 时间。

## Ampere 和 Hopper 架构下优化思路有什么差异？

Ampere 架构引入并强化了 TF32 Tensor Core、异步拷贝 cp.async、改进的 L2 和 Tensor Core 性能。Hopper 架构进一步引入 FP8 Tensor Core、WGMMA、TMA、thread block cluster、distributed shared memory 等机制，使 GEMM、Attention 和大模型训练推理有新的优化空间。

标准答案是：在 Ampere 上常重点利用 Tensor Core、cp.async、多 stage pipeline、shared memory tiling 和合适的数据类型；在 Hopper 上还应考虑 FP8、WGMMA、TMA bulk copy、warp-group 级别矩阵乘和更复杂的 pipeline。不同架构下同一个算子的最佳 tile shape、pipeline stage、shared memory 布局和指令路径可能不同，因此不能只用一套实现覆盖所有 GPU。

## 面试中如何回答“如何优化一个业务训练或推理系统”？

标准回答应该体现端到端思路：

1. 先定义指标：训练看 step time、tokens/s、MFU、扩展效率和稳定性；推理看 TTFT、TPOT、吞吐、P99 延迟、显存和成本。
2. 建立 baseline，固定输入、batch、模型版本和硬件环境。
3. 用监控和 profile 定位瓶颈是在计算、显存、通信、CPU 调度、数据读取还是框架开销。
4. 针对瓶颈优化：计算瓶颈用 Tensor Core、更优算子和融合；内存瓶颈减少读写和优化 layout；通信瓶颈做并行策略、拓扑和 overlap；CPU/launch 瓶颈用 CUDA Graph、batching 和减少同步；数据瓶颈优化 dataloader 和预取。
5. 验证收益和副作用，包括数值精度、稳定性、显存、长尾延迟和不同 shape 的泛化。

好的回答不应该只说“用更快的算子”，而要能说明为什么当前瓶颈适合这种优化。
