## ThreadsPerBlock和Blocks的数量受哪些条件约束。

`ThreadsPerBlock` 主要受下面几类条件约束：

- 硬件上限：每个 block 的最大线程数通常是 1024，单个维度也有上限，例如 `blockDim.x/y/z` 各自不能超过设备属性里的限制。
- warp 粒度：NVIDIA GPU 一个 warp 通常是 32 个线程，因此 block 线程数一般取 32 的倍数，避免最后一个 warp 利用率不足。
- 每个 SM 的资源：一个 block 会消耗 registers、shared memory、thread slots、warp slots、block slots。单个 block 使用资源越多，同一个 SM 能同时驻留的 block 越少。
- 算法访存模式：线程组织要尽量匹配连续内存访问、tile 形状、shared memory bank 布局、边界处理等。
- 同步范围：`__syncthreads()` 只在 block 内有效。如果算法需要更多线程共同同步，block 不能随意切小。

`Blocks` 的数量主要受：

- 问题规模：通常按 `numBlocks = ceil(N / threadsPerBlock)` 或二维/三维 tile 数量计算。
- grid 维度上限：`gridDim.x/y/z` 有硬件和 CUDA runtime 限制，具体看设备属性。
- SM 数量和并行度：blocks 至少要足够多，才能让所有 SM 有任务执行；如果 blocks 太少，GPU 利用率会低。
- 每个 block 的资源占用：资源占用过大时，同一 SM 同时驻留的 block 数变少，可能影响延迟隐藏能力。

经验上，先让 block 大小是 128、256、512 这类 32 的倍数，再结合 occupancy、访存合并、寄存器压力和实际 profiling 调整。

## 理论占用率怎么计算？

理论占用率（theoretical occupancy）指一个 SM 上实际可驻留的 active warps 数量与硬件允许的最大 active warps 数量的比值：

```text
occupancy = active_warps_per_SM / max_warps_per_SM
```

计算 active warps 时，需要同时考虑多个资源约束：

```text
warps_per_block = ceil(threads_per_block / warp_size)

active_blocks_per_SM = min(
    floor(max_threads_per_SM / threads_per_block),
    floor(max_warps_per_SM / warps_per_block),
    floor(registers_per_SM / registers_per_block),
    floor(shared_memory_per_SM / shared_memory_per_block),
    max_blocks_per_SM
)

active_warps_per_SM = active_blocks_per_SM * warps_per_block
occupancy = active_warps_per_SM / max_warps_per_SM
```

其中：

- `registers_per_block = registers_per_thread * threads_per_block`，实际还会受寄存器分配粒度影响。
- `shared_memory_per_block` 包括静态 shared memory 和动态 shared memory。
- `max_warps_per_SM`、`max_blocks_per_SM`、`max_threads_per_SM` 都随 GPU 架构变化。

注意理论占用率不是越高越好。高 occupancy 有利于隐藏访存延迟，但如果 bottleneck 是计算吞吐、shared memory 带宽、寄存器压力或指令依赖，继续提高 occupancy 可能没有收益，甚至会降低性能。

## 什么是warp，什么是warp divergence？

warp 是 NVIDIA GPU 上线程调度和执行的基本单位，通常由 32 个连续线程组成。CUDA 编程模型里看到的是 thread/block/grid，但硬件实际调度时会把一个 block 切成多个 warp。

同一个 warp 内的线程采用 SIMT（Single Instruction, Multiple Threads）方式执行：同一时刻执行同一条指令，但每个线程有自己的寄存器、线程 ID 和谓词状态。

warp divergence 指同一个 warp 内线程走了不同控制流路径，例如：

```cpp
if (threadIdx.x < 16) {
    // path A
} else {
    // path B
}
```

一个 warp 里部分线程走 A，部分线程走 B 时，GPU 通常会串行执行两个路径：执行 A 时屏蔽 B 的线程，执行 B 时屏蔽 A 的线程，最后再汇合。因此 divergence 会降低 warp 的有效并行度。

常见优化方法：

- 让同一个 warp 内线程尽量走相同分支。
- 把分支条件改成 warp 粒度或 block 粒度。
- 对短分支使用 predication 或条件表达式。
- 重新排列数据，让相似分支的数据被相邻线程处理。

## cuda的内存模型里有多少种memory，它们的位置(片上还是板上)，带宽和延迟的相对大小？

CUDA 常见 memory 类型如下：

| memory | 位置 | 作用域 | 典型特点 |
| --- | --- | --- | --- |
| register | 片上 | 单个线程 | 延迟最低、带宽最高、容量很小 |
| local memory | 逻辑上线程私有，物理上在 global memory，通常经过 L1/L2 cache | 单个线程 | 名字叫 local，但实际可能是板上 DRAM，延迟高 |
| shared memory | 片上 | 同一个 block | 手动管理的高速缓存，延迟低，带宽高，可能有 bank conflict |
| global memory | 板上显存 HBM/GDDR | 所有线程、host 可访问 | 容量大，延迟高，带宽高但依赖合并访问 |
| constant memory | 板上显存，带 constant cache | 所有线程只读 | warp 内广播同一地址时效率高，访问发散时变慢 |
| texture memory | 板上显存，带 texture/read-only cache | 所有线程只读 | 适合有空间局部性的只读访问 |
| L1/L2 cache | 片上 cache | L1 通常 SM 私有，L2 全 GPU 共享 | 缓存 global/local/texture 等访问 |

相对延迟通常可以粗略理解为：

```text
register < shared memory / L1 cache < L2 cache < global memory
```

相对带宽通常可以粗略理解为：

```text
register / shared memory 最高，L2 次之，global memory 理论带宽高但延迟大且强依赖访问模式
```

具体数值随 GPU 架构、频率、cache 配置、访问模式变化，不能脱离具体设备固定回答。

## global memory的访存合并是什么？

global memory 访存合并（memory coalescing）指一个 warp 内多个线程访问 global memory 时，如果访问地址连续、对齐、落在少量内存 transaction 中，硬件会把这些访问合并成少数几次内存事务。

例如，一个 warp 的 32 个线程分别读取连续的 `float`：

```cpp
float x = a[blockIdx.x * blockDim.x + threadIdx.x];
```

这类访问通常可以高效合并。反过来，如果每个线程访问的地址间隔很大、随机、未对齐，就会产生更多 transaction，浪费带宽。

访存合并的关键点：

- warp 内线程访问连续地址。
- 地址按数据类型和 cache line / segment 对齐。
- 尽量避免 stride 很大的访问。
- AoS 和 SoA 数据布局要根据访问模式选择；GPU 上经常 SoA 更利于合并访问。
- 矩阵按行主序存储时，通常让 `threadIdx.x` 对应连续列更高效。

访存合并主要影响 global memory 的有效带宽，是 CUDA kernel 性能优化中最基础也最重要的问题之一。

## 什么样的变量会被分配在register上？什么样的变量会被分配在local memory上？

通常会被分配到 register 的变量：

- 标量局部变量，例如 `int i`、`float acc`。
- 编译期能确定索引的小数组元素，且寄存器数量足够。
- 线程私有、生命周期较短、没有取地址的变量。
- 循环中的累加器、临时变量、指针、索引变量。

可能被分配到 local memory 的变量：

- 寄存器不够导致 register spilling。
- 线程私有数组，但数组下标是运行时变量，编译器无法把它完全放进寄存器。
- 大的局部数组或结构体。
- 被取地址的局部变量。
- 函数调用、复杂控制流导致编译器难以优化的线程私有数据。

需要注意：local memory 是每个线程私有的逻辑地址空间，但物理上通常位于 device memory，并经过 cache。因此 local memory 访问可能接近 global memory 的成本。

检查方法：

- `nvcc --ptxas-options=-v` 查看 register 数、spill stores、spill loads。
- Nsight Compute 查看 local memory 访问和寄存器压力。
- 查看 PTX/SASS 中是否出现 local load/store。

## Block是怎么被SM调度执行的？

kernel launch 后，grid 中的 blocks 会被分配到各个 SM 上执行。一个 block 一旦被调度到某个 SM，通常会在该 SM 上执行到结束，不会迁移到其他 SM。

调度大致过程：

1. GPU 收到 kernel 的 grid/block 配置。
2. 硬件调度器把未执行的 block 分配给有足够资源的 SM。
3. SM 为该 block 分配 registers、shared memory、warp slots 等资源。
4. block 内线程被划分成多个 warp。
5. SM 的 warp scheduler 从 resident warps 中选择 ready warp 发射指令。
6. block 内所有 warp 执行完成后，block 占用的资源释放，SM 可接收新的 block。

一个 SM 可以同时驻留多个 block，但数量受下面资源限制：

- 每 SM 最大 threads 数。
- 每 SM 最大 warps 数。
- 每 SM 最大 resident blocks 数。
- 每 block 使用的 registers。
- 每 block 使用的 shared memory。

block 之间没有固定执行顺序，不能依赖 block 调度顺序实现同步。不同 block 间全局同步通常需要拆成多个 kernel，或使用 cooperative groups 等特定机制。

## 什么是cuda core？什么是tensor core？

CUDA core 是 NVIDIA GPU SM 内执行通用标量/向量算术指令的计算单元，主要负责 FP32、INT32 等常规运算。CUDA core 不是 CUDA thread。thread 是编程模型里的逻辑执行单元，CUDA core 是硬件执行资源。

Tensor Core 是专门用于矩阵乘加的硬件单元，面向深度学习和矩阵计算。它通常以 tile 为单位执行 MMA（matrix multiply-accumulate）：

```text
D = A * B + C
```

Tensor Core 支持的数据类型和 tile 形状随架构变化，常见包括 FP16、BF16、TF32、INT8、FP8 等输入，以及 FP32 或更高精度累加。

两者区别：

- CUDA core：通用计算单元，适合标量算术、地址计算、普通控制流。
- Tensor Core：专用矩阵计算单元，适合 GEMM、卷积、attention 等可转化为矩阵乘加的计算。
- Tensor Core 吞吐通常远高于 CUDA core，但需要满足数据类型、矩阵形状、对齐、layout 和 API/指令要求。

常用调用方式：

- cuBLAS/cuDNN 这类库自动使用 Tensor Core。
- CUDA WMMA API。
- PTX/SASS 级别的 MMA 指令。
- CUTLASS、Triton 等框架生成 Tensor Core kernel。

## 什么是bank conflict？怎么避免bank conflict，你能想到多少方法？

bank conflict 通常指 shared memory bank conflict。shared memory 被分成多个 bank，常见是 32 个 bank。一个 warp 内多个线程同时访问 shared memory 时，如果多个不同地址落到同一个 bank，就需要串行处理，形成冲突。

常见 bank 编号计算可粗略写成：

```text
bank_id = (address / bank_width) % num_banks
```

其中 `bank_width` 常见为 4B，`num_banks` 常见为 32。具体随架构和配置变化。

需要区分：

- 多个线程访问同一个地址：很多场景可以 broadcast，不一定算冲突。
- 多个线程访问同一个 bank 的不同地址：会形成 bank conflict。

避免方法：

- padding：例如 `__shared__ float tile[32][33]`，避免转置访问时一整列落在同一 bank。
- 改变数据布局：让 warp 内连续线程访问连续 bank。
- swizzling：用 XOR 等方式重映射 shared memory 下标。
- 调整 tile shape：避免 stride 是 32、64 这类容易和 bank 数产生共振的值。
- 使用 vectorized load/store 时注意对齐和 bank 分布。
- 把部分数据放到 register 中，减少 shared memory 往返。
- 使用 warp shuffle 替代 shared memory 做 warp 内交换。
- 改变线程到数据的映射关系，例如让 `threadIdx.x` 对应连续维度。
- 对不同数据类型调整布局，例如 half、float4 访问要考虑每个线程实际触达的 bank。
- 使用库或框架已有的 layout，例如 CUTLASS 的 shared memory layout。

bank conflict 的优化目标不是完全消除所有冲突，而是消除热点路径上影响吞吐的冲突。

## 描述一下Block reduce的大致实现。

Block reduce 是在一个 block 内把多个线程持有的值规约成一个值，例如求和、最大值、最小值。

常见实现分两层：

1. warp 内 reduce：使用 `__shfl_down_sync` 做规约，不需要 shared memory。
2. warp 间 reduce：每个 warp 的 lane0 把局部结果写入 shared memory，再由一个 warp 读取这些结果继续 reduce。

示例思路：

```cpp
__inline__ __device__ float warp_reduce_sum(float v) {
    for (int offset = 16; offset > 0; offset >>= 1) {
        v += __shfl_down_sync(0xffffffff, v, offset);
    }
    return v;
}

__inline__ __device__ float block_reduce_sum(float v) {
    __shared__ float smem[32]; // 最多支持 1024 threads，即 32 warps

    int lane = threadIdx.x % 32;
    int wid = threadIdx.x / 32;
    int num_warps = (blockDim.x + 31) / 32;

    v = warp_reduce_sum(v);

    if (lane == 0) {
        smem[wid] = v;
    }
    __syncthreads();

    v = (threadIdx.x < num_warps) ? smem[lane] : 0.0f;

    if (wid == 0) {
        v = warp_reduce_sum(v);
    }
    return v;
}
```

实际实现要注意：

- `blockDim.x` 不一定是 32 的整数倍时，要处理有效 warp 数。
- reduce 操作要满足结合律，浮点加法由于舍入误差不严格满足结合律，结果可能和 CPU 顺序求和不同。
- 如果需要输出 block 结果，通常让 `threadIdx.x == 0` 写 global memory。
- 全局 reduce 通常需要多阶段 kernel，或者使用 atomic / cooperative groups。

## 描述一下double buffer(ping pong buffer）的大概原理和实现。

double buffer，也叫 ping-pong buffer，是用两个缓冲区交替承担“加载下一批数据”和“计算当前数据”的任务，从而重叠访存和计算，减少等待时间。

基本思想：

```text
buffer0 加载 tile0
同步

for tile in tiles:
    用当前 buffer 计算
    同时把下一个 tile 加载到另一个 buffer
    交换当前 buffer 和下一个 buffer
```

在 CUDA 中常见于 matmul、convolution、attention 等 tile kernel：

- global memory 到 shared memory 使用两个 shared memory tile。
- shared memory 到 register 也可以做更细粒度 double buffering。
- Ampere 之后可结合 `cp.async` 异步拷贝，把 global->shared 的加载和 Tensor Core 计算流水化。

伪代码：

```cpp
load tile 0 into smem[0];
__syncthreads();

int read_buf = 0;
int write_buf = 1;

for (int k = 0; k < num_tiles; ++k) {
    if (k + 1 < num_tiles) {
        load tile k + 1 into smem[write_buf];
    }

    compute using smem[read_buf];
    __syncthreads();

    swap(read_buf, write_buf);
}
```

更高性能实现中，加载和计算不是简单同步串行，而是通过异步 copy、pipeline stage、barrier、warp specialization 等方式让访存延迟被计算覆盖。

## 什么是roofline model？什么是memory bound，什么是computation bound？

Roofline model 是用来分析程序性能上限的模型。它把 kernel 性能和两个因素联系起来：

- 峰值计算能力：例如 TFLOPS。
- 峰值内存带宽：例如 GB/s。

核心指标是算术强度（arithmetic intensity）：

```text
arithmetic_intensity = FLOPs / bytes_accessed
```

Roofline 性能上限：

```text
performance <= min(peak_compute, arithmetic_intensity * peak_memory_bandwidth)
```

memory bound 指性能主要受内存带宽限制。特点是：

- 算术强度低。
- kernel 大量时间花在读写 global memory。
- 增加计算单元或 Tensor Core 不一定提升性能。
- 优化方向是减少访存、提升访存合并、cache 复用、shared memory 复用、fusion。

computation bound 指性能主要受计算吞吐限制。特点是：

- 算术强度高。
- 计算单元接近满负载。
- 优化方向是使用 Tensor Core、提高指令吞吐、减少依赖、提升 ILP、减少低效指令。

实际 kernel 还可能受 latency、同步、shared memory bank conflict、寄存器 spilling、occupancy 等因素限制，roofline 是上限模型，不是完整性能解释。

## kernel fusion为什么能提升性能？还有哪些好处？举几个fusion的例子。

kernel fusion 是把多个 kernel 合并成一个 kernel。它提升性能的主要原因是减少中间结果的 global memory 读写。

例如：

```text
y = relu(x + bias)
```

如果拆成两个 kernel：

1. `tmp = x + bias` 写 global memory。
2. `y = relu(tmp)` 再从 global memory 读 `tmp`。

fusion 后可以在寄存器里完成：

```text
y = relu(x + bias)
```

主要收益：

- 减少 global memory traffic。
- 减少 kernel launch overhead。
- 提高中间数据在 register/shared memory/cache 中复用的机会。
- 减少临时显存分配和显存占用。
- 可能改善 cache locality。

常见 fusion 例子：

- `bias + activation`：如 bias add + ReLU/GELU/Silu。
- `dropout + residual + layernorm`。
- `softmax` 内部的 max、exp、sum、除法合并。
- attention 中的 scale + mask + softmax。
- optimizer 中参数更新的多个 elementwise 操作合并。
- matmul 后接 bias、activation、residual 的 epilogue fusion。

注意 fusion 也有代价：

- 可能增加寄存器使用，降低 occupancy。
- 可能让代码复杂，编译时间变长。
- 如果 fusion 后破坏数据复用或引入更多分支，不一定更快。

## gpu上有分支预测吗？gpu上有指令集并行吗？

GPU 通常不像 CPU 那样依赖复杂的动态分支预测和乱序执行来提速。GPU 的核心策略是用大量 warps 隐藏延迟。

分支方面：

- GPU 有控制流指令和分支处理机制。
- 同一个 warp 内如果分支发散，会通过 predicate mask 串行执行不同路径。
- 因此性能重点不是“预测是否准确”，而是“warp 内是否 divergence”。
- 某些架构有更复杂的独立线程调度能力，但 warp divergence 仍然是重要性能问题。

指令级并行（ILP）方面：

- GPU 上有 ILP。一个线程或一个 warp 内如果存在多条互不依赖的指令，编译器和硬件可以更好地调度它们。
- 例如一个线程同时维护多个 accumulator，可以提高 FMA/Tensor Core 指令利用率。
- CUDA 优化中常见的 thread coarsening、unroll、多个独立累加器，本质上都在增加 ILP。

GPU 性能通常同时依赖：

- TLP（thread-level parallelism）：足够多 active warps 隐藏延迟。
- ILP（instruction-level parallelism）：单个 warp 内有足够独立指令填满流水线。

## 常用profile工具和方法。

常用工具：

- Nsight Systems：看端到端时间线，包括 CPU、CUDA API、kernel launch、memcpy、stream、通信等。
- Nsight Compute：看单个 kernel 的详细指标，包括 occupancy、访存、指令吞吐、warp stall、shared memory bank conflict 等。
- `nvprof`：老工具，已经逐渐被 Nsight 工具替代。
- CUPTI：底层 profiling API，很多框架 profiler 基于它。
- PyTorch Profiler：分析 PyTorch 模型的算子、CUDA kernel、CPU/GPU 时间线。
- `nvcc --ptxas-options=-v`：查看寄存器数量、shared memory、spill 情况。
- `cuobjdump` / `nvdisasm`：查看 SASS。
- `cuda-memcheck` / compute-sanitizer：检查越界、race、未初始化访问等问题。

常用方法：

1. 先用 Nsight Systems 看整体瓶颈：是 CPU launch、H2D/D2H memcpy、kernel、通信，还是同步。
2. 找到最耗时 kernel 后，用 Nsight Compute 看详细指标。
3. 判断瓶颈类型：memory bound、compute bound、latency bound、occupancy 问题、bank conflict、divergence。
4. 检查访存：global load/store 是否合并，L2/L1 hit rate，dram throughput 是否接近峰值。
5. 检查计算：FP32/Tensor Core pipe 利用率是否高。
6. 检查资源：register 数、shared memory、active warps、spill。
7. 修改 kernel 后用同样输入、warmup、重复多次测量，避免只看单次时间。

## float的计算一定比int消耗更多的cycle吗(主要是加法和乘法）？

不一定。float 和 int 哪个更快取决于 GPU 架构、数据类型、指令类型、吞吐管线和是否使用专用硬件。

一般理解：

- FP32 加法/乘法/FMA 是 GPU 的核心高吞吐路径，通常非常快。
- INT32 加法通常也很快，常用于地址计算。
- INT32 乘法可能比加法更贵，具体取决于架构和指令。
- FP16/BF16/TF32 如果走 Tensor Core，吞吐可能远高于普通 FP32/INT32。
- INT8 如果走 Tensor Core 或 DP4A 等指令，也可能非常快。

所以不能简单说 float 一定比 int 慢。更准确的回答是：

- 同一架构上，不同数据类型有不同吞吐表。
- 对加法来说，FP32 和 INT32 都可能是高吞吐指令。
- 对乘法来说，整数乘法、浮点乘法、FMA、Tensor Core MMA 属于不同路径，成本不能只按 float/int 二分。
- 实际性能还可能被访存、依赖链、occupancy、指令混合限制，而不是单条指令 cycle 决定。

面试中可以补一句：现代 GPU 上 FP32 FMA 是主力优化对象，float 计算不天然比 int 更贵。

## 常见的float格式。fp32，tf32，fp16，bf16的联系和区别？

常见格式：

| 格式 | 总位数 | 符号位 | 指数位 | 尾数位 | 特点 |
| --- | --- | --- | --- | --- | --- |
| FP32 | 32 | 1 | 8 | 23 | 标准单精度，精度高，范围大 |
| TF32 | 19 位有效格式，通常存储在 FP32 中使用 | 1 | 8 | 10 | 指数范围接近 FP32，尾数精度接近 FP16，常用于 Tensor Core |
| FP16 | 16 | 1 | 5 | 10 | 精度和范围都较小，吞吐高，常用于训练/推理 |
| BF16 | 16 | 1 | 8 | 7 | 指数范围接近 FP32，精度低于 FP16，训练稳定性较好 |

联系：

- 它们都是浮点数，通常由符号位、指数位、尾数位组成。
- FP16、BF16、TF32 常用于混合精度计算。
- 训练中常见做法是低精度输入，FP32 accumulate，以兼顾性能和数值稳定性。

区别：

- FP32：精度和范围都较好，但吞吐和显存占用不如低精度。
- FP16：尾数 10 位，精度比 BF16 高一些，但指数只有 5 位，动态范围小，容易 overflow/underflow。
- BF16：指数 8 位，动态范围接近 FP32，尾数只有 7 位，精度比 FP16 低，但训练更稳。
- TF32：NVIDIA Ampere 以后常见，使用 FP32 的指数范围和约 10 位尾数，主要用于加速 FP32 GEMM/conv，通常对用户表现为 FP32 输入但 Tensor Core 内部用 TF32 计算。

## ptx和sass是什么，和cuda的关系？

CUDA 源码到 GPU 执行大致经过：

```text
CUDA C++ -> PTX -> SASS -> GPU 执行
```

PTX（Parallel Thread Execution）：

- NVIDIA 定义的中间表示，类似虚拟 ISA。
- 和具体 GPU 架构解耦，具有一定前向兼容性。
- `nvcc` 可以生成 PTX。
- driver 可以在运行时把 PTX JIT 编译成目标 GPU 的机器码。

SASS：

- NVIDIA GPU 的真实机器指令。
- 和具体架构强相关，例如 sm_80、sm_90 的 SASS 指令不同。
- GPU 最终执行的是 SASS。
- 可以用 `cuobjdump`、`nvdisasm` 查看。

CUDA fatbin 中通常可以同时包含：

- 针对某些架构提前编译好的 SASS。
- PTX 作为后备，供新架构 JIT。

优化时：

- 看 PTX 可以理解编译器中间层表达。
- 看 SASS 更接近真实执行，能看到实际指令、寄存器、load/store、MMA、predicate 等。

## cuda上的排序和topk算法原理和实现。

GPU 排序和 topk 的核心目标是利用大规模并行，同时减少全局同步和随机访存。

常见排序算法：

- bitonic sort：适合小规模、block 内排序，比较交换模式规则，容易并行化。
- radix sort：适合整数或可转换 key 的大规模排序，按 bit 或 digit 分多轮处理，GPU 上很常用。
- merge sort：适合合并多个有序片段，也可并行化。
- sample sort / bucket sort：通过分桶降低全局排序复杂度，适合大规模数据但实现复杂。

bitonic sort 大致流程：

1. 数据加载到 shared memory。
2. 按固定 compare-exchange pattern 形成 bitonic sequence。
3. 多轮比较交换得到有序结果。
4. 写回 global memory。

优点是实现简单、适合小数组；缺点是复杂度较高，通常不适合超大规模完整排序。

radix sort 大致流程：

1. 按低位到高位分多轮处理 key。
2. 每轮统计 digit 的 histogram。
3. 对 histogram 做 prefix sum，得到每个元素的目标位置。
4. scatter 到输出 buffer。
5. ping-pong buffer 进入下一轮。

优点是吞吐高、适合大规模整数 key；缺点是需要多次全局读写和 scan。

TopK 的常见实现：

- 小 K：每个线程或每个 warp 维护局部 topk，最后 block/global 合并。
- 中等 K：block 内用 shared memory heap、bitonic partial sort 或 radix select。
- 大 K：通常接近排序问题，可以用 CUB/Thrust 的 sort/select 组合。
- 神经网络推理里常见 topk：先分块求局部 topk，再对局部候选做二次 topk。

TopK 不一定需要完整排序。只要找出前 K 个元素即可，因此常用 partial sort、selection、heap、radix select 等方法减少工作量。

实际工程中常用库：

- CUB：`DeviceRadixSort`、block/warp primitives。
- Thrust：`sort`、`sort_by_key`、`partial_sort` 等高级接口。
- 深度学习框架通常有定制 topk kernel，针对 batch、K 值、数据类型优化。

## matmul的优化，超级加分题。

矩阵乘法：

```text
C[M, N] = A[M, K] * B[K, N]
```

朴素实现中，每个线程计算一个 `C[i, j]`，会反复从 global memory 读取 A 和 B，数据复用差，性能通常很低。

核心优化方向是分层 tiling，让数据在 global memory、shared memory、register、Tensor Core 之间逐级复用。

典型优化层次：

1. Block-level tiling  
   一个 thread block 负责 `C` 的一个 tile，例如 `128x128`。对应加载 A 的 `128xKtile` 和 B 的 `Ktilex128` 到 shared memory。

2. Shared memory tiling  
   A/B tile 从 global memory 合并读取到 shared memory，block 内多个线程复用，减少 global memory traffic。

3. Register tiling  
   每个线程计算多个 C 元素，把 accumulator 放在寄存器中，提高数据复用和 ILP。

4. Vectorized/coalesced memory access  
   使用连续、对齐访问，例如 `float4`、`half2`，让 global load/store 合并。

5. Double buffering  
   计算当前 K tile 的同时预取下一个 K tile，隐藏 global memory 到 shared memory 的延迟。

6. Bank conflict 优化  
   shared memory layout 使用 padding、swizzle、permuted layout，避免 transpose/load matrix 时的 bank conflict。

7. Tensor Core  
   对 FP16/BF16/TF32/INT8 等类型，使用 WMMA/MMA 指令或 CUTLASS/Triton/cuBLAS，让矩阵乘法走 Tensor Core。

8. Warp-level MMA tiling  
   一个 warp 负责一个或多个 MMA tile，线程协作加载 fragment，执行 `mma.sync`。

9. Pipeline  
   使用多 stage pipeline，例如 `cp.async`、barrier、多级 shared memory buffer，把 global->shared、shared->register、MMA 计算重叠。

10. Epilogue fusion  
    在写回 C 前融合 bias、activation、scale、residual、quantize 等操作，减少额外 kernel 和 global memory 读写。

评价 matmul 优化时常看：

- Tensor Core 利用率。
- global memory load/store 是否合并。
- shared memory bank conflict。
- register 使用和 occupancy。
- L2 cache reuse。
- tile size 是否适合 M/N/K 形状。
- 边界处理是否引入过多分支。
- arithmetic intensity 是否足够高。

工程实践里，手写高性能 matmul 很难超过成熟库。优先使用 cuBLAS、CUTLASS、Triton 或框架已有 kernel；只有在特殊 shape、特殊 epilogue、特殊数据布局时，才值得定制。
