# 大模型推理加速核心技术内容

> 目标：系统梳理大模型推理加速从算子、模型、引擎、专项能力到集群服务的核心技术。  
> 适用对象：推理框架研发、模型部署、性能优化、GPU 集群服务工程。  
> 关键指标：TTFT（首 token 延迟）、TPOT/ITL（token 间延迟）、吞吐 tokens/s、QPS、P95/P99 延迟、GPU 利用率、HBM 带宽、KV Cache 占用、通信耗时。

---

## 总览：推理加速的分层视角

大模型推理通常分为两个阶段：

- **Prefill / Prompt 阶段**：处理完整输入 prompt，构建每层 KV Cache。长 prompt 下计算密集，Attention 复杂度高，影响 TTFT。
- **Decode / Generation 阶段**：逐 token 生成，每步读取历史 KV Cache。计算粒度小，常受显存带宽、KV 访问、调度和通信限制，影响 TPOT/ITL。

优化层次可以拆为：

1. **算子层面**：Tokenizer、前后处理、GEMM、Norm、Attention、MLP、采样 kernel。
2. **模型层面**：KV Cache、GQA/MQA/MLA、MoE、TP/EP/PP、通信、模型源码适配。
3. **框架引擎层面**：GPU 显存调度、Continuous Batching、PagedAttention、动态调度、Tokenizer 并行。
4. **专项特性优化**：长文本扩展、Chunked-Prefill、投机采样、MTP。
5. **集群服务层面**：PD 分离、KV/Prefix Cache 传输、跨机器调度、服务治理。

# 一、算子层面

## 1. Tokenizer 编码逻辑、前后处理算子、常规算子加速优化

### 1.1 Tokenizer 编码逻辑

**原理**：Tokenizer 将文本转为 token id，常见算法包括 BPE、SentencePiece、Unigram、WordPiece、Byte-level BPE。推理链路一般为：

```text
原始文本 -> 文本规范化 -> 分词 -> token ids -> position ids / attention mask -> 模型推理
```

在高并发服务中，Tokenizer 多在 CPU 侧运行，可能成为 GPU 前的隐性瓶颈。

**解决瓶颈**：

- 长 prompt、高 QPS 下 CPU 编码耗时高。
- 多轮对话重复编码历史上下文。
- 模板拼接、特殊 token、stop words、流式 detokenize 阻塞主流程。
- Tokenizer 串行导致 GPU 等输入。

**典型实现 / 框架**：Hugging Face Tokenizers（Rust）、SentencePiece、tiktoken、vLLM 异步 tokenizer、TGI router、SGLang runtime。

**优化方式**：

- 异步 tokenizer 线程池：请求接入、编码、推理调度流水化。
- Prompt token cache：缓存 system prompt、chat template、RAG 模板的编码结果。
- 批量 tokenize：离线批处理或高吞吐场景合并编码。
- 增量 detokenize：避免每轮从完整序列重新 decode。
- Stop words token-level 匹配：减少字符串级扫描。

**关键权衡**：

| 技术 | 收益 | 代价 |
|---|---|---|
| 异步 tokenizer | 降低 GPU 等待 | 线程池与调度复杂度增加 |
| Tokenizer cache | 减少重复编码 | 需要缓存淘汰和多租户隔离 |
| 批量编码 | CPU 吞吐更高 | 可能增加排队延迟 |
| 增量 detokenize | 降低流式输出开销 | 需处理 Unicode 边界 |

**落地要点**：监控 `tokenize_latency`、`detokenize_latency`、CPU 使用率、GPU idle time；RAG/Agent 场景尤其要缓存固定 prompt 与历史前缀。

### 1.2 前后处理算子

**原理**：前处理包含 padding/packing、attention mask、position ids、batch metadata；后处理包含 temperature、top-k/top-p、repetition penalty、softmax、sampling、stop condition、detokenize。

**解决瓶颈**：

- Decode 每 token 都采样，logits processor 开销被输出长度放大。
- GPU logits 拷贝到 CPU 采样会导致 CPU-GPU 同步。
- 动态 batch 状态变化频繁，metadata 更新成本高。
- Padding 浪费计算。

**典型实现**：vLLM GPU sampling、TensorRT-LLM optimized sampling kernels、TGI continuous batching、SGLang structured decoding。

**优化方式**：

- GPU sampling：logits 在 GPU 上完成 softmax/top-k/top-p/采样。
- Fused sampling kernel：融合 temperature、penalty、top-k/top-p、随机采样。
- Guided decoding mask 下沉到 GPU 或高效状态机。
- Packing / Continuous Batching 减少 padding 与 batch 空洞。

**落地要点**：关注每 token CPU-GPU memcpy 次数、采样 kernel 数、logits processor 耗时；结构化输出/JSON schema 会显著增加约束解码成本。

### 1.3 常规算子加速优化

**核心算子**：GEMM、RMSNorm/LayerNorm、RoPE、Softmax、Residual add、Activation、Quantize/Dequantize、KV Cache 读写、AllReduce/AllGather。

**主要瓶颈**：

- MLP/Linear 的 GEMM 计算量大。
- Norm、RoPE、elementwise 算子 FLOPs 低但访存频繁。
- 小 batch decode 的 kernel launch 开销明显。
- 量化推理中的 dequant 可能抵消收益。

**典型实现**：cuBLAS/cuBLASLt、CUTLASS、Triton、TensorRT-LLM、FasterTransformer、vLLM custom ops、FlashInfer。

**优化方向**：

- 算子融合：RMSNorm+Residual、QKV+RoPE+KV 写入、Dequant+GEMM、Linear+Bias+Activation。
- 高性能 GEMM：Tensor Core、FP16/BF16/FP8、INT8/INT4、合适 tile 和 layout。
- Norm 优化：warp-level reduction、向量化加载、fused residual。
- RoPE 优化：sin/cos cache、RoPE 与 Q/K projection 或 cache 写入融合。
- 量化 kernel：避免先完整 dequant 再 GEMM，优先 fused dequant matmul。

**Prefill vs Decode**：Prefill 更接近大矩阵计算，Tensor Core 利用率高；Decode 每步 token 少，更依赖 batching、融合、KV cache layout、低延迟通信。

## 2. Flash-Attention 等注意力算子优化

### 2.1 标准 Attention 瓶颈

标准自注意力：

```text
Attention(Q,K,V)=softmax(QK^T/sqrt(d)+mask)V
```

对序列长度 `N`，计算复杂度约 `O(N^2*d)`，显存中间量 `O(N^2)`。长上下文 prefill 时，attention score 矩阵和 softmax 中间结果带来巨大 HBM 读写。

### 2.2 FlashAttention

**原理**：按 block 加载 Q/K/V，在 SRAM/shared memory 中分块计算，使用 online softmax 累积结果，不把完整 attention matrix 写入 HBM。

```text
传统：QK^T -> HBM -> softmax -> HBM -> PV -> HBM
FlashAttention：分块 Q/K/V -> 片上 online softmax -> 只写最终输出
```

**解决瓶颈**：减少 HBM IO 和中间显存，不改变 Attention 数学结果，也不降低理论 `O(N^2)` FLOPs。

**典型实现**：FlashAttention 1/2/3、PyTorch SDPA、xFormers、TensorRT-LLM attention kernel、FlashInfer、vLLM backend。

**关键权衡**：

- 长序列 prefill 收益最大，短序列收益有限。
- 对 head_dim、dtype、mask、GPU 架构有约束。
- FlashAttention-3 更偏 Hopper/H100 异步能力与 FP8 优化。

### 2.3 PagedAttention

**原理**：将 KV Cache 切成固定大小 block，用 block table 将逻辑 token 映射到物理 KV block，类似虚拟内存分页。

**解决瓶颈**：变长请求显存碎片、KV 预分配浪费、Continuous Batching 下 cache 动态增长、prefix sharing 和 beam search 复用。

**典型实现**：vLLM PagedAttention、TensorRT-LLM paged KV cache、FlashInfer paged attention、SGLang。

**权衡**：block table 间接寻址增加 kernel 复杂度；block size 太小元数据多，太大内部碎片多。

### 2.4 MQA / GQA

**原理**：MHA 每个 Q head 有独立 K/V；MQA 所有 Q heads 共享一组 K/V；GQA 多个 Q heads 共享一组 K/V。

**收益**：Decode 阶段 KV 读取量与 `num_kv_heads` 成正比，减少 KV heads 可显著降低显存与 HBM 带宽。

**权衡**：必须模型训练时支持；直接从 MHA 改结构通常不可无损。

## 3. Transformer 自注意力、FFN、MLP 算子逻辑

### 3.1 Transformer Block

Decoder-only 常见结构：

```text
x -> RMSNorm -> Self-Attention -> Residual
  -> RMSNorm -> FFN/MLP/MoE -> Residual
```

热点包括 QKV projection、Attention、O projection、MLP up/gate/down projection、KV Cache 读写、Norm 和 residual。

### 3.2 自注意力逻辑

流程：

```text
X -> Q/K/V projection -> RoPE(Q,K) -> QK^T -> mask -> softmax -> PV -> output projection
```

Decode 阶段只对当前 token 生成 Q/K/V，并将新 K/V 写入 KV Cache，再与历史 KV 做 attention。

**优化映射**：

| 优化 | 主要阶段 | 解决问题 |
|---|---|---|
| FlashAttention | Prefill | attention 中间矩阵和 HBM IO |
| PagedAttention | Decode/Serving | 动态 KV 管理和长上下文并发 |
| GQA/MQA | Decode | KV Cache 容量和读取带宽 |
| Sliding Window | 长文本 | attention 范围和 KV 增长 |
| KV Quant | 长文本/高并发 | KV 显存占用 |

### 3.3 FFN / MLP 逻辑

常见 MLP：

```text
GELU FFN:   W2 * GELU(W1*x)
SwiGLU:     down_proj(SiLU(gate_proj(x)) * up_proj(x))
```

MLP 是参数和 FLOPs 大户，尤其 FFN intermediate size 通常远大于 hidden size。

**优化方式**：

- Gate/up projection 合并为一次 GEMM：`[gate, up] = x [W_gate, W_up]`。
- Activation + elementwise multiply 融合。
- Weight-only INT8/INT4、FP8、AWQ/GPTQ/SmoothQuant。
- MoE 中使用 grouped GEMM、fused routing、expert batching。

# 二、模型层面

## 1. 算子融合

### 1.1 Attention 与 RMSNorm 融合

**原理**：将 RMSNorm、Residual、QKV projection、RoPE、KV cache 写入、Attention 前处理等相邻操作融合，减少中间张量落显存。

**解决瓶颈**：降低 HBM 读写、kernel launch、decode 小 batch latency；提高 L2/cache locality。

**典型实现**：TensorRT-LLM fused kernels、Transformer Engine、Triton RMSNorm/RoPE kernels、FlashAttention、vLLM custom ops。

**权衡**：融合越深，寄存器压力和代码复杂度越高；对 hidden size、head_dim、dtype、GPU 架构特化明显。

### 1.2 计算与显存读取融合

**原理**：一次读取数据后完成多步计算，避免 `read -> op -> write -> read -> op` 的 HBM 往返。

**典型融合**：RMSNorm+Residual、Dequant+GEMM、QKV+RoPE+KV 写入、SwiGLU activation、Attention score+mask+softmax。

**落地要点**：优先融合 memory-bound 算子；用 Nsight Systems/Compute 观察 kernel 数、HBM 带宽、occupancy。

---

## 2. 自回归推理优化

### 2.1 KV Cache 优化

KV Cache 大小近似：

```text
num_layers × batch_tokens × num_kv_heads × head_dim × 2(K,V) × dtype_size
```

**优化目标**：

- 减容量：GQA/MQA、MLA、KV 量化、低秩压缩。
- 减读取：PagedAttention、FlashDecoding、Sliding Window。
- 强管理：分页、prefix cache、continuous batching、copy-on-write。

**权衡**：KV 量化/压缩可能影响质量；prefix cache 依赖业务前缀重复率；Paged KV 增加间接寻址。

### 2.2 Packing 优化

**原理**：将不同长度序列紧凑拼接，用 `cu_seqlens`、offset、position ids、mask 防止互相 attend，减少 padding。

**典型实现**：FlashAttention varlen API、vLLM continuous batching、TensorRT-LLM in-flight batching、SGLang scheduler。

**落地要点**：离线任务用 length bucketing + packing；在线任务用 continuous batching；监控 padding ratio、有效 token 数、队列等待。

### 2.3 Paged Attention 分片

**单卡内分片**：KV 按 block 存储，降低碎片和预留浪费。  
**多卡/多节点分片**：可按 layer、head、sequence block 或并行策略切分 KV。

**关键参数**：block size、num_gpu_blocks、KV dtype、prefix cache、swap space、max_num_batched_tokens。

## 3. 分布式推理并行策略

### 3.1 Tensor Parallelism（TP）

**原理**：将单层大矩阵按列/行或 head 维度切分到多 GPU。Column parallel 用于 QKV/up/gate，Row parallel 用于 output/down。

**解决瓶颈**：单卡显存不足、单层 GEMM 太大、prefill 算力需求高。

**代价**：每层可能触发 AllReduce/AllGather/ReduceScatter；decode 阶段通信延迟被每 token 放大。

**落地**：TP 优先放节点内 NVLink/NVSwitch；确认 head 数、kv head 数、hidden size 可被 TP size 整除。

### 3.2 Expert Parallelism（EP）

**原理**：MoE 中不同专家放到不同 GPU，token 由 router 选择 top-k experts，经 AllToAll dispatch 到专家计算后 combine。

**解决瓶颈**：超大 MoE 专家权重无法单卡容纳；稀疏激活降低每 token 计算。

**代价**：AllToAll 通信、专家负载不均、小 batch 专家 GEMM 不饱和。

**落地**：统计每层每专家 token count；热点专家可副本化；跨节点 EP 要做拓扑亲和和专家分组。

### 3.3 Pipeline Parallelism（PP）

**原理**：按层切分模型到不同 stage/GPU，激活按 stage 传递。

**适用**：模型层数多、单卡/单节点放不下；跨节点部署时减少层内高频 AllReduce。

**代价**：流水线气泡；decode 每 token 串行过 stage，延迟敏感。

**常见组合**：节点内 TP/EP，节点间 PP。

---

## 4. 分布式通信

### 4.1 通信原语

- **AllReduce**：TP 合并部分结果。
- **ReduceScatter / AllGather**：减少完整张量复制，可替代部分 AllReduce 模式。
- **AllToAll**：MoE EP token dispatch/combine 核心。
- **Broadcast**：权重/状态广播。
- **Send/Recv**：PP stage 间激活传递。

实现基础：NCCL、RCCL、MPI、UCX、InfiniBand/RoCE、PyTorch Distributed。

### 4.2 AllReduce / AReduce 优化

**优化方向**：

- 选择 ring/tree/collnet/NVLS 等算法。
- 小通信融合或 custom allreduce 降低低延迟开销。
- ReduceScatter + AllGather 分解。
- 计算通信 overlap。
- 分层通信：节点内先规约，节点间再规约。

**关键权衡**：decode 小张量更看重延迟，prefill 大张量更看重带宽；融合通信可能增加等待。

### 4.3 跨节点通信优化

原则：**高频通信节点内，低频大粒度通信跨节点**。

常见策略：

- 节点内 TP/EP，节点间 PP。
- GPUDirect RDMA 减少 CPU 中转。
- 拓扑感知 rank mapping：GPU-NIC 亲和、NUMA、交换机层级。
- MoE 专家放置优化，减少跨节点 AllToAll。
- NCCL tests 做基准，关注 P99/P999 网络抖动。

## 5. 模型架构与源码

### 5.1 Qwen3.5、Qwen2.5/Qwen3 核实点

用户提到 **Qwen3.5**，公开生态中更常见的是 **Qwen2.5** 和 **Qwen3**。若是内部版本或新发布版本，应以官方模型卡、`config.json`、源码分支为准。

学习重点：

- `hidden_size`、`num_hidden_layers`、`num_attention_heads`、`num_key_value_heads`。
- RMSNorm、RoPE、GQA/MQA、SwiGLU、chat template、tokenizer。
- vLLM/SGLang/TensorRT-LLM 中是否走高性能模型路径。

### 5.2 DeepSeek-V3 / DeepSeek-32B 架构与源码

DeepSeek 系列重点关注：MoE、MLA、RMSNorm、路由策略、共享专家、低秩 KV 表示、长上下文配置。

落地前确认：具体模型是不是官方 DeepSeek-V3、DeepSeek-R1 蒸馏版、DeepSeek-Coder/V2/32B 等；不同版本结构差异很大。

### 5.3 DeepSeek MLA 优化

**MLA（Multi-head Latent Attention）原理**：不缓存完整 K/V，而缓存更紧凑 latent KV 表示，并在 attention 计算中通过投影/吸收优化参与计算。

**解决瓶颈**：显著降低 KV Cache 容量和 decode 读取带宽，适合长上下文与高并发。

**源码关注字段**：`q_lora_rank`、`kv_lora_rank`、`qk_rope_head_dim`、`qk_nope_head_dim`、`v_head_dim`、专家配置等。

**权衡**：实现复杂，对 RoPE 维度拆分和高性能 kernel 支持依赖强；fallback 实现可能抵消收益。

### 5.4 MoE 模型与专家并行

MoE 推理流程：

```text
hidden -> router logits -> top-k experts -> token dispatch -> expert MLP -> weighted combine
```

优化重点：fused MoE、grouped GEMM、token permutation/combine、专家负载均衡、EP AllToAll、热点专家副本。

# 三、框架引擎层面

## 1. 基础机制：GPU 显存调度、推理整体架构

### 1.1 推理整体架构

典型 serving pipeline：

```text
Client/API -> Request Queue -> Tokenizer -> Scheduler
  -> Batch Builder -> Model Executor -> Sampler -> Detokenizer/Streaming -> Client
```

核心模块：

- API server / router：接入、鉴权、限流、OpenAI-compatible 协议。
- Tokenizer / preprocessor：文本编码、chat template、prompt cache。
- Scheduler：prefill/decode 队列、token budget、优先级、准入控制。
- Model executor：GPU kernels、并行策略、CUDA Graph、采样。
- KV Cache manager：block 分配、释放、prefix cache、swap/recompute。
- Streaming output：增量 detokenize、断连取消、stop condition。

### 1.2 GPU 显存调度

显存组成：

```text
Model Weights + KV Cache + Activation/Temporary Buffers + CUDA Workspace
+ Communication Buffers + Runtime Metadata + Fragmentation/Reserved
```

**核心问题**：KV Cache 是在线推理最动态、最容易 OOM 的部分。显存调度需要进行 request admission control，判断新请求是否能被当前 KV block pool 容纳。

**策略**：

- 预分配显存池，避免频繁 cuda malloc。
- Paged KV block allocator，降低外部碎片。
- 设置 `gpu_memory_utilization` 安全水位。
- 长请求限流、最大上下文限制、超时取消释放 KV。
- CPU swap 或 recompute 作为极端情况下的 preemption 策略。

**典型框架**：vLLM block manager、TensorRT-LLM KV Cache Manager、TGI token limit、SGLang Radix Cache。

---

## 2. 流程优化：Tokenizer 与推理并行、Continuous Batching

### 2.1 Tokenizer 与推理并行

**原理**：CPU tokenizer、GPU prefill/decode、detokenizer/streaming 应流水化执行，避免串行阻塞。

```text
请求A: tokenize -> prefill -> decode -> detokenize
请求B:        tokenize -> prefill -> decode -> detokenize
请求C:               tokenize -> prefill -> decode -> detokenize
```

**落地**：tokenizer 线程池、异步队列、detokenizer 异步 flush、固定 prompt 预编码、监控 tokenizer queue。

### 2.2 Continuous Batching（连续批处理）

**原理**：每个 decode iteration 后动态移除完成请求、加入新请求，而不是等待静态 batch 全部完成。

```text
Step1: [A,B,C]
Step2: [A,B,C]
Step3: [B,C,D]  A完成，D加入
Step4: [B,D,E]  C完成，E加入
```

**解决瓶颈**：静态 batch 空洞、输出长度不均、GPU 利用率低、高并发排队。

**典型实现**：vLLM continuous batching、TensorRT-LLM in-flight batching、TGI continuous batching、Orca iteration-level scheduling。

**调度参数**：`max_num_seqs`、`max_num_batched_tokens`、`max_model_len`、`scheduler_delay`、`enable_chunked_prefill`、`gpu_memory_utilization`。

**权衡**：batch 大吞吐高但 ITL 变差；decode 优先改善流式体验但 TTFT 可能上升；prefill 优先改善新请求首 token 但可能卡住已生成请求。

---

## 3. 显存管理：Paged Attention 动态 KVCache 显存管理、动态推理调度

### 3.1 PagedAttention 动态 KV 管理

核心组件：KV block pool、block table、sequence metadata、paged attention kernel、scheduler。

**优点**：按需分配 KV，降低碎片；支持不同长度请求；支持 prefix sharing、copy-on-write、beam search；和 continuous batching 强绑定。

**关注指标**：block 利用率、free blocks、prefix cache hit rate、KV cache occupancy、swap/recompute 次数。

### 3.2 动态推理调度

调度输入：请求到达时间、prompt 长度、已生成长度、max tokens、优先级、剩余 KV blocks、GPU 当前负载、租户 quota。

调度输出：哪些请求 prefill、哪些 decode、是否 chunked prefill、是否接纳新请求、是否 swap/recompute、是否复用 prefix cache。

常见策略：

- FIFO：简单公平但长请求阻塞短请求。
- Shortest Job First：平均延迟低但长请求可能饥饿。
- Decode 优先：流式输出平滑，ITL 好。
- Prefill 优先：TTFT 好，但可能影响已运行请求。
- Chunked Prefill：长 prompt 切块，与 decode 交错。
- Token Budget Scheduling：每轮限制总 token 预算。
- Priority Scheduling：多租户/付费等级/实时优先。
- Preemption：显存不足时 swap KV 或 recompute。

# 四、专项特性优化

## 1. 长文本扩展：Sliding Window、Dynamic NTK、YaRN

### 1.1 Sliding Window Attention

**原理**：第 `t` 个 token 只关注最近 `W` 个 token，而不是完整历史。

```text
标准 causal: token_t -> [0 ... t]
滑动窗口:    token_t -> [t-W ... t]
```

**解决瓶颈**：降低长序列 attention 复杂度和 KV 读取；decode 可只保留窗口内 KV。

**典型实现**：Mistral、Longformer/BigBird 思想、FlashAttention local attention、TensorRT-LLM cyclic KV cache、vLLM/SGLang 对滑窗模型支持。

**权衡**：减少成本但牺牲远距离依赖；最好训练和推理 attention pattern 一致。

### 1.2 Dynamic NTK

**原理**：对 RoPE 的 base/theta 进行动态缩放，使训练长度之外的位置映射到模型更可接受的频率空间。

**解决问题**：RoPE 外推能力不足；无需重训即可尝试扩展上下文。

**注意**：Dynamic NTK 不减少 attention 计算和 KV 显存，只解决“位置能否外推”的质量问题。

### 1.3 YaRN

**原理**：更平滑地调整 RoPE 频率缩放，兼顾短上下文与长上下文质量，通常优于简单线性 scaling。

**落地要点**：确认模型 config 中 `rope_scaling`、factor、original max position；用 needle-in-a-haystack、长文 QA、多轮一致性验证，不只看最大 context length。

---

## 2. 吞吐提升：Chunked-Prefill、投机采样、MTP

### 2.1 Chunked-Prefill

**原理**：将长 prompt prefill 切成多个 chunk，与 decode 请求交错执行。

```text
chunk1 -> decode batch -> chunk2 -> decode batch -> chunk3 -> ...
```

**解决瓶颈**：长 prompt 一次性 prefill 阻塞 decode，导致 P99 ITL 抖动和 TTFT 过高。

**权衡**：不减少总计算量；改善调度公平和延迟分布；chunk size 需要调参。

### 2.2 投机采样（Speculative Decoding）

**原理**：小 draft model 先生成 K 个候选 token，大 target model 一次性验证，接受若干 token，从而减少 target model 调用次数。

```text
draft 生成 token1...tokenK -> target 并行验证 -> 接受前若干 token -> 继续
```

**影响因素**：draft 速度、draft 质量、接受率、K、采样温度、任务类型。

**典型实现**：vLLM speculative decoding、TensorRT-LLM、Hugging Face assistant generation、Medusa、EAGLE、Lookahead Decoding。

**适合**：decode-bound、低温问答、代码补全；对 prefill-heavy 或高温开放创作收益较低。

### 2.3 MTP（Multi-Token Prediction，多 Token 预测）

**原理**：模型一次 forward 预测未来多个 token，通常依赖训练阶段的多个预测头或特殊目标。

```text
hidden -> head_1 预测 t+1
       -> head_2 预测 t+2
       -> head_3 预测 t+3
```

**与投机采样关系**：MTP heads 可作为 draft，主模型或 verifier 验证候选路径。

**典型相关**：DeepSeek-V3 MTP 训练目标、Medusa 多头、EAGLE 特征预测、tree decoding。

**权衡**：需要模型结构/训练支持；多步预测误差累积；推理框架需支持接受/验证逻辑。

# 五、集群服务层面

## 1. 架构设计：PD 分离架构、缓存传输流水线

### 1.1 PD 分离架构（Prefill-Decode Disaggregation）

**原理**：将 Prefill 和 Decode 部署在不同 GPU worker/实例上：

```text
Client -> Router
       -> Prefill Worker: 处理长 prompt，构建 KV Cache
       -> KV Transfer / Cache Store
       -> Decode Worker: 持续生成 token
       -> Streaming Response
```

**为什么要分离**：

- Prefill 是计算密集型，长 prompt 会占用大量 GEMM/Attention 资源。
- Decode 是低延迟、访存密集型，每 token 小步执行，对 ITL 敏感。
- 两者混跑容易互相干扰：长 prefill 抬高 decode P99，decode 小步又会切碎 prefill 吞吐。

**解决瓶颈**：

- 降低长 prompt 对流式 decode 的干扰。
- 可对 prefill 与 decode 使用不同硬件/并行度配置。
- 提高集群资源利用：prefill 池按 prompt token 吞吐扩缩，decode 池按并发输出扩缩。
- 支持长上下文/RAG 中 KV 预热与复用。

**典型实现 / 方向**：DistServe、Mooncake、SGLang PD disaggregation、vLLM 相关分离式 serving 方案、LMCache 等 KV cache 传输/复用系统。

**关键权衡**：

| 维度 | 收益 | 代价 |
|---|---|---|
| 延迟隔离 | Decode P99 更稳定 | 架构复杂 |
| 资源配置 | Prefill/Decode 可独立扩缩 | 需要准确容量规划 |
| KV 复用 | 长 prompt 可预热 | KV 传输成本高 |
| 多机支持 | 支持大规模服务 | 网络成为新瓶颈 |

### 1.2 缓存传输流水线

PD 分离后，Prefill worker 生成的 KV Cache 必须传给 Decode worker。KV 传输可能很大，成为瓶颈。

**流水线化思路**：

```text
Prefill chunk 1 完成 -> 传输 chunk 1 KV -> Decode 准备
Prefill chunk 2 计算中 -> 传输 chunk 2 KV
Prefill chunk 3 计算中 -> Decode 可开始部分调度
```

**优化方向**：

- Chunked KV transfer：按 layer/block/chunk 分批传。
- 计算通信重叠：prefill 下一个 chunk 时传上一个 chunk KV。
- RDMA / GPUDirect：减少 CPU 拷贝。
- KV 压缩/量化传输：FP8/INT8 KV，减少带宽。
- 拓扑感知：prefill 和 decode worker 尽量 colocate 或同机架。
- Cache locality：同一会话后续请求路由到持有 prefix KV 的 decode worker。

**风险**：KV 传输如果慢于 decode 消费，会导致 decode worker 等待；如果传输占满网络，会影响跨节点 TP/EP 通信。

## 2. 调度与缓存：集群调度优化、Prefix Cache、跨机器通信优化

### 2.1 集群调度优化

集群层调度不同于单机 scheduler，需要考虑 worker 池、模型副本、GPU 拓扑、KV locality、租户优先级、冷热缓存。

**调度输入**：

- 请求 prompt/output 长度估计。
- 当前 prefill/decode worker 负载。
- KV Cache/Prefix Cache 命中位置。
- GPU 显存水位、KV block 剩余量。
- 网络拓扑、跨机带宽、节点健康。
- 租户 quota、优先级、SLA。

**调度策略**：

- **负载均衡**：按 queue length、running tokens、KV blocks、GPU utilization 分配。
- **Cache-aware routing**：有 prefix KV 的 worker 优先，减少重复 prefill。
- **Length-aware routing**：长 prompt 送 prefill 能力强的 worker；短请求走低延迟路径。
- **SLA-aware scheduling**：实时请求优先，离线任务低优先级。
- **Admission control**：超长请求、显存不足、租户超额时排队/拒绝/降级。
- **弹性扩缩容**：根据 prompt token/s 和 output token/s 分别扩 prefill/decode 池。

**关键指标**：queue time、TTFT/ITL P99、worker utilization、KV locality hit rate、cache eviction rate、recompute rate、network transfer time。

### 2.2 Prefix Cache

**原理**：多个请求共享相同前缀时，复用该前缀的 KV Cache，避免重复 prefill。

常见共享前缀：

- 固定 system prompt。
- RAG 模板和工具说明。
- 多轮对话历史。
- Agent 工作流中的公共上下文。
- 多候选采样 / beam search 的公共 prefix。

**实现形式**：

- Hash-based prefix cache：对 token prefix 分块 hash。
- Radix tree cache：SGLang RadixAttention 类思想，按前缀树管理共享 KV。
- Block-level ref count：Paged KV block 可被多个序列引用。
- Copy-on-write：共享 prefix 后续分叉时按需复制。

**收益**：降低 TTFT、减少 prefill FLOPs、节省显存、提升高重复业务吞吐。

**权衡**：

- 命中率依赖业务；开放聊天随机 prompt 命中较低。
- 缓存占显存，需要淘汰策略。
- 多租户场景需隔离，防止跨用户数据泄露。
- 分布式 cache 需要路由命中，否则跨机器拉 KV 可能不划算。

### 2.3 跨机器通信优化

**通信类型**：

- 模型并行通信：TP AllReduce、EP AllToAll、PP Send/Recv。
- KV transfer：PD 分离中 prefill -> decode 的 KV 传输。
- Prefix Cache 迁移：worker 间共享/拉取缓存。
- 控制面通信：调度、健康检查、指标上报。

**优化原则**：

1. 高频小通信尽量留在节点内 NVLink/NVSwitch。
2. 跨节点通信尽量大块、流水线化、可重叠。
3. 用拓扑感知 rank placement 避免跨 NUMA/GPU-NIC 非亲和。
4. 对 MoE AllToAll 做专家亲和和热点专家副本。
5. 对 PD KV transfer 使用 RDMA/GPUDirect，必要时 KV 量化。
6. 通过 cache-aware routing 避免不必要的远程 KV 拉取。

**落地检查清单**：

- 跑 NCCL tests：all_reduce、all_to_all、sendrecv 带宽/延迟。
- 确认 IB/RoCE、GDR、NUMA 绑定、MTU、PFC/ECN 配置。
- 监控跨节点 bytes/s、传输排队、重传、P99 网络延迟。
- 将模型并行通信和 KV transfer 的网络流量隔离或限流。

# 八、关键结论

1. **推理加速不是单点 kernel 优化**，而是算子、模型结构、调度、显存、通信、服务架构的系统工程。
2. **Prefill 与 Decode 的瓶颈不同**：Prefill 更偏计算密集，Decode 更偏 KV 访存、调度和通信延迟。
3. **KV Cache 是在线推理的核心资源**：GQA/MQA/MLA、PagedAttention、Prefix Cache、KV 量化都围绕它展开。
4. **Continuous Batching 和 PagedAttention 是高并发 serving 的基础组合**。
5. **MoE 的瓶颈常在通信和路由，而不只是专家 GEMM**。
6. **长上下文扩展分两类**：RoPE 外推解决“能不能处理长位置”，Sliding Window/Paged KV/Chunked Prefill 解决“成本和延迟”。
7. **投机采样和 MTP 瞄准自回归串行瓶颈**，但收益依赖接受率、模型支持和框架实现。
8. **集群层需要 cache-aware 与 topology-aware 调度**，否则 KV 传输和跨节点通信会抵消单机优化收益。
